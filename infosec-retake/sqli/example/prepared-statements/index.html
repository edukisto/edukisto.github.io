Защита

«Кодировка»​

При подключении к СУБД следует явно указать набор символов («кодировку») БД.

$dsn = "mysql:...;charset=utf8mb4";

Опции​

Третьим аргументом в конструктор объекта PDO можно передать массив опций для настройки соединения с СУБД.

$connection = new PDO($dsn, $username, $password, [
    // Опции.
]);

Обработка ошибок​

PDO предлагает три режима обработки ошибок:

  • PDO::ERRMODE_SILENT (по умолчанию в версиях PHP, предшествующих 8.0.0) — «тихий» режим, код и описание ошибки доступны по вызову $connection->errorCode() и $connection->errorInfo() соответственно;
  • PDO::ERRMODE_WARNING совмещает режим PDO::ERRMODE_SILENT с выводом сообщения об ошибке;
  • PDO::ERRMODE_EXCEPTION (по умолчанию в PHP 8) совмещает режим PDO::ERRMODE_SILENT с порождением исключительной ситуации.

Наиболее современный режим PDO::ERRMODE_EXCEPTION рекомендуется установить явно даже в PHP 8, чтобы обеспечить обратную совместимость сценария с устаревшими версиями языка.

$connection = new PDO($dsn, $username, $password, [
    PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
]);
Дополнительное задание

Самостоятельно изучите инструкцию по обработке исключительных ситуаций посредством try...catch и добейтесь того, чтобы при невозможности подключиться к СУБД или при невыполнении запроса в браузер выводилось как можно меньше информации о серверном ПО.

Эмуляция​

Многие СУБД, в т. ч. устаревшие, не поддерживают подготовленные предложения SQL. PDO позволяет эмулировать подготовку предложений SQL на уровне приложения БД.

Впрочем, и эмулятор, и СУБД способны подготовить только определённые предложения SQL. Перечени таких предложений для MySQL и эмулятора существенно различаются не в пользу последнего. В частности, MySQL, в отличие от эмулятора, разрешает подготовку предложений с LIMIT и OFFSET.

// Номер страницы.
$page = $_GET['page'] ?? 0;
// Максимальное число продуктов на странице.
$perPage = 5;
$query = 'SELECT * FROM products ORDER BY title LIMIT ? OFFSET ?';
$statement = $connection->prepare($query);
// При включённой опции `PDO::ATTR_EMULATE_PREPARES` возникнет ошибка.
$statement->execute([
    $perPage,
    $page * $perPage,
]);
$products = $statement->fetchAll();
var_dump($products);

Эмуляция отключена в популярном web framework’е Laravel.

В силу перечисленных выше причин рекомендуется отключить эмулятор.

$connection = new PDO($dsn, $username, $password, [
    PDO::ATTR_EMULATE_PREPARES => false,
    PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
]);
Эмуляция неизбежна при ошибках

Если СУБД не сможет подготовить предложение SQL, то PDO вернётся к эмуляции. Это следует из исходного кода модуля pdo_mysql.

Дополнительное задание

Сверьтесь с документацией и убедитесь, что MySQL может подготовить используемые Вами предложения SQL.

Постоянные соединения​

Постоянные соединения с СУБД не закрываются по завершении выполнения сценария, а кешируются и используются повторно при подключении с теми же учётными данными. Кеширование снижает накладные расходы на установление нового соединения.

С другой стороны, соедиение с СУБД остаётся открытым в менеджере процессов (например, FastCGI) при аварийном завершении сценария PHP. В результате возникают softlock’и (например, таблицы, заблокированные сломавшимся скриптом). Используя функцию register_shutdown_function, остановы сценариев можно отследить и завершить незавершённые операции, но обычно это делать нецелесообразно: постоянные соединения реализованы на уровне приложения БД и могут проигрывать нативным аналогам в надёжности и эффективности. Тем более, во многих случаях подключение к СУБД не требует особых затрат.

В силу перечисленных выше причин рекомендуется отключить постоянные соединения.

$connection = new PDO($dsn, $username, $password, [
    PDO::ATTR_EMULATE_PREPARES => false,
    PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
    PDO::ATTR_PERSISTENT => false,
]);

Итог​

Итак, подключение к СУБД рекомендуется выполнять следующим образом:

$connection = new PDO($dsn, $username, $password, [
    PDO::ATTR_EMULATE_PREPARES => false,
    PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
    PDO::ATTR_PERSISTENT => false,
]);

Подготовленные предложения​

Уязвимости для SQL­‑инъекций обычно возникают вследствие конкатенации или интерполяции строк, целью которых является вставка в предложения SQL аргументов, полученных извне (например, от пользователя).

Примеры уязвимых предложения SQL:

// Интерполяция («вклеивание»).
$sql = "SELECT * FROM products WHERE id = {$_GET['id']}";
$statement = $connection->query($sql);
// Конкатенация («склеивание»).
$sql = 'SELECT * FROM products WHERE id = ' . $_GET['id'];
$statement = $connection->query($sql);
// Снова интерполяция («вклеивание»).
$sql = "UPDATE products
        SET
            title = '{$_POST['title']}',
            price = {$_POST['price']}
        WHERE
            id = {$_POST['id']}
        ";

$connection->query($sql);

Проблема заключается в том, что аргументы не обрабатываются должным образом. Ничто не мешает злоумышленнику подставить в поле ввода на форме какой­‑либо знак одинарную кавычку, два дефиса подряд или какой­‑либо оператор SQL, чтобы исказить запрос, выполняемый СУБД.

Решением проблемы являются подготовленные предложения SQL.

$statement = $connection->prepare('SELECT * FROM products WHERE id = ?');
$statement->execute([
    $_GET['id'],
]);
$sql = 'UPDATE products
        SET
            title = ?,
            price = ?
        WHERE
            id = ?
        ';
$statement = $connection->prepare($sql);
$result = $statement->execute([
    $_POST['title'],
    $_POST['price'],
    $_POST['id'],
]);

Как мы видим, каждое место подстановки аргумента в предложении SQL отмечено вопросительным знаком (это так называемый placeholder). Одинарные кавычки вокруг строковых аргументов отсутствуют: СУБД поставит их автоматически. Вместо метода query используются два последовательных запроса к СУБД:

  • с первым запросом (метод prepare) передаётся предложение SQL с placeholder’ами, но без аргументов передаётся в СУБД;
  • со вторым запросом (метод execute) передаётся массив аргументов. Число элементов массива должно быть равно числу placeholder’ов.

Затем СУБД обрабатывает аргументы (обрамляет строки апострофами и экранирует апострофы внутри строк), подставляет их вместо вопросительных знаков и выполняет полученное предложение SQL.

Ознакомьтесь с документацией, приведёнными примерами и постарайтесь подобным образом изменить все небезопасные запросы в своём приложении БД.

После внесения правок снова проведите тестирование посредством sqlmap и убедитесь, что все уязвимости к SQL­‑инъекциям устранены.
