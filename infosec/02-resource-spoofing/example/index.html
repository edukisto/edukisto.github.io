Пример выполнения задания

Подмена подресурса через отравление DNS​

  1. Запустим эмулятор терминала.

  2. Откроем на редактирование файл, соотносящий IP­‑адреса с доменными именами.

    • Windows
    • Linux

    1. Запустим приложение «Командная строка» от имени администратора.

    2. Подадим команду

      Notepad "%SystemRoot%\System32\drivers\etc\hosts"

      Соответствующий шаблону файл (обычно это C:\Windows\System32\drivers\etc\hosts) будет открыт в тектовом редакторе «Блокнот».

    1. Запустим эмулятор терминала.

    2. Подадим команду

      sudo nano /etc/hosts

      Указанный файл будет открыт на редактирование в тектовом редакторе nano, запущенном от имени суперпользователя.

  3. Добавим в конец файла:

    127.0.0.1 good.com
    127.0.0.2 cdn.com
    127.0.0.3 evil.com

  4. Сохраним изменения.

  5. Подадим команды:

    ping good.com
    ping cdn.com
    ping evil.com

    При обмене пакетами с указанными узлами вычислительной сети должны быть выведены IP­‑адреса 127.0.0.1, 127.0.0.2 и 127.0.0.3 (соответственно).

    ping вместо nslookup

    Утилита nslookup в Windows (в отличие от Linux) обращается к DNS, не учитывая записи из файла hosts, поэтому она не подойдёт для наших экспериментов.

  6. в

    /etc/apache2/sites-available/good.com.conf
    <VirtualHost 127.0.0.1:80>
        DocumentRoot "/var/www/good.com/"
        ServerName good.com
    </VirtualHost>
    /etc/apache2/sites-available/cdn.com.conf
    <VirtualHost 127.0.0.2:80>
        DocumentRoot "/var/www/cdn.com"
        ServerName cdn.com

        <Directory "/">
            Header always set Access-Control-Allow-Origin "*"
        </Directory>
    </VirtualHost>
    /etc/apache2/sites-available/evil.com.conf
    <VirtualHost 127.0.0.3:80>
        DocumentRoot "/var/www/evil.com"
        ServerName evil.com

        <Directory "/">
            Header always set Access-Control-Allow-Origin "*"
        </Directory>
    </VirtualHost>

  7. Перезапустим Apache после правки конфигурации:

SHA‑256, SHA‑384 или SHA‑512?

Алгоритм SHA‑512, возможно, более устойчив к коллизиям (что может быть полезно в будущем при развитии квантовых компьютеров) и в среднем быстрее на 64­‑битных процессорах (доказательство: openssl speed sha256 sha512), однако хеши SHA‑256 в два раза короче, что экономит сетевой трафик.

Длину контрольной суммы, которая является представлением хеша в base64, можно определить по формуле:

⌊c+c−6×⌊c6⌋3⌋×4,\Bigg \lfloor \frac{c + c - 6 \times \big \lfloor \frac{c}{6} \big \rfloor}{3} \Bigg \rfloor \times 4,⌊3c+c−6×⌊6c​⌋​⌋×4,

где ccc — разрядность в байтах. Таким образом, длина контрольной суммы на основе SHA‑512 и SHA‑256 составит 88 и 44 Б соответственно. Это 44 ГБ сэкономленного трафика на каждый миллион пользователей.

Subresource spoofing через DNS poisoningXAMPP

apache\conf\extra\httpd-vhosts.conf

подсказка

Символ, разделяющий имена элементов файловой системы в записи пути, зависит от операционной системы (ОС):

  • в ОС, ориентируемых на семейство стандартов POSIX, в т. ч. Linux и macOS, используют косую черту («/»; англ. slash);
  • в Windows основным разделителем считают обратную косую черту («\»; англ. backslash), а дополнительным (альтернативным) — косую черту.

Путь к каталогу (папке) принято завершать этим же символом, например: C:\Windows\, /etc/ и т. п. Однако обратная косая черта в конце строки, обозначающей путь в конфигурационном файле Apache, является синтаксической ошибкой:

AH00526: Syntax error... DocumentRoot must be a directory

Исправить ошибку можно двумя способами:

  • удалить обратную косую черту («\»), размещённую в конце строки;
  • заменить все обратные косые черты́ («\») в строке на косые черты́ («/»).
