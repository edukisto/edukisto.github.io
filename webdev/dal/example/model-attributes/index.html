2. Настройка моделей

Модели в Laravel реализуют архитектурный шаблон active record и поэтому содержат методы манипулирования данными в БД. Например, создать в БД новую запись о продукте можно так (подобный код обычно размещают в методе store контроллера, но пока этого делать не нужно, просто прочитайте):

Гипотетический метод store контроллера
// Создаём новый экземпляр класса Product.
$product = new Product();

// Заполняем его свойства.
$product->name = 'Искусство программирования';
$product->price = 4999.99;
$product->description = 'Классическая монография Д. Кнута.';
$product->quantity = 46;
$product->available = true;

// Сохраняем в БД.
$product->save();

Если же мы хотим создать запись о продукте на основе данных, переданных пользователем, то можно создать форму для заполнения.

Гипотетический метод store контроллера
<form action="/products/" method="post">
  <input name="name" type="text" value="Искусство программирования">
  <input name="price" type="number" value="4999.99">
  <textarea name="description">Классическая монография Д. Кнута.</textarea>
  <input name="quantity" type="number" value="46">
  <input name="available" type="checkbox" checked>
  <button type="submit">Создать</button>
</form>

Конечно, нет необходимости создавать форму именно на HTML. Форму можно создать на JSX, в Qt, средствами Windows Forms и т. п. Ну а если пользователем приложения БД является не человек, а другая программа, то формы и вовсе не нужны.

Но так или иначе, запрос придётся обрабатывать на сервере, в методе store контроллера. Для доступа ко входным данным можно использовать параметр $request.

Гипотетический метод store контроллера
// Создаём новый экземпляр класса Product.
$product = new Product();

// Заполняем его свойства значениями полей формы.
$product->name = $request->input('name');
$product->price = $request->input('price');
$product->description = $request->input('description');
$product->quantity = $request->input('quantity');
$product->available = $request->input('available');

// Сохраняем в БД.
$product->save();

Возникают вопрос: можно ли соотнести поля с одноимёнными свойствами модели, чтобы не упоминать их поимённо в исходном коде? Вопрос резонный, и Laravel даёт положительный ответ.

Гипотетический метод store контроллера
// Извлекаем из запроса имена и значения полей и помещаем их в массив.
// Имена полей становятся ключами, значения полей — значениями элементов массива.
$input = $request->validated();

// Создаём новый объект класса Product.
$product = new Product();

// Заполняем модель данными из массива.
$product->fill($input);

// Сохраняем в БД.
$product->save();

То же самое можно записать ещё короче:

Гипотетический метод store контроллера
$data = $request->validated();
$product = Product::create($data);

Теперь Laravel должен соотнести ключи ассоциативного массива $input с именами свойств объекта $product и выполнить «массовое присваивание» значений элементов соответствующим свойствам. Но выполнение сценария завершится исключительной ситуацией:

Illuminate\Database\Eloquent\MassAssignmentException: Add [name] to fillable property to allow mass assignment on [App\Models\Product]...

Дело в том, что «массовое присваивание» запрещено по умолчанию в целях безопасности. Что если основой для разграничения доступа на сайте служит значение колонки is_admin в таблице users БД, и злоумышленник, редактируя свою учётную запись, разместит на форме нештатное поле is_admin со значением 1? Что если вредоносный сценарий в тех же условиях добавит поле password?

Впрочем, опасность легко устранить, создав чёрный (guarded) или белый (fillable) список свойств. Мы выберем более безопасный — белый — список. Откроем на редактирование файлы app/Models/Image.php, app/Models/Product.php, app/Models/Tag.php и добавим в классы моделей защищённое свойство $fillable с перечнем имён тех свойств, которые допущены к «массовому присваиванию» значений.

Файл app/Models/Product.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsToMany;
use Illuminate\Database\Eloquent\Relations\HasMany;

class Product extends Model
{
    /** @use HasFactory<\Database\Factories\ProductFactory> */
    use HasFactory;

    protected $fillable = [
        'name',
        'price',
        'description',
        'quantity',
        'available',
    ];

    public function images(): HasMany
    {
        return $this->hasMany(Image::class);
    }

    public function tags(): BelongsToMany
    {
        return $this->belongsToMany(Tag::class);
    }
}
Файл app/Models/Image.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;

class Image extends Model
{
    /** @use HasFactory<\Database\Factories\ImageFactory> */
    use HasFactory;

    protected $fillable = [
        'path',
        'product_id',
    ];

    public function product(): BelongsTo
    {
        return $this->belongsTo(Product::class);
    }
}
«Массовое присваивание» и ключи таблиц

В данном случае мы допустили до «массового присваивания» свойство product_id в подчинённой модели Image. Не бойтесь включать в белый список свойства, соотнесённые с такими столбцами таблицы БД, которые включены во внешний ключ.

А вот свойство со значением первичного ключа (по умолчанию id) и системные свойства (created_at, updated_at) лучше оградить от «массового присваивания». Мы не включаем их в белый список.

Файл app/Models/Tag.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsToMany;

class Tag extends Model
{
    /** @use HasFactory<\Database\Factories\TagFactory> */
    use HasFactory;

    protected $fillable = [
        'label',
    ];

    public function products(): BelongsToMany
    {
        return $this->belongsToMany(Product::class);
    }
}
