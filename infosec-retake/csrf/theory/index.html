Теоретические сведения

Общие сведения о CSRF​

Подделка межсайтового запроса (англ. cross­‑site request forgery), известная также как CSRF /ˌsiːˈsɝːf — сисёрф/, XSRF, one­‑click attack и session riding, — это атака, при проведении которой вредоносная программа инициирует подачу пользовательским агентом (например, браузером) несанционированного запроса к веб­‑ресурсу, на котором агент авторизован.

В отличие от XSS, эксплуатирующего доверие пользователя к ссылке или сайту, CSRF эксплуатирует доверие сервера клиенту.

Пример 1​

Пусть пользователи вымышленной социальной сети Fakebook имеют возможность поставить под тем или иным сообщением отметку «Нравится!». При этом клиент (браузер) подаёт запрос на веб­‑ресурс https://fakebook.example/messages/ID/likes/, где ID — идентификатор сообщения.

Злоумышленник в целях «накрутки» рейтинга сообщения № 1337 заманивает интернет­‑пользователей, которые могут оказаться авторизованными на Fakebook, на заранее подготовленную веб­‑страницу https://evil.example/trap/, с которой автоматически, без участия пользователя подаётся запрос на https://fakebook.example/messages/1337/likes/. Содержимое вредоносного сценария, проводящего CSRF, может выглядеть примерно так:

<!-- Форма по умолчанию не имеет визуального представления. -->
<form action="https://fakebook.example/messages/1337/likes/" method="post">
    <!-- ... -->
</form>

<!-- Сценарий JavaScript подаёт запрос без участия пользователя. -->
<script>window.onload = document.forms[0].submit();</script>

Пример 2​

Пусть пользователи электронной платёжной системы имеют возможность осуществлять денежные переводы. При переводе авторизованный клиент (браузер) подаёт запрос на ресурс https://bank.example/transfer/?recipient=R&amount=A, где R — номер счёта получателя, A — размер перевода.

Для похищения денежных средств достаточно организовать обращение авторизованного клиента по веб­‑адресу (URI), содержащему номер счёта похитителя и произвольное положительное число в качестве размера перевода. URI мог бы выглядеть примерно так:

<a href="https://bank.example/transfer/?recipient=260602&amount=100500">
  Скачать бесплатно, без регистрации и SMS!
</a>

Установка веб­‑страницы с URI https://bank.example/transfer/?recipient=260602&amount=100500 в качестве стартовой (домашней), а равно автоматическое восстановление вкладок приведут к повторному списанию денежных средств при каждом запуске браузера.

Противодействие CSRF​

Общие рекомендации разработчикам​

  1. Клиент (например, браузер), подавая запрос по HTTP, предлагает серверу применить один из методов протокола. В рассмотренной электронной платёжной системе клиент использует метод GET для создания записи о денежном переводе и тем самым нарушает протокол, поскольку GET определён в стандарте как безопасный и, следовательно, идемпотентный метод, предназначенный для получения клиентом содержимого того ресурса, на который указывает URI. Другими словами, клиент, предлагая серверу применить GET, не ожидает создания целевого ресурса и полагает, что повторные идентичные запросы также не дадут такого эффекта.

    Если бы разработчик использовал неидемпотентный метод POST, предназначенный именно для создания ресурсов на сервере, то для CSRF было бы недостаточно гиперссылки — потребовалась хотя бы форма.

  2. Допускается требовать подтверждений для наиболее важных операций (например, высылая проверочный код на телефонный номер пользователя), но не следует злоупотреблять этим методом: он не решает проблему CSRF коренным образом и досаждает пользователям.

  3. Веб­‑адреса https://fakebook.example/messages/1337/likes/ и https://bank.example/transfer/?recipient=260602&amount=100500 универсальны для всех пользователей — очевидно, идентификатор пользователя, ставящего отметку «Нравится!», и номер счёта отправителя денежного перевода извлекаются сервером из cookie, т. е. злоумышленнику знать их не обязательно. Следовательно, для защиты от CSRF можно внедрить в запрос идентификатор, связанный с пользователем, и проверять его при обработке запроса.

    Как только на форме в Fakebook появится скрытое поле с идентификатором пользователя, ставящего отметку «Нравится!»

    <form action="https://fakebook.example/messages/1337/likes/" method="post">
    <input type="hidden" name="user_id" value="42">
    </form>

    и как только URI денежного перевода приобретёт вид https://bank.example/transfer/?sender=123456&recipient=260602&amount=100500, где значение sender — номер счёта отправителя, массовый CSRF станет невозможен, и злоумышленнику придётся сосредоточиться на эксплуатации только известных ему учётных записей.

  4. Следует ограничить 15–⁠20 минутами сеанса работы с веб­‑приложением в условиях бездействия пользователя. Конечно, для социальной сети это требование чрезмерно, но для онлайн­‑банков оно приемлемо.

Определение источника происхождения запроса по заголовку Referer​

Заголовочное поле Referer в HTTP­‑запросе содержит абсолютный или относительный URI веб­‑ресурса, с которого подан запрос. (Правда, без фрагмента, т. е. без сегмента, расположенного после знака #.) Следовательно, Referer позволит отклонить запросы, поданные из других доменов.

Рассмотрим пример защиты от CSRF посредством сверки Referer.

  1. Клиент запрашивает с веб­‑сайта evil.example, контролируемого злоумышленником, веб­‑страницу /trap/:

    GET /trap/ HTTP/1.1
    Host: evil.example

  2. Веб­‑сайт злоумышленника отправляет клиенту HTML­‑форму с полями sender (номер счёта отправителя), recipient (номер счёта получателя) и amount (сумма денежного перевода), а также средства JavaScript для автоматического предъявления формы веб­‑сайту bank.example:

    HTTP/1.1 200 OK
    Date: Tue, 19 Jan 2038 03:14:08 GMT
    Content-Length: 212

    <form action="https://bank.example/transfer/" method="post">
      <input name="sender" value="123456">
      <input name="recipient" value="260602">
      <input name="amount" value="100500">
    </form>
    <script>window.onload = document.forms[0].submit();</script>

  3. Получив вышеназванный контент, клиент автоматически предъявляет заполненную форму веб­‑сайту bank.example, но в запросе указывает https://evil.example/trap/ как значение заголовочного поля Referer.

    POST /transfer/ HTTP/1.1
    Host: bank.example
    Referer: http://evil.example/trap/
    Content-Length: 44

    sender=123456&recipient=260602&amount=100500

  4. Веб­‑сайт bank.example сравнивает значение поля Referer со своим доменным именем.

    function check_referrer()
    {
        $referrer = $_SERVER['HTTP_REFERER'];
        return
            false !== filter_var($referrer, FILTER_VALIDATE_URL)
            && parse_url($referrer, PHP_URL_HOST) === $_SERVER['SERVER_NAME'];
    }

  5. Определив, что запрос подан из другого домена, веб­‑сайт bank.example отклоняет запрос.

    HTTP/1.1 403 Forbidden

Эксплуатацию Referer затрудняют технические ограничения, лежащие вне области контроля веб­‑разработчика:

  • Referer отсутствует в запросах, отправная точка которых не имеет URI. Возможные причины: ссылающийся ресурс получен пользовательским агентом по схеме file или data; запрос стал результатом прямого (ручного или программного) ввода в адресную строку; запрос стал результатом манипуляций с window.location;
  • поля Referer нет в запросах по схеме https;
  • поле Referer может быть модифицировано, очищено или удалено некоторыми аппаратными и программными средствами: межсетевыми экранами, проксирующими серверами, расширениями браузера;
  • некоторые версии браузеров и встраиваемых сред выполнения уязвимы для подмены значения Referer (англ. referrer spoofing).

Смягчение на основе токенов​

Смягчение CSRF посредством токена (англ. token­‑based CSRF mitigation) подразумевает использование токена — конфиденциального проверочного кода, создаваемого криптографически стойким генератором псевдослучайных чисел (далее — КСГПСЧ) для каждого сеанса. (В особо важных веб­‑приложениях, например онлайн­‑банках, токены генерируют под каждый запрос, хотя преимущество этого подхода не доказано.)

Каждый запрос, который может повлечь изменение состояния сервера, должен содержать две копии токена:

  • один экземпляр рекомендуется размещать в скрытом поле формы, а при использовании технологии AJAX — в метаданных документа HTML или в заголовках HTTP­‑запроса. Ни в коем случае не размещайте токен в URI: во­‑первых, браузер вносит веб­‑адреса в журнал посещений, а во­‑вторых, пользователь может случайно скопировать веб­‑адрес вместе с токеном и передать его третьим лицам;
  • другой экземпляр должен быть размещён в cookie или, как вариант, в сеансовых данных, к которым сервер может получить доступ через авторизационную cookie.

Естественно, метод требует, чтобы веб­‑приложение не было уязвимо для XSS и (желательно) шифровало трафик.

Приняв запрос, сервер сравнивает два токена. Если они не совпадают, а равно если один или оба из них отсутствуют, то сервер отклоняет запрос и направляет клиенту ответ с состоянием 403 Forbidden («Запрещено»). Защита построена на том, что злоумышленнику, подделывающему запрос, будет практически невозможно угадать правильный токен.

Известны три способа формирования токена:

  • шаблон токена­‑синхронизатора (англ. synchronizer token pattern);
  • шаблон токена, основанного на шифровании (англ. encryption­‑based token pattern). В качестве алгоритма шифрования OWASP рекомендует AES‑256‑GCM;
  • шаблон токена, основанного на HMAC (англ. HMAC­‑based token pattern), предполагает размещение в токене срока окончания сеанса и кода аутентификации сообщения на основе хеша. Код задан криптографической хеш­‑функцией, например HMAC‑SHA‑512‑256 или HMAC‑SHA3‑256, от секретного ключа и сообщения — идентификатора сеанса и той же метки времени. Секретный ключ хранится на сервере.
