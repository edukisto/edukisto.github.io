Теоретические сведения

В течение жизненного цикла какой­‑либо системы баз данных могут быть изменены требования не только к приложению, но и к схеме БД, вследствие чего администратору, вероятно, придётся создавать новые таблицы, переименовывать столбцы, налагать дополнительные ограничения целостности (англ. constraints) и т. д. [1]. Подобные поправки следует оформлять в виде сценариев миграции [2; 3]. Под миграцией в данном случае понимают перенос данных между схемами разных версий.

Сценарий миграции характеризуется способом выполнения и уровнем автоматизации. Так, сценарий ручной миграции, состоящий из указаний для администратора БД, может быть записан на естественном (например, русском) языке. Сценарий автоматизированной миграции обычно записывают на языке программирования (например, SQL) [2].

В каждом сценарии миграции описаны различия предыдущей и последующей версий схемы БД, при этом сценарий пригоден как для обновления схемы, так и для отмены этого обновления: например, столбец, добавленный при миграции (ALTER TABLE products ADD quantity INTEGER NOT NULL;), будет удалён при её откате (ALTER TABLE products DROP COLUMN quantity;). Сценарии миграции упорядочены хронологически, накапливаются в течение всего жизненного цикла системы и отражают историю изменений, внесённых в схему БД [2].

Некоторые каркасы программных систем, такие как Django и Symfony, в полуавтоматическом режиме генерируют сценарии миграции по результатам сравнения ранее созданных сценариев с текущим состоянием модели предметной области. Laravel и Ruby on Rails не имеют подобных встроенных инструментов, вынуждая разработчиков вручную писать сценарии миграции. Впрочем, автоматизированная генерация также не исключает ручной правки.

Сценарии миграции в Laravel совместимы как минимум с четырьмя СУБД: Microsoft SQL Server, MySQL, PostgreSQL, SQLite.

Преимущества метода​

Использование сценариев миграции

  • позволяет версионировать схемы БД [2];
  • позволяет каждому уполномоченному участнику проекта получить актуальную локальную копию схемы БД, с которой можно экспериментировать [3];
  • углубляет и упорядочивает взаимодействие администратора БД и разработчика приложения БД;
  • позволяет автоматизировать создание «поддельных» (англ. fake) БД для интеграционного тестирования и тестирования системы [3];
  • способствует непрерывной интеграции (англ. continuous integration, CI) [3].

Условия проведения миграций​

Для успешного и эффективного применения метода необходимо:

  • не использовать средства администрирования БД (MySQL Workbench, phpMyAdmin, Microsoft SQL Server Management Studio и т. п.) для подачи DDL­‑запросов (CREATE, ALTER, DROP и т. п.) в интерактивном режиме;
  • автоматизировать миграцию посредством сценария, записанного не на естественном языке, а на языке программирования [2; 3];
  • выполнить резервное копирование данных, хранящихся в рабочей (англ. production) БД, перед миграцией [2];
  • протестировать сценарий миграции [2; 3], используя «поддельные» данные и резервную копию данных из рабочей БД [2];
  • сохранить сценарий миграции в системе контроля версий (например, Git) [2; 3] не позднее соответствующих изменений в исходном коде приложения БД;
  • не изменять сценарий ранее выполненной миграции [2].

Некоторые идиомы для сценариев миграций, записанных на SQL:

  • соблюдайте правила оформления исходного кода;
  • используйте терминатор (точку с запятой), даже если сценарий состоит из одного предложения SQL;
  • убедитесь в существовании таблицы перед её удалением (DROP TABLE IF EXISTS products;);
  • удалите одноимённую таблицу перед созданием новой (DROP TABLE IF EXISTS products; CREATE TABLE products (...););
  • предпочтите модификацию колонки (ALTER COLUMN) её удалению с повторным добавлением. Впрочем, если БД заполнена и требуется заменить тип данных колонки на несовместимый (например, VARCHAR() на DATE), то следует переименовать старую колонку, создать новую, перенести данные (UPDATE), после чего удалить старую колонку;
  • поскольку индексация может отнимать много времени, перед модификацией колонки (например, перед ALTER TABLE products ALTER COLUMN quantity SET DEFAULT 0;) или корректировкой (UPDATE) множества значений удалите связанный с колонкой индекс (DROP INDEX) и воссоздайте его по завершении модификации/⁠корректировки (CREATE INDEX);
  • если требуется, используйте в сценариях миграции DML­‑запросы (INSERT, SELECT, UPDATE, DELETE) [2];
  • объедините все предложения в сценарии миграции в одну транзакцию [2].
