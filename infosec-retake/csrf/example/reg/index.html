Регистрация и аутентификация

Нередко модель предметной области содержит сущности, управление которыми разрешено только определённым аутентифицированным и авторизованным пользователям или их группам. В качестве примера можно привести сообщения в социальной сети: все сообщения всех пользователей хранятся в одной таблице базы данных, но пользователь имеет право редактировать и удалять только те сообщения, для которых он является автором.

Регистрация​

Запретите пустые и слишком короткие пароли.

Заставьте пользователя использовать сложные пароли.

Запретите слишком длинные пароли: пароли длиной свыше 160 Б могут способствовать атакам DoS.

Не ограничивайте алфавит. Даже если пароль хешируется, ограничение алфавита каким­‑либо множеством, например множеством английских букв или арабских цифр, упрощает атаки грубой силой и с помощью радужных таблиц.

Хранение​

Не храните нехешированные пароли.

Используйте криптографические хеш­‑функции.

Не используйте скомпрометированные криптографические хеш­‑функции, в том числе популярные MD5 и SHA1.

Коллизии​

Хеши́рование ⁠1 ⁠— это преобразование сообщения произвольной длины в сообщение фиксированной длины.

Колли́зия (англ. collision) для хеш­‑функции 𝑓 ⁠— это такая пара значений 𝑎, 𝑏 её аргумента, что 𝑎 ⁠≠ ⁠𝑏 и 𝑓(𝑎) ⁠= ⁠𝑓(𝑏).

Алгоритм​

По мнению OWASP, рейтинг криптографических хеш­‑функций выглядит так:

  1. Argon2, опередивший 23 хеш­‑функции в соревновании Password hashing competition, которое проводилось в 2013–⁠2015 гг. [@phc];
  2. PBKDF2, необходимый при сертификации FIPS;
  3. scrypt, хорошо противодействующий атакам с применением аппаратной акселерации;
  4. bcrypt.

У лидера рейтинга есть две основных версии:

  • Argon2i ⁠— «медленный» вариант с прямым доступом к памяти (DMA), который следует применять с многочисленными раундами для предотвращения атак по побочным каналам (в частности, по времени, затраченному системой на хеширование можно определить хеш­‑функцию);
  • «быстрый» Argon2d, напротив, не использует DMA, следовательно, устойчив к атакам с применением графических процессоров и предназначен для программ, не подвержденным атакам по времени (например, для криптовалют) [@GitHub/Argon2].

Реализация Argon2 на языке C89 перешла в общественное достояние.

Соль​

При каждом хешировании используйте длинную, псевдослучайную соль, которая предотвращает атаку по радужным таблицам, хотя и не влияет на атаки грубой силой или по словарю.

Соль снижает вероятность совпадения учётных данных разных пользователей и повышает энтропию независимо от сложности открытого сообщения.

Криптографически стойкий генератор псевдослучайных чисел должен соответствовать как минимум двум требованиям: прохождение статистических тестов на случайность и сохранение непредсказуемости, если часть исходного или текущего состояния становится известна криптоаналитику. Пример для подражания ⁠— класс SecureRandom в Java.

Схема безопасности не должна опираться на сокрытие соли.

Используйте уникальную соль для каждого пароля.

Длина соли, в зависимости от хеш­‑функции, должна составлять не менее 32 или 64 Б.

Восстановление пароля​

Предложение прислать пароль, указанный при регистрации, на электронную почту, указанную при регистрации же,⁠— верный признак того, что пароли хранятся в БД в нешифрованном виде.

Аутентификация​

Ограничьте количество попыток.
