Контейнеризация PHP

Сборка образа​

  1. В каталоге проекта создайте каталог php.

  2. В этом каталоге php создайте текстовый файл Dockerfile.

    Dockerfile, а не Dockerfile.txt

    При создании текстового файла некоторые ОС автоматически добавляют к его имени расширение txt. Убедитесь, что у имени созданного Вами Dockerfile нет расширения.

    Соблюдайте регистр букв

    В отличие от Windows, Unix­‑подобные ОС, в т. ч. macOS и ОС семейства Linux, чувствительны к регистру букв в именах файлов. Docker не найдёт Dockerfile с именем, записанным в другом регистре (DOCKERFILE, dockerfile и т. п.).

  3. Откройте веб­‑сайт Docker Hub и воспользуйтесь функцией поиска, чтобы найти документацию на образ php, отмеченный в поисковой выдаче логотипом языка (

    ).

  4. Из найденной документации следует, что выпуск образов php осуществляется в нескольких вариантах. Подберите тег, соответствующий образу php с ОС Alpine Linux, актуальной стабильной версией PHP (её номер должен быть в формате X.Y.Z, без идентификаторов alpha, beta, RC и т. п.) и менеджером процессов FPM, но без веб­‑сервера Apache. На момент составления настоящих методических указаний такой образ может быть обозначен тегом php:8.4-fpm-alpine.

  5. Добавьте в Dockerfile инструкцию FROM с выбранным тегом в качестве аргумента. FROM задаёт базовый образ для новой виртуальной машины, которую можно будет настроить последующими инструкциями.

    Файл php/Dockerfile
    FROM php:8.4-fpm-alpine

  6. Подайте команду на сборку образа новой виртуальной машины. Образ будет сохранён на носителе данных для последующего запуска.

    docker image build --tag=project:php -- ./php/

    Аргумент ./php/ указывает на каталог, в котором размещён Dockerfile с инструкциями по сборке.

    Опция --tag присваивает образу тег project:php.

    Два дефиса­‑минуса (--) согласно рекомендации POSIX отделяют перечень опций (их записывают с - или -- перед именем) от перечня аргументов. Без этого разделителя командная строка может ошибочно принять за опцию аргумент, начинающийся с дефиса (в нашем случае это могло быть имя каталога).

  7. Подайте команду на вывод перечня образов.

    docker image list

    Будет выведено:

    REPOSITORY   TAG   IMAGE ID   CREATED   SIZE
    project      php   ...        ...       ...

Управление контейнером​

Контейнер — это образ виртуальной машины, загруженный в оперативную память (ОЗУ). Синтаксис команды на создание и запуск контейнера таков (в квадратные скобки заключены необязательные элементы):

docker container run [опции] образ [команда для гостевой ОС] [аргументы команды для гостевой ОС...]

Контейнер как утилита​

Попробуем вывести номер версии транслятора PHP, установленного в гостевой ОС.

  1. Подайте команду на создание и запуск контейнера.

    docker container run --rm --tty -- project:php php -v

    Опция --rm по завершении работы контейнера удаляет с носителя данных файловую систему гостевой ОС.

    Когда использовать --rm?

    Создавайте контейнер с опцией --rm, если

    1. планируете только однократный его запуск (например, для использования в качестве утилиты);
    2. планируете его многократный запуск с разным набором опций и/⁠или аргументов;
    3. не требуется сохранение его состояния (например, в файловой системе гостевой ОС не размещена БД).

    Опция --tty позволяет хостовой и гостевой ОС совместно использовать текущий терминал. Без этой опции не сработают текстовые эффекты, применяемые приложениями в гостевой ОС (например, раскраска синтаксиса).

    Аргумент project:php — тег образа.

    Аргументы php и -v — это элементы команды php -v, которая будет выполнена внутри гостевой ОС.

    Должно быть выведено:

    PHP 8.4.4 (cli) (built: Feb 14 2025 19:20:49) (NTS)
    Copyright (c) The PHP Group
    Built by https://github.com/docker-library/php
    Zend Engine v4.4.4, Copyright (c) Zend Technologies

  2. Подайте команду на вывод перечня работающих контейнеров:

    docker container list

    или

    docker ps

    Должна быть выведена пустая таблица, что свидетельствует о завершении работы контейнера:

    CONTAINER ID   IMAGE     COMMAND   CREATED   STATUS    PORTS     NAMES

  3. Подайте команду на вывод перечня всех (работающих и остановленных) контейнеров:

    docker container list --all

    или

    docker ps -a

    Снова должна быть выведена пустая таблица, подтверждающая, что завершивший работу контейнер автоматически удалён с носителя данных:

    CONTAINER ID   IMAGE     COMMAND   CREATED   STATUS    PORTS     NAMES

Контейнер как служба​

Попробуем запустить в гостевой ОС отладочный веб­‑сервер, встроенный в PHP.

Создание и запуск​

  1. В каталоге проекта создайте каталог src.

  2. В каталоге src создайте текстовый файл index.php.

  3. Разместим в index.php следующий исходный код:

    <?php
    echo 'Привет, мир!';

  4. Подайте команду на создание и запуск контейнера.

    • macOS
    • Ubuntu
    • Windows 11
    docker container run \
      --detach \
      --name=project-php \
      --publish=8080:8008 \
      --rm \
      --tty \
      --volume="./src/:/var/www/html/" \
      -- \
      project:php \
      php -S 0.0.0.0:8008
    docker container run \
      --detach \
      --name=project-php \
      --publish=8080:8008 \
      --rm \
      --tty \
      --volume="./src/:/var/www/html/" \
      -- \
      project:php \
      php -S 0.0.0.0:8008
    docker container run ^
      --detach ^
      --name=project-php ^
      --publish=8080:8008 ^
      --rm ^
      --tty ^
      --volume="%CD%\src\:/var/www/html/" ^
      -- ^
      project:php ^
      php -S 0.0.0.0:8008
    Что такое %CD%?

    Переменная %CD% содержит абсолютный путь к текущей рабочей директории. Таким образом, %CD%\src\ это синоним пути C:\Users\ИМЯ_ПОЛЬЗОВАТЕЛЯ_ОС\docker-project\src\.

    Опция --detach запускает контейнер в фоновом режиме, чтобы не блокировать ввод команд в терминал, и выводит идентификатор контейнера.

    Опция --name задаёт имя контейнера.

    Опция --publish привязывает (англ. bind) порт хостовой ОС к порту гостевой ОС. Номера портов разделены двоеточием. Например, если браузер из хостовой ОС подаст запрос на локальный порт 8080, то Docker перенаправит его на порт 8008 гостевой ОС, прослушиваемый веб­‑сервером (см. php -S 0.0.0.0:8008).

    Опция --volume примонтирует каталог, размещённый в хостовой ОС (src, в котором размещён исходный код веб­‑сайта), к каталогу, размещённому внутри гостевой ОС (/var/www/html/, с которым взаимодействует веб­‑сервер). Пути к каталогам разделены двоеточием. Если браузер из хостовой ОС подаст запрос к файлу index.php, то веб­‑сервер извлечёт его из каталога /var/www/html/ в гостевой ОС, а фактически — из каталога src в хостовой ОС.

    Задавайте абсолютные пути

    Windows требует абсолютных (включая букву диска), а не относительных путей.

    Docker размещает файловую систему гостевой ОС на носителе данных и при отсутствии опции --rm продолжает хранить её по завершении работы контейнера (docker container start). С одной стороны, это позволяет восстановить состояние контейнера при повторном запуске (docker container start). С другой, остановленные контейнеры захламляют носитель данных — для их удаления придётся подавать команду docker container prune.

  5. Подайте команду на вывод перечня работающих контейнеров:

    docker container list

    Должна быть выведена следующая таблица:

    CONTAINER ID   IMAGE         ...   STATUS   PORTS         ...   NAMES
    ...            project:php   ...   Up...    9000/tcp...   ...   project-php

    Обратите внимание на сведения о портах: 9000/tcp, 0.0.0.0:8080->8008/tcp, [::]:8080->8008/tcp. Здесь связь между портами хостовой и гостевой ОС обозначена стрелкой (->). Как мы видим, порт 9000 гостевой ОС, ассоциированный с FPM, не связан ни с каким портом хостовой ОС — так и должно быть в интересах безопасности. Порт 8008 временно открыт нами при запуске контейнера.

Тест​

Обратитесь через браузер по веб­‑адресу http://localhost:8080/index.php.

В браузер должно быть выведено сообщение «Привет, мир!».

Изучение​

Синтаксис команды на выполнение команды в работающем контейнере (в квадратные скобки заключены необязательные элементы):

docker container exec [опции] контейнер команда_для_гостевой_ОС [аргументы команды для гостевой ОС...]

  1. Подайте команду на запуск терминала внутри работающего контейнера.

    • macOS
    • Ubuntu
    • Windows 11
    docker container exec \
      --interactive \
      --tty \
      -- \
      project-php \
      /bin/sh
    docker container exec \
      --interactive \
      --tty \
      -- \
      project-php \
      /bin/sh
    docker container exec ^
      --interactive ^
      --tty ^
      -- ^
      project-php ^
      /bin/sh

    Опция --interactive оставляет поток стандартного ввода (stdin) в терминал открытым для гостевой ОС. Без этой опции мы не увидим, что гостевая ОС выводит в терминал.

    Аргумент project-php — имя работающего контейнера, внутри которого нужно выполнить команду.

    В гостевой ОС будет запущена оболочка командной строки, а в текущий терминал будет выдано приглашение ко вводу команд (#) — теперь всё, что Вы вводите в терминал, адресовано гостевой ОС.

  2. Подайте команду, чтобы вывести информацию о конфигурационных файлах PHP.

    php --ini
    Configuration File (php.ini) Path: /usr/local/etc/php
    Loaded Configuration File:         (none)
    Scan for additional .ini files in: /usr/local/etc/php/conf.d
    Additional .ini files parsed:      /usr/local/etc/php/conf.d/docker-fpm.ini,
    /usr/local/etc/php/conf.d/docker-php-ext-sodium.ini

    PHP сообщит, что загруженных конфигурационных файлов нет.

  3. Перейдите в каталог, в котором размещены конфигурационные файлы PHP.

    cd "${PHP_INI_DIR}"

    Если в этом каталоге будет размещён файл php.ini, то PHP автоматически загрузит его как конфигурационный.

  4. Подайте команду на вывод перечня содержимого в текущем рабочем каталоге.

    ls -1p

    Опции -1 и -p выводят в одну колонку и добавляют косую черту после имён каталогов.

    Должно быть выведено:

    conf.d/
    php.ini-development
    php.ini-production

    Как видно, вместо php.ini в каталоге размещены файлы php.ini-development и php.ini-production — шаблоны конфигурационного файла для этапов разработки и эксплуатации соответственно.

  5. Подайте команду, чтобы создать копию php.ini-development и назвать её php.ini.

    cp php.ini-development php.ini

  6. Снова подайте команду.

    php --ini
    Configuration File (php.ini) Path: /usr/local/etc/php
    Loaded Configuration File:         /usr/local/etc/php/php.ini
    Scan for additional .ini files in: /usr/local/etc/php/conf.d
    Additional .ini files parsed:      /usr/local/etc/php/conf.d/docker-fpm.ini,
    /usr/local/etc/php/conf.d/docker-php-ext-sodium.ini

    Как мы видим, PHP автоматически загрузил файл /usr/local/etc/php/php.ini как конфигурационный.

  7. Подайте команду на выход из гостевой ОС.

    exit

Останов​

  1. Подайте команду на останов контейнера. В качестве аргумента можно задать либо идентификатор контейнера (не образа!), либо имя контейнера (project-php).

    docker container stop project-php

  2. Повторно подайте команду на вывод перечня работающих контейнеров:

    docker container list

    Должна быть выведена пустая таблица.

Пересборка образа​

Было бы удобно, чтобы конфигурационный файл был скопирован автоматически ещё на этапе сборки образа.

  1. Добавьте в Dockerfile инструкцию RUN и задайте команду копирования.

    Файл php/Dockerfile
    FROM php:8.4-fpm-alpine

    RUN cp "${PHP_INI_DIR}/php.ini-development" "${PHP_INI_DIR}/php.ini"

    RUN sed -i 's/max_execution_time = 30/max_execution_time = 180/' "${PHP_INI_DIR}/php.ini"

    Инструкция RUN в Dockerfile выполняет команду в гостевой ОС во время сборки образа.

    После копирования происходит вызов невизуального текстового редактора sed, чтобы увеличить время, выделяемое на выполнение сценария PHP, со стандартных 30 секунд до 180.

  2. Удалите старый образ.

    docker image remove project:php

    или

    docker rmi project:php

    В качестве аргумента передают либо тег, либо идентификатор образа:

  3. Соберите новый образ.

    docker image build --tag=project:php -- ./php/
