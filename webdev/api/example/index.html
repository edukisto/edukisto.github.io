Пример выполнения задания

Настройка моделей​

Модели в Laravel реализуют шаблон Active record и поэтому содержат методы для сохранения и удаления из БД. Например, создать новую запись о продукте можно так (подобный код обычно размещают в методе store контроллера, но пока этого делать не нужно, просто прочитайте):

Метод store контроллера
// Создаём новый экземпляр класса Product.
$product = new Product();

// Заполняем его свойства.
$product->name = 'Искусство программирования';
$product->price = 4999.99;
$product->description = 'Классическая монография Д. Кнута.';
$product->quantity = 46;
$product->available = true;

// Сохраняем в БД.
$product->save();

Если же мы хотим создать запись о продукте на основе данных, переданных пользователем, то можно создать форму для заполнения.

Метод create контроллера
<form action="/products/" method="post">
  <input name="name" type="text" value="Искусство программирования">
  <input name="price" type="number" value="4999.99">
  <textarea name="description">Классическая монография Д. Кнута.</textarea>
  <input name="quantity" type="number" value="46">
  <input name="available" type="checkbox" checked>
  <button type="submit">Создать</button>
</form>

Конечно, нет необходимости создавать форму именно на HTML. Форму можно создать на JSX, в Qt, средствами Windows Forms и т. п. Ну а если пользователем приложения БД является не человек, а другая программа, то формы и вовсе не нужны!

Но так или иначе, запрос нужно обрабатывать на сервере, в методе store контроллера. Параметр $request можно использовать для доступа ко входным данным.

Метод store контроллера
// Создаём новый экземпляр класса Product.
$product = new Product();

// Заполняем его свойства значениями полей формы.
$product->name = $request->input('name');
$product->price = $request->input('price');
$product->description = $request->input('description');
$product->quantity = $request->input('quantity');
$product->available = $request->input('available');

// Сохраняем в БД.
$product->save();

Возникают вопрос: можно ли соотнести поля с одноимёнными свойствами модели, чтобы не упоминать их поимённо в исходном коде? Вопрос резонный, и Laravel даёт положительный ответ.

// Извлекаем из запроса имена и значения полей и помещаем их в массив.
// Имена полей становятся ключами, значения полей — значениями элементов массива.
$input = $request->validated();

// Создаём новый объект класса Product.
$product = new Product();

// Заполняем модель данными из массива.
$product->fill($input);

// Сохраняем в БД.
$product->save();

То же самое можно записать ещё короче:

$data = $request->validated();
$product = Product::create($data);

Теперь Laravel должен соотнести ключи ассоциативного массива $input с именами свойств объекта $product и выполнить «массовое присваивание» значений элементов соответствующим свойствам. Но выполнение сценария завершится исключительной ситуацией:

Illuminate\Database\Eloquent\MassAssignmentException: Add [name] to fillable property to allow mass assignment on [App\Models\Product]...

Дело в том, что «массовое присваивание» запрещено по умолчанию в целях безопасности. Что если основой для разграничения доступа на сайте служит значение колонки is_admin в таблице users БД, и злоумышленник, редактируя свою учётную запись, разместит на форме нештатное поле is_admin со значением 1? Что если вредоносный сценарий в тех же условиях добавит поле password?

Впрочем, опасность легко устранить, создав чёрный (guarded) или белый (fillable) список свойств. Мы выберем более безопасный — белый — список. Откроем на редактирование файлы app/Models/Image.php, app/Models/Product.php, app/Models/Tag.php и добавим в классы моделей защищённое свойство $fillable с перечнем имён тех свойств, которые допущены к «массовому присваиванию» значений.

Файл app/Models/Product.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsToMany;
use Illuminate\Database\Eloquent\Relations\HasMany;

class Product extends Model
{
    use HasFactory;

    protected $fillable = [
        'name',
        'price',
        'description',
        'quantity',
        'available',
    ];

    public function images(): HasMany
    {
        return $this->hasMany(Image::class);
    }

    public function tags(): BelongsToMany
    {
        return $this->belongsToMany(Tag::class);
    }
}
Файл app/Models/Image.php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;

class Image extends Model
{
    use HasFactory;

    protected $fillable = [
        'path',
        'product_id',
    ];

    public function product(): BelongsTo
    {
        return $this->belongsTo(Product::class);
    }
}
«Массовое присваивание» и ключи таблиц

В данном случае мы допустили до «массового присваивания» свойство product_id в подчинённой модели Image. Не бойтесь включать в белый список свойства, соотнесённые с такими столбцами таблицы БД, которые включены во внешний ключ.

А вот свойство со значением первичного ключа (по умолчанию id) и системные свойства (created_at, updated_at) лучше оградить от «массового присваивания». Мы не включаем их в белый список.

Файл app/Models/Tag.php

Запросы​

В каталоге app/Http/Requests размещены файлы с классами, отвечающими за обработку клиентских запросов. На каждую модель приходится по два таких класса: Store...Request отвечает за «запросы на создание», а Update...Request — за «запросы на корректировку» информации в БД.

Авторизация​

Метод authorize в классах Store...Request и Update...Request проверяет, предоставлено ли пользователю право подавать запросы на хранение и корректировку данных в соответствующей модели. По умолчанию подача запросов запрещена — метод возвращает значение false, но мы заменим его на true. Задачи авторизации мы впоследствии возложим на отдельную подсистему.

Во всех файлах в каталоге app/Http/Requests/ приведите метод authorize к следующему виду:

public function authorize(): bool
{
    return false;
    return true;
}

Валидация​

Во время работы seeder’ов («сеялок») мы могли наблюдать ошибки, возникающие при вставке в таблицу БД недопустимых значений, таких как неуникальное наименование продукта, цена в неправильном формате, слишком длинное описание и т. п. Ошибки возникали на уровне СУБД, но источником недопустимых значений были фабрики.

Конечно, в достаточно было исправить фабрики, чтобы избавиться от ошибок. Однако на этапе эксплуатации наше приложение БД столкнётся с пользователями, которые будут умышленно или неумышленно отправлять запросы с недопустимыми данными. Независимо от намерений пользователей, нам следует отклонять некорректные запросы как можно раньше, не позволяя недопустимым входным данным проникать в нижние слои приложения БД.

Метод rules в классах Store...Request и Update...Request позволяет задать ограничения целостности для входных данных, поступающих на веб­‑сайт с запросами HTTP, в том числе из форм HTML. Метод должен вернуть двумерный массив, ключи которого соответствуют именам полей запроса, а значения — одному или нескольким правилам валидации. Если запрос не удовлетворяет требованиям, то обработка запроса завершится исключительной ситуацией. Другими словами, если раньше некорректные входные данные вызывали ошибку на уровне СУБД, то теперь работа веб­‑сайта остановится гораздо раньше, приблизительно на уровне контроллера.

Общий синтаксис правил:

public function rules(): array
{
    return [
        'имя_поля_1' => [
            'правило_валидации_1',
            // ...
            'правило_валидации_M',
        ],
        // ...
        'имя_поля_K' => [
            'правило_валидации_1',
            // ...
            'правило_валидации_N',
        ],
    ];
}

Полный перечень правил валидации приведён здесь: https://laravel.com/docs/11.x/validation#available-validation-rules. Рассмотрим важнейшие из них:

  • required — поле обязательно к заполнению;

    public function rules(): array
    {
        return [
            // Поле для ввода фамилии...
            'family_name' => [
                // ...обязательно к заполнению.
                'required',
            ],
            // Поле для ввода имени...
            'given_name' => [
                // ...обязательно к заполнению.
                'required',
            ],
            // Поле для ввода отчества...
            'patronymic' => [
                // ...не обязательно к заполнению.
                // Пустая строка преобразуется в NULL.
                'nullable',
            ],
        ];
    }

  • after:N, after_or_equal:N, before:N, before_or_equal:N, где N — имя другого поля или дата в допустимом формате, — дата должна быть позже, не раньше, раньше или не позже заданной даты;

    public function rules(): array
    {
        return [
            // Дата начала мероприятия...
            'begin_date' => [
                // ...не должна быть раньше сегодняшней даты,
                'after_or_equal:today',
                // ...не должна быть позже даты завершения.
                'before_or_equal:end_date',
                'required',
            ],
            // Дата завершения мероприятия...
            'end_date' => [
                // ...не должна быть раньше даты завершения.
                'after_or_equal:begin_date',
                'required',
            ],
        ];
    }

  • boolean, date, integer, string — значение поля должно принадлежать заданному типу данных;

  • decimal:M,N — значение поля должно быть действительным числом, у которого не менее M и не более N разрядов в дробной части. Например, правилу decimal:0,2 соответствуют числа 1 (нет дробной части), 89.5 (один разряд в дробной части), 199.99 (два разряда в дробной части), но не 1.234 (три разряда в дробной части);

    public function rules(): array
    {
        return [
            // Значение поля price...
            'price' => [
                // ...должно иметь 0, 1 или 2 разряда в дробной части.
                'decimal:0,2',
                'required',
            ],
        ];
    }

  • exists:имя_таблицы,имя_столбца — значение поля должно встречаться в указанном столбце указанной таблицы. Правило пригодится при валидации значения, предназначенного для вставки во внешний ключ. Например, если между категориями и товарами установлено отношение «один ко многим», и с запросом на вставку товара пришёл идентификатор категории, то не лишним будет проверить, существует ли в таблице категорий запись с указанным первичным ключом;

    public function rules(): array
    {
        return [
            // Значение поля category_id...
            'category_id' => [
                // ...должно встречаться в столбце id таблицы categories.
                'exists:categories,id',
                'required',
            ],
        ];
    }

  • file — значение поля должно быть файлом;

  • image — значение поля должно быть графическим файлом;

  • max:N (min:N), где N — натуральное число, — это максимальная (минимальная) длина строки, максимальное (минимальное) значение числа или максимальный (минимальный) размер файла в кибибайтах (1 КиБ = 1024 Б). Правила не применяются, если поля нет в запросе, т. е. отсутствующее поле не считается полем с нулевой длиной;

  • unique:имя_таблицы,имя_столбца — значение поля не должно встречаться в указанном столбце указанной таблицы.

    public function rules(): array
    {
        return [
            // Значение поля label...
            'label' => [
                // ...не должно встречаться в столбце label таблицы tags.
                'unique:tags,label',
                'required',
            ],
        ];
    }

Контроллеры​

Контроллер — модуль (компонент) управления программой. Контроллер реализует операции (т. е. функции) программы. В Laravel контроллеры размещены в каталоге Http, что подчёркивает их зависимость от протокола передачи данных.

В Laravel различают

  • контроллеры для прикладного программного интерфейса (API). В этих контроллерах по умолчанию определены методы для создания (store), корректировки (update), удаления (destroy) и вывода либо перечня ресурсов (index), либо отдельно взятого ресурса (show);
  • контроллеры для пользовательского интерфейса (UI) в дополнение к методам API­‑контроллера содержат методы, предназначенные для вывода HTML­‑форм: create и edit.

Приведём следующие файлы к указанному виду.

Файл app/Http/Controllers/ProductController.php
<?php

namespace App\Http\Controllers;

use App\Http\Requests\StoreProductRequest;
use App\Http\Requests\UpdateProductRequest;
use App\Models\Product;

class ProductController extends Controller
{
    /**
     * Display a listing of the resource.
     */
    public function index()
    {
        $products = Product::all();
        return response($products);
    }

    /**
     * Store a newly created resource in storage.
     */
    public function store(StoreProductRequest $request)
    {
        $data = $request->validated();
        $product = Product::create($data);
        return response()->noContent(201)->withHeaders([
            'Location' => route('products.show', [
                'product' => $product->id,
            ]),
        ]);
    }

    /**
     * Display the specified resource.
     */
    public function show(Product $product)
    {
        return response($product);
    }

    /**
     * Update the specified resource in storage.
     */
    public function update(UpdateProductRequest $request, Product $product)
    {
        $data = $request->validated();
        $product->update($data);
        return response()->noContent(204);
    }

    /**
     * Remove the specified resource from storage.
     */
    public function destroy(Product $product)
    {
        $product->delete();
        return response()->noContent(204);
    }
}
Файл app/Http/Controllers/ImageController.php

Маршруты​

Теперь следует обеспечить клиентам HTTP, в т. ч. браузерам, возможность вызывать методы контроллеров, подавая запросы через стек протоколов Интернета. Для этого мы должны ассоциировать (сопоставить, связать) каждый метод контроллера с некоторым методом HTTP и универсальным идентификатором ресурса (URI) — проще говоря, веб­‑адресом, который указывают в адресной строке браузера, гиперссылке и т. д.

При создании веб­‑адресов мы будем руководствоваться REST — шаблоном проектирования API. Программные системы, реализующие REST, получают определение RESTful. Современные RESTful­‑системы позволяют клиентам манипулировать ресурсами на сервере, подавая запросы по HTTP. При этом метод протокола определяет операцию над ресурсом, а URI идентифицирует ресурс, над которым выполняется операция.

Веб­‑адреса обычно начинаются с существительного в форме множественного числа.

Метод HTTPURIОперация
POST/примеры/Создание примера
GET/примеры/Вывод множества примеров
GET/примеры/{id}/Вывод примера с идентификатором {id}
PATCH/примеры/{id}/Корректировка примера с идентификатором {id}
PUT/примеры/{id}/Замена примера с идентификатором {id}
DELETE/примеры/{id}/Удаление примера с идентификатором {id}

Разумеется, набор операций операций может отличаться от системы к системе. Например, в Laravel контроллер для API не содержит методов для корректировки или удаления множества ресурсов (т. е. ресурсы можно корректировать или удалять только по одному), а контроллер для UI содержит методы для вывода HTML­‑форм.

Приступим к реализации. Для начала создадим группу маршрутов с префиксом /api/v1/. Для этого откроем на редактирование файл routes/api.php и добавим в конец следующие строки:

Файл routes/api.php
Route::prefix('v1')->group(function () {
    //
});

Несложно догадаться, что метод group создаёт группу маршрутов (пока пустую), а метод prefix задаёт всем веб­‑адресам в группе префикс v1. Почему не /api/v1/? Потому что ко всем веб­‑адресам, определённым в файле routes/api.php, префикс /api/ будет добавлен автоматически — об этом сказано в файле app/Providers/RouteServiceProvider.php.

Product​

  1. Спроектируем RESTful API для модели Product:

    Метод HTTPШаблон URIДействие
    POST/api/v1/products/Создать запись о товаре
    GET/api/v1/products/Вывести записи о товарах
    GET/api/v1/products/{product}/Вывести запись о товаре
    PUT/api/v1/products/{product}/Корректировать запись о товаре
    DELETE/api/v1/products/{product}/Удалить запись о товаре

    Шаблонное выражение {product} отмечает место, предназначенное для подстановки идентификатора товара, а именно значения первичного ключа из таблицы БД, slug’а и т. п. Таким образом, шаблону /api/v1/products/{product}/ будут соответствовать веб­‑адреса /api/v1/products/1/, /api/v1/products/2/, /api/v1/products/3/ и т. д.

  2. Отредактируем файл routes/api.php — добавим маршруты в группу.

    Файл routes/api.php
    Route::prefix('v1')->group(function () {
        Route::apiResource('products', ProductController::class);
    });

    Метод Route::apiResource сопоставляет методы index, store, show, update, destroy контроллера ProductController с веб­‑адресами, объединёнными префиксом products, и даёт маршрутам информативные имена: products.index, products.store и т. д.

  3. Подадим команду

    php artisan route:list

    Будет выведен перечень всех маршрутов, среди которых будут маршруты, связанные с контроллером ProductController:

    GET|HEAD  api/v1/products ................ products.index › ProductController@index
    POST      api/v1/products ................ products.store › ProductController@store
    GET|HEAD  api/v1/products/{product} ........ products.show › ProductController@show
    PUT|PATCH api/v1/products/{product} .... products.update › ProductController@update
    DELETE    api/v1/products/{product} .. products.destroy › ProductController@destroy

    Для наглядности представим эти данные в табличном виде:

    Метод HTTPШаблон URIМетод ProductControllerИмя маршрута
    GET или HEAD/api/v1/products/indexproducts.index
    POST/api/v1/products/storeproducts.store
    GET или HEAD/api/v1/products/{product}/showproducts.show
    PUT или PATCH/api/v1/products/{product}/updateproducts.update
    DELETE/api/v1/products/{product}/destroyproducts.destroy
    Запись веб­‑адресов

    Каждый веб­‑адрес можно записать в виде строковой константы или вызова функции с именем маршрута в качестве аргумента. Например,

    '/api/v1/products/'

    и

    route('products.index')

    — это два варианта записи одного и того же веб­‑адреса, ведущего к перечню товаров в БД.

    Если маршрут содержит шаблонное выражение (например, {product}), то функции route нужно передать второй параметр — массив, ключи которого соответствуют имени внутри шаблонного выражения. Например, ссылку на веб­‑страницу, посвящённую товару с первичным ключом 123, можно записать как

    '/api/v1/products/123/'

    или

    route('products.show', ['product' => 123])

    Разумеется, вызов функции всегда предпочтителен.

  4. Запустим отладочный сервер:

    php artisan serve

  5. Проверим все маршруты посредством Postman.

Ресурсы​

В Laravel любой объект или массив, возвращаемый методом контроллера, будет автоматически конвертирован в формат JSON. Веб­‑ресурс получит медиатип application/json.

Создать, скрыть или переименовать свойства объекта, модифицировать их значения позволит специализированный инструмент, который в Laravel назван Resource. С точки зрения шаблона Model — view — controller (MVC) такой инструмент реализует компонент View.

Отдельный Product​

  1. Подадим запрос методом GET на веб­‑адрес http://127.0.0.1:8000/api/v1/products/123/ и рассмотрим тело ответного сообщения.

    {
      "id": 123,
      "title": "Искусство программирования",
      "price": "4999.99",
      "description": "Классическая монография Д. Кнута.",
      "quantity": 46,
      "available": 1,
      "created_at": "2023-04-13T18:22:53.000000Z",
      "updated_at": "2023-04-20T21:30:57.000000Z"
    }

    Как мы видим, Laravel представил объект модели Product в виде объекта JSON. Веб­‑ресурс имеет немало недостатков:

    • данные, извлечённые из БД, внесены непосредственно в корневой объект JSON, что может повлечь конфликт имён при внесении в этот же объект метаданных HATEOAS;
    • значение свойства price должно принадлежать действительному, а не строковому типу данных;
    • значения свойства available бу́лева типа данных заданы не константами false или true, а нулями или единицами — в таком виде их хранит СУБД MySQL, но для JSON это нонсенс;
    • нарушены рекомендации JSON:API, согласно которым имена свойств created_at и updated_at следовало бы записать как createdAt и updatedAt соответственно;
    • возможно, некоторые свойства не потребуются на уровне API.

  2. Подадим команду на создание класса, который будет отвечать за представление отдельно взятых объектов модели Product в виде веб­‑ресурсов формата JSON. Настоятельно рекоменуется составить название класса из имени модели (здесь — Product) и слова Resource.

    php artisan make:resource ProductResource

    Будет создан файл app/Http/Resources/ProductResource.php.

  3. Применим ProductResource. Откроем на редактирование файл app/Http/Controllers/ProductController.php и отредактируем метод show.

    Файл app/Http/Controllers/ProductController.php
    public function show(Product $product)
    {
        return response($product);
        return new ProductResource($product);
    }

  4. Повторно подадим запрос методом GET на веб­‑адрес http://127.0.0.1:8000/api/v1/products/123/ и рассмотрим тело ответного сообщения.

    {
      "data": {
        "id": 123,
        "title": "Искусство программирования",
        "price": "4999.99",
        "description": "Классическая монография Д. Кнута.",
        "quantity": 46,
        "available": 1,
        "created_at": "2023-04-13T18:22:53.000000Z",
        "updated_at": "2023-04-20T21:30:57.000000Z"
      }
    }

    Теперь извлечённая из БД информация размещена под ключом data, что позволяет добавить в ресурс метаинформацию: гиперссылки на узлы API, номера страниц, сообщения об ошибках и т. п. Например, Laravel разместит сообщения об ошибках в свойстве message и т. д.

    При желании можно изменить имя data, добавив в класс ресурса свойство public static $wrap = 'новое_имя_обёртки';.

  5. Теперь приведём в порядок свойства. Начнём с того, что откроем на редактирование файл app/Http/Resources/ProductResource.php и удалим тело метода toArray.

    public function toArray(Request $request): array
    {
        return parent::toArray($request);
    }

  6. Метод toArray должен вернуть массив элементов, ключи которых станут свойствами объекта JSON. Ключи можно назначить произвольно, но мы будем следовать стандарту JSON:API:

    • основные данные выведем под ключом data;
    • в data.type укажем условный тип объекта (скажем, products), а в data.id — идентификатор объекта;
    • выгруженные из БД данные разместим под ключом data.attributes;
    • если возникла ошибка, вместо data выведем errors;
    • в data.links.self приведём ссылку на текущий ресурс.

    Приведём файл app/Http/Resources/ProductResource.php к следующему виду. К свойствам модели, выгруженным из БД, можно обращаться через переменную $this.

    Файл app/Http/Resources/ProductResource.php
    <?php

    namespace App\Http\Resources;

    use Illuminate\Http\Request;
    use Illuminate\Http\Resources\Json\JsonResource;

    class ProductResource extends JsonResource
    {
        /**
         * Transform the resource into an array.
         *
         * @return array<string, mixed>
         */
        public function toArray(Request $request): array
        {
            return [
                'type' => 'products',

                // Используем первичный ключ как идентификатор объекта.
                // Приведём к строковому типу данных, чтобы клиент его не исказил.
                'id' => (string) $this->id,

                'attributes' => [
                    // Оставляем наименование продукта без изменений.
                    'title' => $this->title,

                    // Приводим цену к типу действительных чисел.
                    'price' => (float) $this->price,

                    // Оставляем описание продукта без изменений.
                    'description' => $this->description,

                    // Приводим остаток продукта на складе к целочисленному типу.
                    'quantity' => (int) $this->quantity,

                    // Приводим флаг доступности к бу́леву типу.
                    // Снятые с продажи товары можно отсеивать при выборке из БД.
                    'available' => (bool) $this->available,

                    // Посмотрим, как можно переименовать свойства, приводя их
                    // имена к нижнему верблюжьему регистру (lowerCamelCase).
                    // Однако вряд ли свойства created_at и updated_at нужны в API.
                    // 'createdAt' => $this->created_at,
                    // 'updatedAt' => $this->updated_at,

                    // Добавляем цену в долларах США.
                    // Обменный курс можно получить на сайте Центробанка России.
                    'priceInUSD' => round(4999.99 / 92.7463, 2),
                ],
                'links' => [
                    'self' => route('products.show', ['product' => $this->id]),
                ],
            ];
        }
    }

  7. Снова подадим запрос методом GET на веб­‑адрес http://127.0.0.1:8000/api/v1/products/123/.

    {
      "data": {
        "type": "products",
        "id": "123",
        "attributes": {
          "title": "Искусство программирования",
          "price": 4999.99,
          "description": "Классическая монография Д. Кнута.",
          "quantity": 46,
          "available": true,
          "priceInUSD": 53.91
        },
        "links": {
          "self": "http://127.0.0.1:8000/api/v1/products/123"
        }
      }
    }

    • формат ресурса соответствует стандарту JSON:API;
    • лишние для API свойства created_at и updated_at исключены;
    • значения свойств price и available приведены к надлежащим типам данных;
    • добавлено свойство priceInUSD (цена товара в долларах США).

За операции над веб­‑ресурсом отвечает компонент URI, который называют «запрос» (англ. query). Компонент ограничен слева вопросительным знаком (?), справа — концом URI или октоторпом (#), который начинает следующий компонент URI — фрагмент.

Поиск​

Реализуем поиск по одному или нескольким столбцам таблицы БД.

  1. Спроектируем URI, руководствуясь стандартом JSON:API:
  • /api/v1/products/?search[имя]=значение обеспечивает поиск по одному атрибуту;
  • /api/v1/products/?search[имя_1]=значение_1&search[имя_2]=значение_2&search[имя_N]=значение_N — по неограниченному числу атрибутов.

Примеры:

  • /api/v1/products/?search[title]=Искусство+программирования — наименование товара равно строке «Искусство программирования». В PHP строка запроса превратится в массив

    [
        'search' => [
            'title' => 'Искусство+программирования'
        ]
    ]

  • /api/v1/products/?search[title]=%программ% — наименование содержит подстроку «%программ%». В PHP строка запроса превратится в массив

    [
        'search' => [
            'title' => '%программ%'
        ]
    ]

  • /api/v1/products/?search[title]=Искусство%&search[description]=%Кнут% — наименование начинается со слова «Искусство», описание содержит фамилию «Кнут». В PHP строка запроса превратится в массив

    [
        'search' => [
            'title' => 'Искусство%',
            'description' => '%Кнут%'
        ]
    ]
  1. ...
