Теоретические сведения

Межсайтовая сценаризация (англ. cross­‑site scripting, XSS) — это атака, в рамках которой пользовательский агент (например, браузер) выполняет вредоносный сценарий, внедрённый в уязвимое веб­‑приложение. XSS подвержены все языки разметки, поддерживающие выполнение сценариев: HTML, SVG, XHTML и пр. Вредоносный сценарий обычно пишут на языке программирования JavaScript.

Классификация​

Классификация XSS по способу внедрения:

  • серверные (англ. server­‑side);
  • на основе объектной модели документа (англ. DOM­‑based).

Классификация XSS по способу хранения полезной нагрузки:

  • хранимые (англ. stored) — вредоносный код размещён в хранилище данных уязвимого веб­‑приложения (например, в БД) и выполняется после извлечения;
  • отражённые (англ. reflected) — вредоносный сценарий выполняется в контексте уязвимого веб­‑приложения немедленно, без размещения в хранилище данных.

Примеры​

Пример 1​

Пусть существует социальная сеть, позволяющая публиковать текстовые сообщения. Пользовательский веб­‑интерфейс сети выполнен на HTML. Сообщение при выводе заключено между тегами <div> и </div>.

Злоумышленник, став пользователем указанной сети, публикует сообщение следующего содержания:

<script>alert('XSS');</script>

При выводе такого сообщения сервер выполняет следующий код (извлечение из БД для наглядности заменено на присваивание):

Небезопасный код
<?php
$data = "<script>alert('XSS');</script>";
echo "<div>{$data}</div>";

В результате браузер получает следующий фрагмент разметки:

<div><script>alert('XSS');</script></div>

Текст, размещённый между тегами <script> и </script>, браузер рассматривает как сценарий на языке JavaScript, подлежащий немедленному выполнению. Таким образом, вредоносный сценарий может быть выполнен в браузере каждого пользователя социальной сети, читающего сообщение злоумышленника. Атака является примером хранимого XSS.

Конечно, в рассмотренном случае атака приведёт лишь к созданию модального окна с сообщением «XSS», однако обнаруженный вектор атаки позволит включить в полезную нагрузку гораздо менее безобидный код для манипулирования элементам веб­‑страницы, программными интерфейсами браузера, его адресной строкой и т. д.

Пример 2​

Пусть существует поисковая система в домене search.example. Пользовательский интерфейс системы содержит форму HTML, позволяющую подать поисковый запрос методом HTTP GET. После выполнения запроса браузер отображает веб­‑страницу, на которой размещены результаты поиска и, что особенно важно, поле ввода с искомым текстом (аналогичная страница в «Яндексе», аналогичная страница в Google).

При выводе искомого текста в поле ввода сервер выполняет следующий код:

  • PHP 7
  • PHP
Небезопасный код
<input name="query" value="<?= $_GET['query'] ?? '' ?>">
Это точно PHP?

Да.

  1. Теги <?= и ?> предназначены для вывода содержимого. Начиная с PHP 5.4 они исключены из перечня отключаемых коротких тегов. Фрагмент

    <?= $_GET['query'] ?? '' ?>

    эквивалентен фрагменту

    <?php echo $_GET['query'] ?? ''; ?>

  2. Бинарный оператор слияния с null (??), введённый в PHP 7.0.0, возвращает левый операнд, если он не равен null, или правый операнд — в противном случае. ?? — это синтаксический сахар, сочетающий в себе возможности функции isset и тернарного оператора.

    Таким образом, фрагмент с оператором слияния с null

    <?= $_GET['query'] ?? '' ?>

    экивалентен фрагменту с тернарным оператором

    <?php
    echo isset($_GET['query']) ? $_GET['query'] : '';
    ?>

    и фрагменту с условным оператором

    <?php
    if (isset($_GET['query'])) {
        echo $_GET['query'];
    } else {
        echo '';
    }
    ?>
Небезопасный код
<input name="query" value="<?php
    if (isset($_GET['query'])) {
        echo $_GET['query'];
    }
?>">

Злоумышленник подготавливает фрагмент разметки HTML с внедрённым сценарием JavaScript.

" autofocus onfocus="alert('XSS')

Затем злоумышленник фабрикует гиперссылку, в которой вредоносный фрагмент задан как искомый текст:

<a href="https://search.example/?query=%22+autofocus+onfocus%3D%22alert%28%27XSS%27%29">
    Жми
</a>

Если пользователь перейдёт по такой гиперссылке, то поисковая система вставит искомый текст в поле ввода, и браузер получает следующий фрагмент разметки:

<input name="query" value="" autofocus onfocus="alert('XSS')">

Браузер воспримет кавычку перед autofocus как закрывающую атрибут value, токены autofocus и onfocus — как имена атрибутов, а alert('XSS') — как значение атрибута onfocus. Кавычка, которая должна была закрыть атрибут value, закроет атрибут onfocus. Благодаря атрибуту autofocus поле ввода автоматически получит фокус, сработает обработчик onfocus, и браузер выведет модальное окно с сообщением «XSS».

Атака является примером отражённого XSS. Отличие от хранимого XSS заключается в том, что внедрение в веб­‑приложение (при запросе) и немедленное выполнение вредоносного сценария (при обработке ответа на этот запрос) осуществляет браузер жертвы.

Методы предотвращения​

Кодирование​

Модель предотвращения XSS, разработанная организацией OWASP, рассматривает документ HTML как шаблон с реквизитами (ячейками) для вывода данных. Ассортимент ячеек ограничен белым списком, причём для каждой их разновидности установлены индивидуальные правила валидации, экранирования и кодирования.

Все данные, попадающие в систему из внешних источников (неважно, из БД или браузера), считаются опасными по умочанию.

Кодирование содержимого элемента​

Знаки, указанные в таблице, следует замещать ссылками в данных, выводимых между начальным и конечным тегами элемента с потоковым (англ. flow) или фразовым (англ. phrasing) содержимым, например <h1>здесь</h1> и <p>здесь</p>.

"U+0022 QUOTATION MARK&quot;
&U+0026 AMPERSAND&amp;
'U+0027 APOSTROPHE&#x27; или &#039;
<U+003C LESS-THAN SIGN&lt;

Именованная ссылка &apos; на U+0027 есть только в XML 1.0 и выше, XHTML 1.0 и HTML5, а ссылка &sol; на U+002F появилась только в HTML5, поэтому эти знаки в целях обеспечения обратной совместимости следует представлять не именованными, а шестнадцатеричными числовыми ссылками [HTML 4.01, пункт 24.2.1; HTML5, подраздел 8.5; XHTML 1.0, раздел A.2; XML 1.0, раздел 4.6].

Защитим социальную сеть, обеспечив кодирование пользовательских сообщений перед выводом.

  • PHP
  • PHP 8.1.0
  • Laravel
<?php
$data = "<script>alert('XSS');</script>";
$encodedData = htmlspecialchars($data, ENT_QUOTES | ENT_SUBSTITUTE | ENT_HTML401);
// Без ENT_QUOTES функция не кодирует U+0027 APOSTROPHE.
echo "<div>$encodedData</div>";
<?php
$data = "<script>alert('XSS');</script>";
$encodedData = htmlspecialchars($data);
echo "<div>$encodedData</div>";

Встроенный в Laravel шаблонизатор Blade предлагает парный оператор {{...}}, который автоматически передаёт содержимое в функцию PHP htmlentities. Она похожа на htmlspecialchars, но кодирует все знаки, для которых существует именованная ссылка.

<div>{{ $data }}</div>

Теперь переданный злоумышленником вредоносный сценарий

<script>alert('XSS');</script>

перед отправкой клиенту будет закодирован сервером как

&lt;script&gt;alert(&#039;XSS&#039;);&lt;/script&gt;

В итоге в браузер придёт последовательность

<div>&lt;script&gt;alert(&#039;XSS&#039;);&lt;/script&gt;</div>

Благодаря кодированию тегов <script> и </script> браузер воспримет их содержимое не как сценарий JavaScript, а как обычный текст, и отобразит его на экране без выполнения.

<script>alert('XSS');</script>

Кодирование значения атрибута​

Большинству атрибутов вовсе не следует присваивать значения, полученные из внешнего источника. Исключение составляют атрибуты, условно называемые безопасными:

  • alt (элементы <area>, <img>, <input>) — текст;
  • color (элемент <link>) — идентификатор цвета CSS;
  • cols, rows (элемент <textarea>) — натуральное число;
  • colspan, rowspan (элементы <td>, <th>) — целое неотрицательное число;
  • coords (элемент <area>) — последовательность вещественных чисел;
  • height, width (элементы <canvas>, <embed>, <iframe>, <img>, <input>, <object>, source внутри <picture>, video) — целое неотрицательное число;
  • rel (элементы <a>, <area>, <link>);
  • shape (элемент <area>);
  • span (элементы <col>, <colgroup>) — натуральное число;
  • usemap (элемент <img>);
  • value (элементы <button>, <data>, <input>, <li>, <meter>, <option>, <progress>) — текст или число;
  • class, dir, lang, tabindex, title (все элементы).

Требования к кодированию зависят от типа данных, но общее правило звучит так: в значениях «безопасных» атрибутов следует замещать шестнадцатеричными ссылками (&#xH;, где H — кодовая позиция) все знаки, за исключением арабских цифр [U+0030..U+0039], прописных [U+0041..U+005A] и строчных [U+0061..U+007A] букв английского алфавита [OWASP].

Одна из реализаций этого правила — фильтр html_attr в шаблонизаторе Twig, который интегрирован в каркас программной системы Symfony.

Причина столь агрессивного кодирования — требования стандарта HTML5 к значениям, приведённым без кавычек. Помимо знаков разметки (", ', <, >), такие значения не должны содержать пробел (U+0020), знак «равно» (U+003D), гра́вис (U+0060) и некоторые управляющие символы: горизонтальную табуляцию (U+0009), перевод формата U+000C), а также U+000A и U+000D, которые применяют для разделения строк. Все вышеперечисленные знаки злоумышленники используют для вывода вредоносного кода из контекста значения атрибута в контекст элемента, его содержимого или его родителя. Кроме того, кодирования требует амперса́нд («&»; код U+0026).

Итак, защитим поисковую систему, обеспечив кодирование искомого текста перед выводом.

<?php
function encode_html_attribute_value($value)
{
    $result = preg_replace_callback('/[^0-9A-Za-z]/Su', function ($matches) {
        $character = $matches[0];
        $codePoint = mb_ord($character);
        return sprintf('&#x%X;', $codePoint);
    }, $value);
    if (is_null($result)) {
        throw new InvalidArgumentException('');
    }
    return $result;
}
?>
<input name="query" value="<?=
    isset($_GET['query']) ? encode_html_attribute_value($_GET['query']) : ''
?>">

Функция preg_replace_callback принимает три аргумента: регулярное выражение, функцию обратного вызова и строку. Регулярное выражение /[^0-9A-Za-z]/Su находит в строке все знаки, кроме 0–⁠9, A–⁠Z и a–⁠z. Функция обратного вызова заменяет каждый из найденных знаков последовательностью &#x%X;, где %X — шестнадцатеричное представление кодовой позиции знака. Например,

  • &#x9; — горизонтальная табуляция (U+0009);
  • &#x22;, &#x26;, &#x27;, &#x3C;, &#x3E; — знаки ", &, ', <, > соответственно;
  • &#x44F; — русская буква «я» (U+044F).

Следовательно, введённый злоумышленником искомый текст

" autofocus onfocus="alert('XSS')

будет закодирован как

&#22;&#20;autofocus&#20;onfocus&#3D;&#22;alert&#28;&#27;XSS&#27;&#29;

и будет размещён в поля ввода как обычный текст:

<input
  name="query"
  value="&#22;&#20;autofocus&#20;onfocus&#3D;&#22;alert&#28;&#27;XSS&#27;&#29;"
  >

Общие рекомендации​

Проект OWASP предостерегает от вывода ненадёжных данных в других контекстах:

  • между начальным и конечным тегами элементов script и style (<script>здесь</script> и <style>здесь</style>);
  • между начальным и конечным ограничителями комментария (<!--здесь-->);
  • в имя элемента (<здесь>);
  • в имя атрибута (<element здесь>);
  • во вложенный контекст, такой как URL внутри документа HTML.

К общим можно отнести рекомендацию не использовать синтаксические послабления HTML5 и всегда заключать значение атрибута в суррогатные кавычки.

Явно указывайте кодировку UTF‑8 до вывода пользовательских данных, чтобы предотвратить атаки с переключением кодировки (как это было в Internet Explorer с UTF‑7).

  • HTML5
<!DOCTYPE html>
<html lang="ru-luna1918">
<head>
  <meta charset="UTF-8">
  <!-- Данные из ненадёжных источников выводим ниже. -->
  <title><?= htmlspecialchars($title) ?></title>
</head>
<body>
  <?= htmlspecialchars($body) ?>
</body>
</html>

X-XSS-Protection​

В некоторые пользовательские агенты (браузеры), выпущенные в 2009–2019 гг., встроены компоненты для пресечения отражённых XSS: в Internet Explorer 8–⁠11 и Edge 12–⁠17 — XSS Filter, в Chrome 4.0.249–⁠77 — XSS Auditor. Эти компоненты проверяли текст, который, будучи переданным клиентом на сервер через URL или тело HTTP­‑запроса, был «отражён» сервером и возвращён клиенту в теле HTTP­‑ответа. Если такой текст напоминал полезную нагрузку для XSS, т. е. содержал теги script, кавычки с методами DHTML и пр., то компонент, в зависимости от настроек, либо вырезал подозрительный фрагмент, либо полностью блокировал отображение веб­‑страницы.

Такими компонентами можно было управлять через X-XSS-Protection — нестандартизованное заголовочное поле ответного (отправляемого сервером клиенту) сообщения HTTP. У поля три возможных общеупотребительных значения: 0 — фильтр выключен; 1 — фильтр включён, и небезопасные фрагменты сценариев будут удалены; 1; mode=block — фильтр включён, и при обнаружении XSS отображение веб­‑страницы будет блокировано (по умолчанию в Chrome 57–⁠73).

К сожалению, фильтрующие компоненты оказались надёжными средствами защиты. Причины таковы:

  • отсутствие аналогов в Firefox;
  • ложные срабатывания, как в том случае, когда сценарий во входных данных идентичен сценарию в выходных данных, но никак с ним не связан;
  • уязвимости, уязвимости;
  • невозможность управления в некоторых версиях браузеров.

Конечно, XSS Auditor и XSS Filter могут защитить от XSS устаревшие браузеры Chrome 4.0.249–⁠13 и Internet Explorer 8–⁠11, в которых не реализован или не полностью реализован стандарт Content Security Policy 1.0. Но эти же компоненты могут создать уязвимости в других браузерах. Поэтому, как ни прискорбно, следует выключить встроенную защиту, отправив в ответе HTTP заголовочное поле X-XSS-Protection: 0, и опереться на кодирование данных как на универсальную защиту от XSS.

  • Apache
  • nginx
  • PHP
<VirtualHost *:80>
    ServerName доменное.имя.сайта
    DocumentRoot "путь/к/каталогу/с/сайтом"
    Header always set X-XSS-Protection "0"
</VirtualHost>
server {
    listen 80;
    listen [::]:80;
    root "путь/к/каталогу/с/сайтом";
    server_name доменное.имя.сайта;

    add_header X-XSS-Protection "0" always;
}
<?php
header('X-XSS-Protection: 0');

Content Security Policy​

Если запретить выполнение встроенных в веб­‑страницу сценариев, то и XSS станет невозможен.

Таким образом, разработчикам не следует указывать unsafe-inline или base: в качестве допустимых источников в политиках.

<meta
 http-equiv="Content-Security-Policy"
 content="default-src 'none'; script-src-elem 'self'; style-src-elem 'self'"
>
