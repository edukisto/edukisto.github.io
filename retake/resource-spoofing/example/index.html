Пример выполнения задания

Подготовка​

  1. Создадим каталоги cdn.example, evil.example, good.example и файлы следующего содержания:

    Файл cdn.example/script.js
    window.addEventListener('load', function () {
      window.alert('Штатный JavaScript!');
    });
    Файл cdn.example/style.css
    body {
      background-color: green;
    }
    Файл evil.example/script.js
    window.addEventListener('load', function () {
      window.alert('Вредоносный JavaScript!');
    });
    Файл evil.example/style.css
    body {
      background-color: red;
    }
    Файл good.example/index.html
    <!DOCTYPE html>
    <html lang="ru-luna1918">
    <head>
      <meta charset="UTF-8">
      <title>Сайт, уязвимый для subresource spoofing</title>
      <link rel="stylesheet"
            href="http://cdn.example/style.css">
      <script defer
              src="http://cdn.example/script.js"></script>
    </head>
    <body>
      <article>
        <h1>Сайт, уязвимый для <i lang="en">subresource spoofing</i></h1>
      </article>
    </body>
    </html>

  2. Подготовим исходный код веб­‑сайтов.

    • Windows 7 SP 1, 11
    • Ubuntu
    1. Запустим контрольную панель XAMPP.
    2. Нажмём на кнопку Explorer. Будет открыта папка, в которую установлен XAMPP.
    3. Откроем папку htdocs и перенесём в неё каталоги cdn.example, evil.example и good.example.

    Скопируем каталоги cdn.example, evil.example и good.example в каталог /var/www/.

    sudo cp -R cdn.example /var/www/
    sudo cp -R evil.example /var/www/
    sudo cp -R good.example /var/www/

  3. Создадим виртуальные хосты.

    • Windows 7 SP 1, 11
    • Ubuntu

    Найдём в папке, в которую установлен XAMPP, файл apache\conf\extra\httpd-vhosts.conf? добавим в него следующую конфигурацию и сохраним изменения.

    Файл apache\conf\extra\httpd-vhosts.conf в папке XAMPP
    <VirtualHost 127.0.0.1:80>
        ServerName cdn.example
        DocumentRoot "C:\xampp\htdocs\cdn.example"
        Header always set Access-Control-Allow-Origin "*"
    </VirtualHost>

    <VirtualHost 127.0.0.2:80>
        ServerName evil.example
        DocumentRoot "C:\xampp\htdocs\evil.example"
        Header always set Access-Control-Allow-Origin "*"
    </VirtualHost>

    <VirtualHost 127.0.0.3:80>
        ServerName good.example
        DocumentRoot "C:\xampp\htdocs\good.example"
    </VirtualHost>

    1. Подготовим файлы cdn.example.conf, evil.example.conf и good.example.conf.

      Файл cdn.example.conf
      <VirtualHost 127.0.0.1:80>
          ServerName cdn.example
          DocumentRoot "/var/www/cdn.example/"
          Header always set Access-Control-Allow-Origin "*"
      </VirtualHost>
      Файл evil.example.conf
      <VirtualHost 127.0.0.2:80>
          ServerName evil.example
          DocumentRoot "/var/www/evil.example/"
          Header always set Access-Control-Allow-Origin "*"
      </VirtualHost>
      Файл good.example.conf
      <VirtualHost 127.0.0.3:80>
          ServerName good.example
          DocumentRoot "/var/www/good.example/"
      </VirtualHost>

    2. Разместим файлы cdn.example.conf, evil.example.conf и good.example.conf в каталоге /etc/apache2/sites-available/.

    sudo cp cdn.example.conf /etc/apache2/sites-available/
    sudo cp evil.example.conf /etc/apache2/sites-available/
    sudo cp good.example.conf /etc/apache2/sites-available/
    «\» или «/»?

    Символ, разделяющий имена элементов файловой системы в записи пути, зависит от операционной системы (ОС):

    • в ОС, ориентируемых на семейство стандартов POSIX, в т. ч. Linux и macOS, используют косую черту («/»; англ. slash);

    • в Windows основным разделителем является обратная косая черта («\»; англ. backslash), а дополнительным (альтернативным) — обычная косая черта.

    Путь к каталогу (папке) принято завершать этим же символом, например: C:\Windows\, /etc/ и т. п. Однако обратная косая черта в конце любой строки, обозначающей путь в конфигурационном файле Apache, является синтаксической ошибкой. Это правило касается в частности директивы DocumentRoot:

    AH00526: Syntax error... DocumentRoot must be a directory

    Исправить ошибку можно двумя способами:

    • удалить обратную косую черту («\»), размещённую в конце строки;
    • заменить все обратные косые черты́ («\») в строке на обычные косые черты́ («/»).

  4. Перезапустим Apache после правки конфигурации.

    • Windows 7 SP 1, 11
    • Ubuntu
    1. Переключимся на контрольную панель XAMPP.
    2. Если напротив метки Apache видна кнопка Stop, то нажмём на неё и дождёмся останова сервера.
    3. Нажмём на кнопку Start, размещённую напротив метки Apache.
    # Включаем модуль заголовков.
    sudo a2enmod headers

    # Активируем сайты.
    sudo a2ensite cdn.example.conf
    sudo a2ensite evil.example.conf
    sudo a2ensite good.example.conf

    # Заставляем Apache заново считать конфигурацию.
    sudo systemctl reload apache2

  5. Откроем на редактирование файл, соотносящий IP­‑адреса с доменными именами.

    • Windows 7 SP 1, 11
    • Ubuntu

    1. Запустим приложение «Командная строка» (не «Терминал»!) от имени администратора.

    2. Подадим команду

      Notepad "%SystemRoot%\System32\drivers\etc\hosts"

      Соответствующий шаблону файл (обычно это C:\Windows\System32\drivers\etc\hosts) будет открыт в текстовом редакторе «Блокнот».

    1. Запустим эмулятор терминала.

    2. Подадим команду

      sudo nano /etc/hosts

      Файл /etc/hosts будет открыт в текстовом редакторе nano, запущенном от имени суперпользователя.

  6. Добавим в конец файла:

    127.0.0.1 cdn.example
    127.0.0.2 evil.example
    127.0.0.3 good.example

  7. Сохраним изменения и закроем текстовый редактор.

  8. Подадим команды:

    ping cdn.example
    ping evil.example
    ping good.example

    При обмене пакетами с указанными узлами вычислительной сети должны быть выведены IP­‑адреса 127.0.0.1, 127.0.0.2 и 127.0.0.3 (соответственно).

    ping вместо nslookup

    Утилита nslookup в Windows (в отличие от Linux) обращается к DNS, не учитывая записи из файла hosts, поэтому она не подойдёт для наших экспериментов.

  9. Откроем в браузере вкладки с http://cdn.example/, http://evil.example/ и http://good.example/.

    По веб­‑адресу http://good.example/ браузер должен отобразить веб­‑страницу с зелёным фоном и вывести модальное окно с сообщением «Штатный JavaScript!» Это свидетельство того, что содержимое подресурсов script.js и style.css загружено с http://cdn.example/.

    403 Forbidden?

    Если браузер вывел сообщение Forbidden, то, скорее всего, в конфигурации Apache неправильно указан DocumentRoot.

    Не забываем перезапустить Apache после правки конфигурации.

Атака​

Выступим в роли злоумышленника и осуществим подмену подресурса через отравление DNS (англ. DNS poisoning).

  1. Изучив разметку ресурса http://good.example/index.html, установим, что его подресурсы размещены на cdn.example.

  2. Откроем на редактирование файл, соотносящий IP­‑адреса с доменными именами (см. выше) и заменим IP­‑адрес домена cdn.example на тот, который находится в распоряжении злоумышленника (127.0.0.2).

    127.0.0.1 cdn.example
    127.0.0.2 cdn.example
    127.0.0.2 evil.example
    127.0.0.3 good.example
    Внимание!

    DNS poisoning может осуществить не только провайдер, но и вирус на компьютере жертвы.

  3. Изменим конфигурацию на веб­‑сайте злоумышленника таким образом, чтобы он был готов к обработке запросов, адресованных домену cdn.example.

    <VirtualHost 127.0.0.2:80>
        ServerName evil.example
        ServerName cdn.example
        DocumentRoot "C:\xampp\htdocs\evil.example"
        Header always set Access-Control-Allow-Origin "*"
    </VirtualHost>

  4. Подадим команды:

    ping cdn.example

    При обмене пакетами с cdn.example должен быть выведен IP­‑адрес 127.0.0.2.

  5. Обновим в браузере вкладку с http://good.example/.

    Теперь браузер должен отобразить веб­‑страницу с красным фоном и вывести модальное окно с сообщением «Вредоносный JavaScript!» Это свидетельство того, что теперь браузер загружает подресурсы не с cdn.example, а с сайта злоумышленника.

    Ничего не происходит?

    Браузеры не всегда обновляют кеш по первому запросу. Следует принудительно обновить кеш, нажав ⇧ Shift + F5.

Реализация Subresource integrity​

  1. Перейдём в каталог cdn.example, в котором размещены штатные подресурсы.

    cd cdn.example

  2. Рассчитаем контрольную сумму файла JavaScript.

    openssl dgst -sha256 -binary script.js | openssl enc -e -base64 -A
    Подсказка

    Утилита openssl есть в XAMPP.

    Левая часть выражения хеширует содержимое файла по алгоритму SHA‑256, а правая — кодирует полученный хеш по схеме base64 согласно RFC 4648. Флаг -A заставляет выводить сообщение без разрывов строк.

    Будет выведено примерно следующее:

    F6XhIMakQg+3KFmDWhFgvU8WNupEZIdsa7MmBiB3lUo=

  3. Рассчитаем контрольную сумму файла CSS.

    openssl dgst -sha256 -binary style.css | openssl enc -e -base64 -A

    Будет выведено примерно следующее:

    +eq9BFFTx3IupdZ9ibxCJup4eRmiNa+7ddBxnJR92p0=
    SHA‑256, SHA‑384 или SHA‑512?

    Алгоритм SHA‑512, вероятно, наиболее устойчив к коллизиям, что может быть полезно в будущем при развитии квантовых компьютеров, и в среднем быстрее SHA‑256 на 64­‑битных процессорах (доказательство: openssl speed sha256 sha512).

    С другой стороны, хеши SHA‑256 в два раза короче хешей SHA‑512, что позволяет экономить сетевой трафик. Убедимся в этом.

    Длину контрольной суммы, которая является представлением хеша в base64, можно определить по формуле:

    ⌊c+c−6×⌊c6⌋3⌋×4,\Bigg \lfloor \frac{c + c - 6 \times \big \lfloor \frac{c}{6} \big \rfloor}{3} \Bigg \rfloor \times 4,⌊3c+c−6×⌊6c​⌋​⌋×4,

    где ccc — разрядность в байтах.

    Таким образом, длина контрольной суммы на основе SHA‑512 и SHA‑256 составит 88 и 44 Б соответственно. Это 44 ГБ сэкономленного трафика на каждый миллион пользователей.

  4. Перейдём в каталог good.example, откроем на редактирование файл index.html и приведём теги, используемые для подключения подресурсов, к следующему виду:

    Пример подключения CSS­‑подресурса
    <link
      rel="stylesheet"
      href="http://cdn.example/style.css"
      integrity="sha256-+eq9BFFTx3IupdZ9ibxCJup4eRmiNa+7ddBxnJR92p0="
      crossorigin="anonymous"
      >
    Пример подключения JavaScript­‑подресурса
    <script
      defer
      src="http://cdn.example/script.js"
      integrity="sha256-F6XhIMakQg+3KFmDWhFgvU8WNupEZIdsa7MmBiB3lUo="
      crossorigin="anonymous"
      ></script>

    Обратите внимание, что контрольной сумме предшествует префикс, указывающий на используемый алгоритм хеширования (sha256-).

  5. Обновите содержимое вкладки. Ресурсы, подменённые злоумышленником, должны быть заблокированы, поскольку вычисленные для них контрольные суммы не совпадают с указанными в атрибутах integrity.

    Failed to find a valid digest in the 'integrity' attribute for resource 'http://cdn.example/script.js' with computed SHA-256 integrity 'oMye660nZ4++PK0o/8usQUB3F8REiA79uEmrt5IoEHQ='. The resource has been blocked.
    Failed to find a valid digest in the 'integrity' attribute for resource 'http://cdn.example/style.css' with computed SHA-256 integrity 'One6+Ijl7vGRK8CqU4cfV1me+424AcvPLhuuiRTWQik='. The resource has been blocked.
