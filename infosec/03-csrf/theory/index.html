Теоретические сведения

Общие сведения о CSRF​

Подделка межсайтового запроса (англ. cross­‑site request forgery), известная также как CSRF /ˌsi ˈsɜrf/, XSRF, one­‑click attack и session riding, — это атака, во время которой пользовательский агент (например, браузер) обращается с несанционированным запросом к веб­‑ресурсу, на котором пользователь авторизован.

В отличие от XSS, эксплуатирующего доверие пользователя к ссылке или сайту, CSRF эксплуатирует доверие сервера клиенту.

Пример 1​

Пусть пользователи вымышленной социальной сети Fakebook имеют возможность поставить под тем или иным сообщением отметку «Нравится!». Для этого клиент (браузер) подаёт на ресурс https://fakebook.example/like/ запрос с идентификатором выбранного сообщения (message_id).

Злоумышленник в целях «накрутки» рейтинга сообщения № 1337 заманивает авторизованных пользователей Fakebook на заранее подготовленную веб­‑страницу https://evil.example/trap.html, с которой автоматически, без участия пользователя подаётся запрос на https://fakebook.example/like/. Содержимое вредоносного сценария, проводящего CSRF, может выглядеть примерно так:

<!-- Форма не имеет визуального представления... -->
<form action="https://fakebook.example/like/" method="post">
  <!-- ...а поле ввода скрыто. -->
  <input type="hidden" name="message_id" value="1337">
</form>

<!-- Сценарий JavaScript отправляет форму без участия пользователя. -->
<script>window.onload = document.forms[0].submit();</script>

Пример 2​

Пусть пользователи электронной платёжной системы имеют возможность осуществлять денежные переводы. При переводе авторизованный клиент (браузер) подаёт запрос на ресурс https://bank.example/transfer/?account=R&amount=A, где R — номер счёта получателя, A — размер перевода.

Для похищения денежных средств достаточно осуществить обращение авторизованного клиента по веб­‑адресу (URI), содержащему номер счёта получателя и произвольное положительное число в качестве суммы перевода. URI мог бы выглядеть примерно так:

<a href="https://bank.example/transfer/?account=260602&amount=100500">
  Скачать бесплатно, без регистрации и SMS!
</a>

Установка веб­‑страницы с URI https://bank.example/transfer/?account=260602&amount=100500 в качестве стартовой (домашней), а равно автоматическое восстановление вкладок приведут к повторному списанию денежных средств при каждом запуске браузера.

Противодействие CSRF​

Общие рекомендации разработчикам​

  1. Клиент (например, браузер), подавая запрос по HTTP, предлагает серверу применить один из методов протокола. В рассмотренной электронной платёжной системе клиент использует метод GET для создания записи о денежном переводе и тем самым нарушает протокол, поскольку GET определён в стандарте как безопасный и, следовательно, идемпотентный метод, предназначенный для получения клиентом содержимого того ресурса, на который указывает URI. Другими словами, клиент, предлагая серверу применить GET, не ожидает создания целевого ресурса и полагает, что повторные идентичные запросы также не дадут такого эффекта.

    Если бы разработчик использовал неидемпотентный метод POST, предназначенный именно для создания ресурсов на сервере, то для CSRF было бы недостаточно гиперссылки — потребовалась хотя бы форма.

  2. Допускается требовать подтверждения для наиболее важных операций (например, высылая проверочный код на телефонный номер пользователя), но не злоупотребляйте этим методом: он не только не решает проблему CSRF коренным образом, но и досаждает пользователям.

  3. Можно внедрить в каждый URI идентификатор, связанный с пользователем, от имени которого подаётся запрос. URI https://fakebook.example/like/ и https://bank.example/transfer/ универсальны для всех пользователей — очевидно, идентификатор пользователя, ставящего отметку «Like!», и номер счёта отправителя денежного перевода извлекаются сервером из cookie, и злоумышленник не обязан их знать. Как только URI приобретут вид https://fakebook.example//имя­‑пользователя­‑ставящего­‑отметку/like/ и https://bank.example/номер­‑счёта­‑отправителя/transfer/, массовый CSRF станет невозможным, и злоумышленнику придётся сосредоточиться на эксплуатации только известных ему учётных записей.

  4. Следует ограничить 15–⁠20 минутами сеанса работы с веб­‑приложением в условиях бездействия пользователя. Конечно, для социальной сети это требование чрезмерно, но для онлайн­‑банков оно приемлемо.

Определение источника происхождения запроса по заголовку Referer​

Заголовочное поле Referer в HTTP­‑запросе содержит абсолютный (без фрагмента, т. е. без сегмента, расположенного после знака #) или относительный URI ресурса, с которого подан запрос. Следовательно, Referer позволит отклонить запросы, поданные из других доменов.

  1. Клиент запрашивает у сайта evil.example содержимое файла trap.html:

    GET /trap.html HTTP/1.1
    Host: evil.example

  2. Сайт злоумышленника отправляет клиенту содержимое файла trap.html. В документе размещена HTML­‑форма с полями account (номер счёта получателя) и amount (сумма денежного перевода), а также средства JavaScript для автоматического предъявления формы сайту bank.example:

    HTTP/1.1 200 OK
    Date: Tue, 19 Jan 2038 03:14:08 GMT
    Content-Length: 212

    <form action="https://bank.example/transfer/" method="post">
      <input name="account" value="260602">
      <input name="amount" value="100500">
    </form>
    <script>window.onload = document.forms[0].submit();</script>

  3. Получив содержимое файла trap.html, клиент автоматически предъявляет заполненную форму сайту bank.example и указывает https://evil.example/trap.html как значение заголовочного поля Referer.

    POST /transfer/ HTTP/1.1
    Host: bank.example
    Referer: http://evil.example/trap.html
    Content-Length: 28

    account=260602&amount=100500

  4. Веб­‑сайт bank.example сопоставляет значение поля Referer со своим доменным именем.

    function check_referrer()
    {
        $referrer = $_SERVER['HTTP_REFERER'];
        return
            false !== filter_var($referrer, FILTER_VALIDATE_URL)
            && parse_url($referrer, PHP_URL_HOST) === $_SERVER['SERVER_NAME'];
    }

  5. Веб­‑сайт bank.example отказывается выполнять запрос:

    HTTP/1.1 403 Forbidden

Однако эксплуатацию Referer затрудняет ряд других технических ограничений, также лежащих вне области контроля веб­‑разработчика:

  • Referer отсутствует в запросах, отправная точка которых не имеет URI: ссылающийся ресурс получен пользовательским агентом по схеме file или data; запрос стал результатом ручного ввода в адресную строку; запрос стал результатом манипуляций с window.location; осуществлён переход по гиперссылке, присланной по электронной почте и т. п.;
  • поля Referer нет в запросах по схеме https;
  • поле Referer может быть очищено или удалено некоторыми аппаратными и программными средствами: межсетевыми экранами, проксирующими серверами, расширениями браузера;
  • некоторые версии браузеров и встраиваемых сред выполнения (Flash Player, JVM и пр.) уязвимы для подмены значения Referer (англ. referrer spoofing).

Смягчение на основе токенов​

Смягчение CSRF посредством токена (англ. token­‑based CSRF mitigation) подразумевает использование токена — конфиденциального проверочного кода, создаваемого криптографически стойким генератором псевдослучайных чисел (далее — КСГПСЧ) для каждого сеанса. (В особо важных веб­‑приложениях, например онлайн­‑банках, токены генерируют под каждый запрос, хотя преимущество этого подхода не доказано.)

Каждый запрос, который может повлечь изменение состояния сервера, должен содержать две копии токена:

  • один экземпляр рекомендуется размещать в скрытом поле формы, а при использовании технологии AJAX — в метаданных документа HTML или в параметрах запроса. Ни в коем случае не размещайте токен в URI: во­‑первых, браузер вносит их в журнал посещений, а во­‑вторых, пользователь может случайно скопировать URI вместе с токеном и передать его третьим лицам;
  • другой экземпляр должен быть размещён в cookie или, как вариант, в сеансовых данных, к которым сервер может получить доступ через cookie с идентификатором сеанса.

Приняв запрос, сервер сравнивает два токена. Если один из них отсутствует, сервер отклоняет запрос и направляет клиенту ответ с состоянием 403 Forbidden («Запрещено»). Защита построена на том, что злоумышленник, подделывая запрос, не сможет угадать правильный токен.

Естественно, метод требует, чтобы веб­‑приложение было неуязвимо для XSS и (желательно) шифровало трафик.

Известны три способа формирования токена [@OWASP/OCSS/CSRF]:

  • шаблон токена­‑синхронизатора (англ. synchronizer token pattern);

  • шаблон токена, основанного на шифровании (англ. encryption­‑based token pattern). В качестве алгоритма шифрования OWASP рекомендует AES‑256‑GCM. Пример реализации шаблона на PHP 7.1 и OpenSSL 1.0.1:

    encryption_based_csrf_token_manager.php


    aes256gcm.php


  • шаблон токена, основанного на HMAC (англ. HMAC­‑based token pattern), предполагает размещение в токене срока окончания сеанса и кода аутентификации сообщения на основе хеша. Код задан криптографической хеш­‑функцией, например HMAC‑SHA‑512‑256 или HMAC‑SHA3‑256, от секретного ключа и сообщения — идентификатора сеанса и той же метки времени. Секретный ключ хранится на сервере. Пример реализации шаблона на PHP:

    hmac_based_csrf_token_manager.php
