Пример выполнения задания

Документирование API​

Установка L5 Swagger​

  1. Перейдём в каталог проекта.

    cd путь/к/каталогу/

  2. Подадим команду:

    composer require darkaonline/l5-swagger

    Должно быть выведено:

    Using version ^8.5 for darkaonline/l5-swagger

  3. Подадим команду:

    php artisan vendor:publish --provider="L5Swagger\L5SwaggerServiceProvider"

    Будут созданы

    • файл config/l5-swagger.php;
    • каталог resources/views/vendor/l5-swagger/.

  4. Изучим и отредактируем файл config/l5-swagger.php.

    Ознакомившись с конфигурацией, сделаем вывод, что

    • сгенерированная документация будет опубликована по адресу api/documentation (см. пункт documentations.default.routes.api);
    • сгенерированные корневые файлы будут размещены в каталоге storage_path('api-docs'), что по умолчанию равнозначно storage/api-docs/ (см. пункт defaults.paths.docs).

    Стандарт OAS рекомендует назначить корневому документу OpenAPI имя openapi.json или openapi.yaml [OAS, разд. 4.3]. Найдём строки

    'docs_json' => 'api-docs.json',
    'docs_yaml' => 'api-docs.yaml',
    'generate_yaml_copy' => env('L5_SWAGGER_GENERATE_YAML_COPY', false),

    и заменим на

    'docs_json' => 'openapi.json',
    'docs_yaml' => 'openapi.yaml',
    'generate_yaml_copy' => env('L5_SWAGGER_GENERATE_YAML_COPY', true),

  5. На этапе отладки удобно обновлять документ OpenAPI при каждом обращении к документации через браузер, без подачи соответствующей команды через консоль. В конец файла .env добавим:

    L5_SWAGGER_GENERATE_ALWAYS=true

    Учтём, что в этом режиме не видны сообщения о причинах ошибок.

    Разумеется, этот файл .env не будет использоваться на этапе эксплуатации.

  6. Убедимся, что появился маршрут, по которому будет опубликована документация на API.

    php artisan route:list

    Интересующий нас веб­‑адрес должен содержать следующую строку:

    GET|HEAD api/documentation .......... l5-swagger.default.api...

  7. Запустим отладочный сервер:

    php artisan serve

    В данном руководстве используется хост по умолчанию (http://127.0.0.1:8000/).

  8. Запустим браузер и обратимся по веб­‑адресу: http://127.0.0.1:8000/api/documentation/. В браузер должно быть выведено сообщение Failed to load API definition («Не удалось загрузить определение (описание) API»). Ничего страшного: документ OpenAPI пока не существует, потому как не существует метаданных. Создадим их.

Метаданные​

Информация об API​

  1. Откроем на редактирование файл app/Http/Controllers/Controller.php.

  2. В части между декларацией пространства имён и классом добавим новую строку, наберём на ней use openapi, выберем из перечня автодополнения пункт OpenApi, нажмём на ↵ Enter, поставим обратную косую черту (\), наберём attr, выберем из списка автодополнения Attributes, снова нажмём на ↵ Enter и через пробел допишем: as OA;. В итоге должна получиться строка:

    use OpenApi\Attributes as OA;

    Это предложение означает, что внутри контроллера класс OpenApi\Attributes будет доступен под произвольно назначенным нами коротким именем OA. Заметим, что эта аббревиатура последовательно используется в документации на swagger-php.

  3. Вставим перед объявлением класса (над строкой class Controller extends BaseController) атрибут OA\Info, который содержит следующие метаданные:

    • название веб­‑сайта (title);
    • версия API;
    • имя и почтовый адрес контактного лица.
    Без разрывов строк!

    Между атрибутом и классом или функцией не должно быть ни одной пустой строки.

    Файл app/Http/Controllers/Controller.php
    <?php

    namespace App\Http\Controllers;

    use Illuminate\Foundation\Auth\Access\AuthorizesRequests;
    use Illuminate\Foundation\Validation\ValidatesRequests;
    use Illuminate\Routing\Controller as BaseController;
    use OpenApi\Attributes as OA;

    #[OA\Info(
        title: 'Интернет­‑магазин «Десяточка»',
        version: '1.0.0',
        contact: new OA\Contact(
            email: 'ivanov@example.com',
            name: 'Иван Иванов',
        )
    )]
    class Controller extends BaseController
    {
        use AuthorizesRequests, ValidatesRequests;
    }

    Эта метаинформация будет применяться при обращении к любому контроллеру.

  4. Обновим веб­‑страницу документации в браузере и увидим, что сообщение об ошибке не исчезло. Ничего страшного!

  5. Подадим команду

    php artisan l5-swagger:generate

    Будет выведено сообщение:

    Required @OA\PathItem() not found

    В исходном коде не найден обязательный атрибут @OA\PathItem, т. е. не определён ни один путь (относительный веб­‑адрес) API. Эту проблему мы решим в следующих разделах.

    Как отладить код для генератора документации?

    При внесении правок в исходный код веб­‑сайта пакет L5 Swagger может автоматически создавать и обновлять веб­‑интерфейс с документацией. Однако при возникновении ошибок это бесполезная функция. Например, синтаксическая ошибка приведёт к выводу малоинформативного сообщения Failed to load API definition.

    Если ошибка возникла, следует провести отладку посредством подачи команды

    php artisan l5-swagger:generate

    И тогда можно будет получить представление о природе ошибки. Примеры:

    • Required @OA\Info() not found — в исходном коде не найдена обязательная аннотация @OA\Info (информация об API);
    • Missing required field "title" for @OA\Info() — обязательная аннотация @OA\Info найдена в исходном коде, но в ней пропущено обязательное поле title (название API);
    • @OA\Get() requires at least one @OA\Response() — к аннотации @OA\Get() должна прилагаться хотя бы одна аннотация @OA\Response.

Информация о схемах данных​

Добавим метаинформацию в модель Product.

  1. Откроем на редактирование файл app/Models/Product.php.

  2. В части между декларацией пространства имён и классом снова укажем

    use OpenApi\Attributes as OA;

  3. Вставим перед объявлением класса (над строкой class Product extends Model) атрибут OA\Schema, который позволяет представить класс как схему данных.

  4. Добавим в класс Product объявления свойств с именами столбцов соответствующей таблицы БД; типы данных свойств должны быть приближены к типам данных SQL, указанным в сценарии миграции. Отметим атрибутом OA\Property те свойства, которые доступны через API.

    В итоге файл должен быть приведён к следующему виду:

    Файл app/Models/Product.php
    <?php

    namespace App\Models;

    use Illuminate\Database\Eloquent\Factories\HasFactory;
    use Illuminate\Database\Eloquent\Model;
    use Illuminate\Database\Eloquent\Relations\BelongsToMany;
    use Illuminate\Database\Eloquent\Relations\HasMany;
    use OpenApi\Attributes as OA;

    #[OA\Schema()]
    class Product extends Model
    {
        use HasFactory;

        #[OA\Property()]
        private string $name;

        #[OA\Property()]
        private float $price;

        #[OA\Property()]
        private string $description;

        #[OA\Property()]
        private int $quantity;

        #[OA\Property()]
        private int $available;

        protected $fillable = [
            'name',
            'price',
            'description',
            'quantity',
            'available',
        ];

        public function images(): HasMany
        {
            return $this->hasMany(Image::class);
        }

        public function tags(): BelongsToMany
        {
            return $this->belongsToMany(Tag::class);
        }
    }

Информация об операциях​

Добавим метаинформацию в один из дочерних контроллеров — ProductController.

  1. Откроем на редактирование файл app/Http/Controllers/ProductController.php.

  2. В части между декларацией пространства имён и классом снова укажем

    use OpenApi\Attributes as OA;

  3. Документируем операцию API, реализованную методом index. Над его объявлением размещён документирующий комментарий с кратким описанием метода (summary) в формате PHPDoc:

    /**
     * Display a listing of the resource.
     */

    Удалим документирующий комментарий и заменим его аннотацией в форме атрибута PHP:

    #[OA\Get(
        description: 'Возвращает перечень товаров',
        path: '/api/v1/products/',
    )]
    #[OA\Response(
        content: new OA\JsonContent(),
        description: 'Перечень товаров',
        response: 200,
    )]
    #[OA\Response(
        content: new OA\JsonContent(),
        description: 'Товар не найден',
        response: 404,
    )]

    Аннотация OA\Get несёт в себе описание операции (description) и её относительный веб­‑адрес (path). Возможные ответы сервера описаны в аннотациях OA\Response. В каждой из них приведено описание ответа (description), а также код ответа (response) и медиатип содержимого (content; в данном случае — JSON).

    PHPDoc не нужен?

    На сегодняшний день swagger-php использует метаданные из раздела summary документирующего комментария, если поле summary не задано в OA\Get и тому подобных аннотациях. Впрочем, документирующие комментарии в целом устарели, и вряд ли стоит использовать их в новом проекте.

  4. Обновим веб­‑страницу документации в браузере. В браузер должна быть выведена документация на API в целом и метод index — в частности.

    Пользовательский интерфейс документации

  5. Документируем операцию API, реализованную методом show. Над его объявлением размещён документирующий комментарий:

    /**
     * Display the specified resource.
     */

    Удалим его и заменим его аннотацией в форме атрибута PHP:

    #[OA\Get(
        description: 'Возвращает товар по идентификатору',
        parameters: [
            new OA\Parameter(ref: '#/components/parameters/product'),
        ],
        path: '/api/v1/products/{product}/',
    )]
    #[OA\Response(
        content: new OA\JsonContent(),
        description: 'Товар',
        response: 200,
    )]
    #[OA\Response(
        content: new OA\JsonContent(),
        description: 'Товар не найден',
        response: 404,
    )]

    Здесь появляется новый класс аннотации — OA\Parameter. Он описывает значение {product} в веб­‑адресе '/api/v1/products/{product}/', указанном в path. Мы предполагаем, что {product} встретится в веб­‑адресах ещё нескольких операциях, поэтому в OA\Parameter приведена только ссылка на объявление параметра, а само объявление мы разместим прямо над объявлением класса контроллера (перед class ProductController extends Controller):

    #[OA\Parameter(name: 'product', description: 'Первичный ключ', in: 'path')]

    В этой аннотации всё просто: name — имя параметра (соответствует значению в фигурных скобках), description — описание параметра, in — место размещения параметра в веб­‑адресе (например, path — внутри пути, т. е. между косыми чертами, или query — после вопросительного знака)

  6. Обновим веб­‑страницу документации в браузере. В браузер должна быть выведена документация на API в целом и метод show — в частности.

  7. Документируем операцию API, реализованную методом store. Над его объявлением размещён документирующий комментарий:

    /**
     * Store a newly created resource in storage.
     */

    Удалим его и заменим его аннотацией в форме атрибута PHP:

    #[OA\Post(
        description: 'Создаёт запись о товаре',
        path: '/api/v1/products/',
        requestBody: new OA\RequestBody(
            content: new OA\MediaType(
                mediaType: 'application/x-www-form-urlencoded',
                schema: new OA\Schema(ref: '#/components/schemas/Product'),
            ),
            required: true,
        ),
    )]
    #[OA\Response(
        content: new OA\JsonContent(),
        description: 'Запись о товаре создана',
        response: 201,
    )]

    Аргумент requestBody предъявляет требования к телу запроса (OA\RequestBody), а именно к медиатипу его содержимого (mediaType) и набору полей (schema). Чтобы не перечислять эти поля, сошлёмся на ранее описанную схему Product.

  8. Продолжая описывать операции, приведём контроллер к следующему виду:

    Файл app/Http/Controllers/ProductController.php
    <?php

    namespace App\Http\Controllers;

    use App\Http\Requests\StoreProductRequest;
    use App\Http\Requests\UpdateProductRequest;
    use App\Models\Product;
    use OpenApi\Attributes as OA;

    #[OA\Parameter(name: 'product', description: 'Первичный ключ', in: 'path')]
    class ProductController extends Controller
    {
        #[OA\Get(
            description: 'Возвращает перечень товаров',
            path: '/api/v1/products/',
        )]
        #[OA\Response(
            content: new OA\JsonContent(),
            description: 'Перечень товаров',
            response: 200,
        )]
        #[OA\Response(
            content: new OA\JsonContent(),
            description: 'Товар не найден',
            response: 404,
        )]
        public function index()
        {
            $products = Product::all();
            return response($products);
        }

        #[OA\Post(
            description: 'Создаёт запись о товаре',
            path: '/api/v1/products/',
            requestBody: new OA\RequestBody(
                content: new OA\MediaType(
                    mediaType: 'application/x-www-form-urlencoded',
                    schema: new OA\Schema(ref: '#/components/schemas/Product'),
                ),
                required: true,
            ),
        )]
        #[OA\Response(
            content: new OA\JsonContent(),
            description: 'Запись о товаре создана',
            response: 201,
        )]
        public function store(StoreProductRequest $request)
        {
            $data = $request->validated();
            $product = Product::create($data);
            return response()->noContent(201)->withHeaders([
                'Location' => route('products.show', [
                    'product' => $product->id,
                ]),
            ]);
        }

        #[OA\Get(
            description: 'Возвращает товар по идентификатору',
            parameters: [
                new OA\Parameter(ref: '#/components/parameters/product'),
            ],
            path: '/api/v1/products/{product}/',
        )]
        #[OA\Response(
            content: new OA\JsonContent(),
            description: 'Товар',
            response: 200,
        )]
        #[OA\Response(
            content: new OA\JsonContent(),
            description: 'Товар не найден',
            response: 404,
        )]
        public function show(Product $product)
        {
            return response($product);
        }

        #[OA\Patch(
            description: 'Корректирует запись о товаре',
            parameters: [
                new OA\Parameter(ref: '#/components/parameters/product'),
            ],
            path: '/api/v1/products/{product}/',
            requestBody: new OA\RequestBody(
                content: new OA\MediaType(
                    mediaType: 'application/x-www-form-urlencoded',
                    schema: new OA\Schema(ref: '#/components/schemas/Product'),
                ),
                required: false,
            ),
        )]
        #[OA\Response(
            content: new OA\JsonContent(),
            description: 'Запись о товаре откорректирована',
            response: 204,
        )]
        public function update(UpdateProductRequest $request, Product $product)
        {
            $data = $request->validated();
            $product->update($data);
            return response()->noContent(204);
        }

        #[OA\Delete(
            description: 'Удаляет запись о товаре',
            parameters: [
                new OA\Parameter(ref: '#/components/parameters/product'),
            ],
            path: '/api/v1/products/{product}/',
        )]
        #[OA\Response(
            content: new OA\JsonContent(),
            description: 'Запись о товаре удалена',
            response: 204,
        )]
        public function destroy(Product $product)
        {
            $product->delete();
            return response()->noContent(204);
        }
    }

Тестирование API​

  1. Подадим команду на создание сценария автоматизированного тестирования контроллера ProductController.

    php artisan make:test ProductControllerTest

    Имя создаваемого класса сценария обязательно должно заканчиваться на слово Test, иначе инструмент тестирования не найдёт его.

    Должен быть создан файл tests/Feature/ProductControllerTest.php.

  2. Откроем вышеназванный файл на редактирование.

    Файл tests/Feature/ProductControllerTest.php
    <?php

    namespace Tests\Feature;

    use Illuminate\Foundation\Testing\RefreshDatabase;
    use Illuminate\Foundation\Testing\WithFaker;
    use Tests\TestCase;

    class ProductControllerTest extends TestCase
    {
        /**
         * A basic feature test example.
         */
        public function test_example(): void
        {
            $response = $this->get('/');

            $response->assertStatus(200);
        }
    }

    Все методы тестирования должны начинаться со слова test, иначе инструмент тестирования не найдёт их.

  3. Выполним тестирование, запустив ⁠PHPUnit:

    • Windows
    • Linux
    • macOS
    vendor\bin\phpunit
    ./vendor/bin/phpunit
    ./vendor/bin/phpunit

    Должно быть выведено

    OK (2 tests, 2 assertions)

  4. Удалим метод test_example.

  5. Создадим метод для тестирования операции API, реализованной как метод index.

    public function testIndex(): void
    {
        $response = $this->get('/api/v1/products/');
        $response->assertStatus(200);
    }

    Контрольный пример найдёт ошибку, если метод index контроллера не даёт ответ HTTP с состоянием 200.

  6. Выполним тестирование, запустив ⁠PHPUnit.

  7. Создадим метод для тестирования операции API, реализованной как метод store.


    Контрольный пример найдёт ошибку, если метод store контроллера не даёт ответ HTTP с состоянием 201.

  8. Если нам не хочется вручную удалять из БД записи, внесённые туда при тестировании, то мы можем подключим к классу трейт, очищающий БД (именно БД, а не отдельную таблицу!) после тестирования.

    use RefreshDatabase;

  9. Убедимся, что файл приобрёл приблизительно следующий вид:

    Файл tests/Feature/ProductControllerTest.php
    <?php

    namespace Tests\Feature;

    use App\Models\Product;
    use Illuminate\Foundation\Testing\RefreshDatabase;
    use Illuminate\Foundation\Testing\WithFaker;
    use Tests\TestCase;

    class ProductControllerTest extends TestCase
    {
        // use RefreshDatabase;

        public function testIndex(): void
        {
            // Подаём запрос на сервер.
            $response = $this->get(route('products.index'));

            // Проверяем статус ответа сервера.
            $response->assertStatus(200);
        }

        public function testStore(): void
        {
            // Вызываем фабрику и преобразуем объект в массив.
            $data = Product::factory()->make([
                // Здесь можно заменить значения полей, созданные фабрикой.
                // 'name' => 'Новое наименование товара',
                // 'price' => 100.99,
            ])->toArray();

            // Подаём запрос, передавая массив как тело сообщения.
            $response = $this->post(route('products.store'), $data);

            // Проверяем статус ответа сервера.
            $response->assertStatus(201);
        }

        public function testShow()
        {
            // Выбираем случайную запись из БД.
            $product = Product::all()->random(1)->firstOrFail();

            // Подаём запрос, используя значение первичного ключа.
            $response = $this->get(route('products.show', [
                'product' => $product->id,
            ]));

            // Проверяем статус ответа сервера.
            $response->assertStatus(200);
        }
    }

  10. Выполним тестирование, запустив ⁠PHPUnit.

    Дополнительное задание

    Если Вы претендуете на оценку «отлично», самостоятельно доработайте тестовый комплект, добавив в него методы для автоматизированного тестирования операций API, реализованных методами update и destroy.
