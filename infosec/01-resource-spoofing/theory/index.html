Теоретические сведения

Ресурсы и подресурсы​

Всемирная паутина (англ. World­‑wide web, WWW) — это информационное пространство, элементы которого, называемые ресурсами, идентифицированы посредством глобальных единообразных идентификаторов (URI) [WebArch, разд. 1]. Например, идентификатор https://example.com/index.html указывает на ресурс, представляющий собой документ HTML.

Подресурс (субресурс) — это ресурс, который извлекается при обработке другого ресурса. Например, файлы, содержащие CSS, сценарии на языке JavaScript или шрифты, могут являться подресурсами документа HTML.

Нередко веб­‑разработчики размещают подресурсы на других доменах, в т. ч. в сетях доставки контента — content delivery network, сокращённо CDN. В некоторых случаях это позволяет упорядочить и ускорить обработку подресурсов.

Атака subresource spoofing​

Однако злоумышленник (например, администратор CDN, хостинга, репозитория) имеет возможность изменить содержимое подресурса или вовсе перенаправить запрос на другой узел. Такую атаку называют subresource spoofing («подмена подресурса»; если быть точным, происходит подмена не самого подресурса, а его содержимого, так как URL остаётся прежним).

Объектом атаки является пользовательский агент (браузер), который получает и выполняет вредоносное содержимое подресурса. Субъектами атаки могут выступать как третьи лица (взломщики хостинга или CDN), так и сопроводители подресурсов — другими словами, ничто не мешает автору изменить исходный код библиотеки, не изменяя гиперссылку на неё. Злонамеренность авторов — не редкость. Авторов нескольких пакетов Node.js уличили в размещении в своих творениях кода для cryptojacking’а — несанкционированного использования вычислительных мощностей устройств, на которых выполняется сценарий, для майнинга криптовалют; координация расчётов велась через сервисы Coinhive, CoinIMP и т. п.

Протокол TLS не пресечёт атаку, поскольку гарантирует подлинность узла, но не целостность контента, а подложный узел наверняка предъявит годный сертификат.

Нет CDN — нет subresource spoofing?

В недавнем прошлом пользовательские агенты (браузеры) обрабатывали подресурсы таким образом, что подключение одной и той же версии подресурса (например, сверхпопулярной библиотеки Bootstrap) из одной и той же CDN ко множеству сайтов позволяло пользовательскому агенту (браузеру) при обращении к одному из этих сайтов скачать и кешировать содержимое подресурса на срок, установленный CDN, а при последующих обращениях в течение этого срока заменить повторное скачивание извлечением из кеша, с локального носителя (т. е. с жёсткого диска клиента), что ускоряло отображение сайтов.

Изолированный кеш, который разработчики браузеров внедряют в связи с реализацией в 2019 г. атаки HTTP cache cross­‑site leaks (XSLeaks), в значительной мере обесценивает идею CDN. Теперь содержимое подресурса, подключённого к нескольким разным сайтам из одной и той же CDN, хранится у клиента в виде изолированных копий. Если тот же подресурс будет подключён к ещё одному сайту, то содержимое будет заново скачано браузером, а не взято из кеша.

Однако возможное исчезновение CDN не решает проблему resource spoofing, так как подмена подресурса не обязательно связана с использованием CDN.

Защита посредством SRI​

Для борьбы с resource spoofing Консорциум Всемирной паутины (W3C) издал спецификацию Subresource integrity (далее — SRI). Она определяет алгоритм, позволяющий пользовательскому агенту (например, браузеру) проверить целостность некоторых видов подресурсов.

Алгоритм​

Согласно спецификации, пользовательский агент (браузер) отказывает в обработке подресурса, контрольная сумма которого не совпадает со значения атрибута integrity. Рекомендация 2016 г. и редакторский черновик за 30.11.2022 г. обязывают пользовательский агент обеспечить расчёт контрольных сумм посредством хеш­‑функций SHA‑256, SHA‑384, SHA‑512 для подресурсов, подключаемых через элементы HTML link и script. Поддержка a, audio, embed, iframe, img, object, source, track, video ожидается в последующих ревизиях спецификации.

Сначала пользовательский агент (браузер) обрабатывает атрибут crossorigin, который обычно имеет значение anonymous (подресурс размещён на другом доменном имени, т. е. используется cross­‑origin resource sharing, CORS, без аутентификации клиента). CORS позволяет загружать содержимое подресурсов только по ограниченному набору схем: data (не рекомендуется), http (не рекомендуется), https, поэтому URL со схемой file просто не будет обработан (см. консоль по F12):

Access to script at 'file:///путь/к/сценарию.js' from origin 'null' has been blocked by CORS policy: Cross origin requests are only supported for protocol schemes: http, data, chrome, chrome-extension, https.
Access to CSS stylesheet at 'file:///путь/к/стилю.css' from origin 'null' has been blocked by CORS policy: Cross origin requests are only supported for protocol schemes: http, data, chrome, chrome-extension, https.

Можно попробовать полностью убрать атрибут crossorigin, но без него перестанет работать SRI:

Subresource Integrity: The resource 'file:///путь/к/сценарию.js' has an integrity attribute, but the resource requires the request to be CORS enabled to check the integrity, and it is not. The resource has been blocked because the integrity cannot be enforced.
Subresource Integrity: The resource 'file:///путь/к/стилю.css' has an integrity attribute, but the resource requires the request to be CORS enabled to check the integrity, and it is not. The resource has been blocked because the integrity cannot be enforced.

После скачивания содержимого подресурса пользовательский агент (браузер) анализирует значение атрибута integrity. Фрагмент значения, расположенный после дефиса, содержит ожидаемую контрольную сумму, а префикс (фрагмент до дефиса) указывает на хеш­‑функцию, посредством которой вычислена эта сумма. Пользовательский агент (браузер) самостоятельно рассчитывает контрольную сумму полученного содержимого подресурса и сверяет результат с ожидаемым. Если хеши не совпали, подресурс не будет применён, а в консоль (F12) будет выведено сообщение с тем хешем, который вычислил пользовательский агент (браузер):

Failed to find a valid digest in the 'integrity' attribute for resource 'http://127.0.0.1:8000/script.js' with computed SHA-256 integrity 'A77/FIoHOGjeI5i/rPLyY3QGTSXkJ3r+pbdozEsKtTg='. The resource has been blocked.
Failed to find a valid digest in the 'integrity' attribute for resource 'http://127.0.0.1:8000/style.css' with computed SHA-256 integrity 'WwvhFMMGzGuT3vWhT1U/DIrGBH0hL/MIurzYJonkfpQ='. The resource has been blocked.

Условия применения​

Для полноценной работы SRI необходимо соблюсти ряд условий:

  • как говорилось выше, хост подресурса должен разрешить совместное использование своих ресурсов разными источниками (англ. cross­‑origin resource sharing, CORS), передав в ответном сообщении HTTP заголовок Access-Control-Allow-Origin. Его значение * означает, что к ресурсам сервера можно обратиться из любого домена;

    <VirtualHost *:80>
        ServerName доменное.имя
        DocumentRoot "путь/к/каталогу/с/сайтом"
        Header always set Access-Control-Allow-Origin "*"
    </VirtualHost>

  • отсутствие уязвимости для атаки MiTM, в результате которой и содержимое подресурса, и значение атрибута inregrity могут быть подменены во время передачи по каналу связи. Контроль целостности информации, передаваемой по каналу связи, может обеспечить протокол TLS (схема https);

  • отсутствие у злоумышленника возможности воздействовать на объектную модель документа HTML в пользовательском агенте (браузере). Например, посредством XSS злоумышленник может добавить на веб­‑страницу элемент script со ссылкой на вредоносный сценарий. Такие приёмы выходят за рамки ответственности SRI — потребуется устранить уязвимости для XSS и вести белый список источников посредством Content-Security-Policy: script-src;

  • оценка безопасности подресурсов: SRI не поможет, если разработчик веб­‑сайта опрометчиво или умышленно создаст ссылку на вредоносный сценарий и укажет правильную контрольную сумму.

Ограничения SRI, установленные в отношение подресурса, не распространяются автоматически на его собственные подресурсы. Рассмотрим пример. Пусть файл a.js размещён в сети доставки контента (англ. CDN) и подключён к документу HTML в соответствии с SRI:

Файл http://victim.example/index.html
<!DOCTYPE html>
<html lang="ru-luna1918">
<head>
  <meta charset="UTF-8">
  <title>Атакуемый сайт</title>
  <script
    defer
    src="http://cdn.example/a.js"
    integrity="sha256-F5pZeHHm0pWtW+R7qlP1/kiuffGQkZAcQRvXNaiP2Fk="
    crossorigin="anonymous"
    ></script>
</head>
<body>
  <h1>Атакуемый сайт</h1>
</body>
</html>

Файл a.js содержит следующий сценарий JavaScript:

Файл http://cdn.example/a.js
(function () {
  'use strict';

  window.addEventListener('load', function () {
    console.log('Выполнен сценарий JavaScript с cdn.example!');

    var head = document.getElementsByTagName('head');
    if (head.length) {
      var script = document.createElement('script');
      script.defer = 'defer';
      script.src = 'http://third-party.example/b.js';
      head[0].appendChild(script);
    }
  });
})();

Как мы видим, сценарий a.js создаёт и добавляет в документ HTML элемент script, который обязывает браузер скачать и выполнить сценарий b.js, размещённый на стороннем сайте. Сценарий b.js может не являться вредоносным изначально, однако подключён к документу HTML без учёта SRI. Следовательно, злоумышленник в любой момент может подменить или модифицировать b.js.

Решением будет насаждение ограничений SRI в файле a.js.

Файл http://cdn.example/a.js
(function () {
  'use strict';

  window.addEventListener('load', function () {
    console.log('Выполнен сценарий JavaScript с cdn.example!');

    var head = document.getElementsByTagName('head');
    if (head.length) {
      var script = document.createElement('script');
      script.defer = 'defer';
      script.src = 'http://third-party.example/b.js';
      script.integrity = 'sha256-OdRL64E0NQAJcfA0EtBLnr1hUU6IbTW4xPrB+iTcsww=';
      script.crossOrigin = 'anonymous'; // Именно crossOrigin, а не crossorigin!
      head[0].appendChild(script);
    }
  });
})();
