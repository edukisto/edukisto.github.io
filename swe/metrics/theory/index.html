Теоретические сведения

Метрики размера​

Число строк кода (LOC)​

Существует множество разновидностей метрики LOC:

  • число LOC (lines of code) — общее число строк исходного кода;
  • число PSS (physical source statements) — число непустых строк исходного кода. Число PSS для строк комментариев вычисляется отдельно от числа PSS для остальных непустых строк [1, с. 4, 5–⁠7];
  • число LSS (logical source statements) — число предложений в исходном коде независимо от физического формата, в котором они представлены [1, с. 3, 5–⁠7], т. е. независимо от того, сколько строк они занимают. Число LSS для комментариев вычисляется отдельно от числа LSS для остальных предложений;
  • число LLOC (logical lines of code) — число «логических» строк исходного кода; обычно синонимично LSS;
  • число SLOC (source lines of code) — число строк исходного кода, влияющих на выполнение программы (ГОСТ 28397‑89), т. е. без учёта пустых строк, строк комментариев и т. п.
Что такое «предложение»?

Под термином «предложение» в определении LSS понимается синтаксический элемент, состоящий из одного или нескольких выражений. Если на одной строке записано два предложения, LSS фрагмента равен 2. Если одно предложение записано на двух строках, LSS фрагмента равен 1.

В языке программирования C точка с запятой разделяет простые предложения, а пара фигурных скобок объединяет предложения в блок (составное предложение). В Python для тех же целей используют соответственно разрыв строки и индентацию.

Что такое LLOC?

Значение числа LLOC (синоним LSS) можно объяснить на примере. Составное предложение

if index != len(string2):
    raise ValueError('Строки разной длины')

можно записать как

if index != len(string2): raise ValueError('Строки разной длины')

Первый вариант записи оценивается так:

    LOC: 2
    LLOC: 2
    SLOC: 2

Второй оценивается иначе:

    LOC: 1
    LLOC: 2
    SLOC: 1

Как мы видим, число строк (LOC, SLOC) во втором фрагменте действительно сократилось с 2 до 1, но число предложений (LLOC) осталось равным двум: первое предложение содержит условный оператор if, второе — оператор raise. Для LLOC не имеет значение, на скольких строках записано предложение.

Особенности:

  • число строк для модулей на разных языках следует учитывать раздельно [1, с. 5] и оценивать с учётом коэффициента, зависящего от языка программирования . Так, при сравнении программ, записанных на C и Python’е, необходимо домножить числа LSS и PSS, вычисленные для Python, на 6 и 6,5 соответственно, поскольку считается, что именно во столько раз предложения и строки на Python выразительнее предложений и строк на C.

Прецеденты использования LOC:

  • число SLOC используют в расчёте индекса удобства обслуживания программы в интегрированной среде разработки Microsoft Visual Studio 2022 [2].

Меры измерения Холстеда​

Меры измерения сложности, которые в 1977 г. предложил Мо́рис Хо́лстед [3].

Область применения:

  • сравнение сложности программ.

Недостатки:

  • изначально меры предназначались для оценки алгоритмов, а не программ [4, с. 9];
  • Холстед использовал, но не определил термин «оператор»;
  • меры измерения игнорируют связь между элементами системы (условия, циклы).

Прецеденты применения:

  • объём программы по Холстеду учитывают при расчёте индекса удобства обслуживания программы в интегрированной среде разработки Microsoft Visual Studio 2022 [2];
  • в программных продуктах IBM: AD Analyze, Rational Asset Analyzer, DevOps Test Embedded;
  • длина программы по Холстеду — одна из мер измерения, которые ГОСТ Р МЭК 62279‑2016 рекомендует для оценки программного обеспечения систем управления и защиты на железных дорогах, а ГОСТ 34332.5‑2021 — для оценки программных систем, связанных с безопасностью зданий и сооружений.

Базовые меры измерения​

  • η1\eta_{1}η1​ (следует читать как «э́та ма́лое первое») — число различных операторов;
  • η2\eta_{2}η2​ — число различных операндов (величин, над которыми выполняются операции);
  • N1\Nu_{1}N1​ — общее число операторов;
  • N2\Nu_{2}N2​ — общее число операндов [4, с. 3; 5, п. С.2.3.2].

Для каждого языка программирования должны быть установлены частные критерии определения принадлежности лексем к операторам. Например, для Си­‑подобных языков справедливы следующие правила:

  • комментарии, объявления переменных и т. п. не учитывать;
  • условный оператор (if...else), операторы цикла (for, while) и т. п. не учитывать;
  • знаки препинания, в т. ч. скобки, не учитывать.

При расчёте учитывают арифметические операторы (в т. ч. знак числа), логические операторы и операторы сдвига.

Размер словаря программы​

Размер словаря (англ. vocabulary) программы [4, с. 3; 6, с. 178]:

η=η1+η2.\eta = \eta_{1} + \eta_{2}.η=η1​+η2​.

Длина программы​

Длина (англ. length) программы [4, с. 3]:

N=N1+N2.\Nu = \Nu_{1} + \Nu_{2}.N=N1​+N2​.

Расчётная длина программы​

Длину программы можно предсказать, если известен словарь программы. Расчётная длина (англ. calculated length) программы [4, с. 5; 6, с. 178] выражается формулой

N^=η1log⁡2η1+η2log⁡2η2,\widehat\Nu = \eta_{1} \log_{2}\eta_{1} + \eta_{2} \log_{2}\eta_{2},N=η1​log2​η1​+η2​log2​η2​,

где log⁡2η1\log_{2}\eta_{1}log2​η1​ и log⁡2η2\log_{2}\eta_{2}log2​η2​ — минимальное число битов (разрядов двоичного числа), на которых можно записать номер любого оператора или операнда соответственно. Например, если в программе 4 уникальных оператора (log⁡2η1=4\log_{2}\eta_{1} = 4log2​η1​=4), то для записи номера любого из них требуется 2 бита: 00200_{2}002​, 01201_{2}012​, 10210_{2}102​, 11211_{2}112​.

Объём программы​

Объём (англ. volume) программы [4, с. 3–⁠4; 6, с. 178] показывает, сколько битов (разрядов двоичного числа) требуется для записи номеров всех (впервые и повторно встреченных) операторов и операндов в программе. Объём программы выражается формулой

V=Nlog⁡2η,V = \Nu \log_{2} \eta,V=Nlog2​η,

где log⁡2η\log_{2} \etalog2​η — минимальное число битов (разрядов двоичного числа), на которых можно записать номер любого слова из словаря программы (η\etaη). Например, если словарь содержит 8 слов (η=8\eta = 8η=8), то для записи номера любого из них требуется 3 бита: 0002000_{2}0002​, 0012001_{2}0012​, ..., 1112111_{2}1112​.

Объём программы можно определить как

  • оценку размера реализации алгоритма [3, с. [19]];
  • число мысленных сравнений, необходимых для проектирования программы [3, с. [47]].

Значение VVV для функции должно удовлетворять условию 20⩽V⩽100020 \leqslant V \leqslant 100020⩽V⩽1000. Значение выше 1000 показывает, что функция решает слишком много задач, ответственность за решение которых можно разделить между менее объёмными функциями.

Значение VVV для программного модуля должно удовлетворять условию 100⩽V⩽8000100 \leqslant V \leqslant 8000100⩽V⩽8000.

Сложность программы​

Сложность (англ. difficulty) программирования и понимания при code review [4, с. 7]:

D=η12×N2η2.D = \frac{η_1}{2} \times \frac{Ν_2}{η_2}.D=2η1​​×η2​N2​​.

Сложность программирования возрастает с добавлением в исходный код ранее отсутствовавших в нём операторов (растёт η12\frac{η_1}{2}2η1​​) и/⁠или повторным использованием операндов (растёт отношение общего числа операндов N2Ν_2N2​ к числу уникальных операндов η2η_2η2​).

Усилия​

Усилия (англ. effort), т. е. число различений (логических операций), выполняемых мозгом человека при разработке программы, выражается отношением сложности программы к её объёму [4, с. 4–⁠5]:

E=D×V.E = D \times V.E=D×V.

Усилия возрастают с ростом сложности и/⁠или увеличением объёма программы.

Время программирования​

Время (англ. time), которое требуется на разработку программы, выражается формулой

T=ES,T = \frac{E}{S},T=SE​,

где SSS — число Стра́уда, по умолчанию равное 18 [4, с. 7–⁠8].

Почему 18?

Психолог Джон Страуд считал, что человеческий мозг способен выполнять «от 5 до 20 или чуть меньше» различений в секунду [4, с. 7; 7], что коррелирует с частотой кадров, необходимой для достижения иллюзии движения. Впрочем, концепция Страуда не получила всеобщего признания. Следовательно, корректность формулы подвержена критике [4, с. 11].

Число ошибок в программе​

Число ошибок (англ. bugs) в программе:

B=E23E0.B = \frac{E^\frac{2}{3}}{E_{0}}.B=E0​E32​​.

Чаще, в частности в Radon и Rational Test RealTime от IBM, используют формулу

B=V3000.B = \frac{V}{3000}.B=3000V​.

Значение BBB следует держать ниже 2.

Индекс сопровождаемости​

Классическая формуля для расчёта индекса сопровождаемости (англ. index of maintainability, MI [8]) выглядит так:

MI=171−5,2ln⁡V−0,23×v(G′)−16,2ln⁡L+50sin⁡2,4C,MI = 171 - 5,2 \ln V - 0,23 \times v(G') - 16,2 \ln L + 50 \sin \sqrt{2,4C},MI=171−5,2lnV−0,23×v(G′)−16,2lnL+50sin2,4C​,

где VVV — объём программы по Холстеду; v(G′)v(G')v(G′) — расширенная цикломатическая сложность; LLL — число LOC; CCC — процентное отношение числа комментариев к числу строк исходного кода [9, с. 231].

Положительный коэффициент при CCC отражает мнение автора, что комментарии в исходном коде повышают сопровождаемость. Однако комментарии, содержащие многословные лицензионные соглашения или крупные фрагменты исходного кода, либо не влияют, либо отрицательно влияют на сопровождаемость. Если таких комментариев много, то можно использовать сокращённую формулу:

MI=171−5,2ln⁡V−0,23×v(G′)−16,2ln⁡L.MI = 171 - 5,2 \ln V - 0,23 \times v(G') - 16,2 \ln L.MI=171−5,2lnV−0,23×v(G′)−16,2lnL.

На практике вместо оригинальных формул применяются немного доработанные функции, область значения которых лежит в области неотрицательных чисел.
