Теоретические сведения

Уровни тестирования​

Общим уровням тестирования соответствуют элементы (объекты) тестирования:

  1. модульное/⁠покомпонентное тестирование (unit/⁠component testing) — это тестирование изолированных (взятых по отдельности, независимо друг от друга) модулей/⁠компонентов, таких как функция сложения двух чисел, функция сортировки элементов массива, процедура записи текстовых сообщений в файл, класс «Товар» и т. п.;
  2. интеграционное тестирование (integration testing) — это тестирование взаимодействия [пар] модулей/⁠компонентов. В частности, элементом (объектом) такого тестирования может выступать взаимодействие системы управления базами данных (СУБД) с так называемым слоем доступа к данным — модулем/⁠компонентом программы (приложения БД), который подаёт SQL­‑запросы к этой СУБД;
  3. тестирование системы (system testing) — это тестирование полной системы, т. е. программы после интеграции (объединения) всех её модулей/⁠компонентов. Элементом (объектом) этого уровня тестирования может являться игровое приложение, текстовый редактор, сайт, приложение для смартфона и т. п.;
  4. интеграционное тестирование системы (system integration testing) — это тестирование взаимодействия полной системы со средой выполнения конечного пользователя, как например: взаимодействие игрового приложения с операционной системой, заявленной в требованиях к программе; взаимодействие веб­‑приложения с программным окружением на сервере и т. п.;
  5. приёмочное испытание (acceptance testing), включающее, в частности, альфа‑ и бета­‑тестирование, демонстрирует конечным пользователям приемлемость полной системы с точки зрения их требований [1; 2].

Уровни обычно выполняют в указанном порядке [1].

Белый (стеклянный, прозрачный) и чёрный (непрозрачный) ящики — это указание на то, доступен ли тестеру исходный код элемента тестирования. Например, при бета­‑тестировании игры тестеру передают исполняемый файл игры, но не её исходный код, и это связано не только с коммерческими интересами разработчика, но и с тем, что бета­‑тестер должен оценивать приемлемость игры исключительно с точки зрения конечного пользователя (потребителя).

Каждый уровень тестирования делится на два этапа:

  1. разработка контрольных примеров;
  2. выполнение контрольных примеров.

Контрольные примеры​

Контрольный пример (англ. test case; также переводят как тестовый вариант или тестовый случай) разрабатывают для управления выполнением элемента тестирования для достижения целей тестирования [1; 2], таких как корректная реализация, идентификация ошибок, проверка качества и т. п.

Разработка контрольных примеров​

Контрольный пример содержит

  1. предварительные условия, охватывающие тестовую среду́ (англ. test environment), элемент тестирования и используемые им данные (например, из базы данных);
  2. ввод данных или другое воздействие на элемент тестирования;
  3. ожидаемый результат, полученный сопоставлением фактического результата с критерием успешного прохождения теста.

Таким образом, контрольный пример для функции сложения двух чисел должен содержать её аргументы (например, 5 и 7) и соответствующее ожидаемое значение функции (12).

Контрольный пример не содержит других контрольных примеров, но может входить в состав тестового комплекта (англ. test suite) [1]. Термин набор тестов (англ. test set [2]) следует признать устаревшим.

Выполнение контрольных примеров​

Выполнение теста на элементе тестирования (например, модуле или программе) и проверка фактического результата осуществляется либо с использованием инструментов тестирования (автоматизированное тестирование), либо без такового (ручное тестирование) [1, п. 3.46].

Таким образом, при выполнении контрольного примера для функции сложения двух чисел необходимо вызвать эту функцию с заданными аргументами (5 и 7) и сравнить фактическое значение функции с ожидаемым значением (12).

Для каждого актуального языка программирования существует один или несколько инструментов тестирования:

  • для C и C++ — Catch2 (основной недостаток — излишние простота и самобытность), GoogleTest (основной недостаток — сложен в установке);
  • для C# — Test Explorer (встроен в Visual Studio), xUnit.net и т. п.;
  • для Java — JUnit;
  • для JavaScript и TypeScript — Jest и т. п.;
  • для PHP — PHPUnit;
  • для Python — unittest (поставляется вместе с компилятором), pytest и т. п.

Спецификация тестирования​

Спецификация тестирования (англ. test specification) — это полная документация на проект тестирования, контрольные примеры (англ. test case) и процедуры тестирования (англ. test procedure) для конкретного элемента тестирования [1, п. 3.125]. Спецификация тестирования может принимать формы

  • процедуры тестирования (англ. test procedure), поддерживающей выполнение теста вручную;
  • сценарий автоматизированного тестирования (англ. automated test script), поддерживающего автоматизированное выполнение теста [1, п. 4.3.2.4.2].

Спецификация тестирования может быть изготовлена множеством способов, в том числе как программный документ или как исходный код в совокупности с записями из БД [1, п. 3.125].
