Теоретические сведения

Набор знаков​

Исходный код должен быть представлен в UTF‑8 без byte order mark.

Язык​

Идентификаторы (имена переменных, функций и т. п.) и комментарии следует записывать на английском языке.

Пустые строки​

Отделяйте объявления функции и класса верхнего уровня от предшествующего и последующего текста двумя пустыми строками.

Отделяйте объявление метода от предшествующего и последующего текста одной пустой строкой.

Длина строки​

Длину строки следует ограничить 79 знаками. Более длинные строки следует разбить, заключив их в круглые или фигурные скобки.

Длину строк, отведённых под комментарии и документирующие комментарии, следует ограничить 72 знаками.

Ограничения на длину строк позволяют использовать сплит экрана и избежать появление горизонтальных полос прокрутки (например, на GitHub).

Соглашение об именах​

Не давайте имён, состоящих из одной буквы I, O или l, так как их легко спутать с цифрами 0 или 1.

Регистр:

  • имена переменных, функций, методов — snake_case (нижний регистр с подчёркиваниями вместо пробелов);
  • имена классов — UpperCamelCase;
  • имена констант — UPPER_CASE_WITH_UNDERSCORES (верхний регистр с подчёркиваниями вместо пробелов).

Имена переменных следуют тому же соглашению, что и имена функций.

Аргументы функции и метода​

Всегда используйте self как первый аргумент метода объекта.

Всегда используйте cls как первый аргумент статического метода класса.

Методы​

Используйте одно начальное подчеркивание только для непубличных методов и переменных экземпляра.

Чтобы избежать конфликтов имен с подклассами, используйте два начальных символа подчеркивания для вызова правил искажения имен Python.

Объявление функции или метода должно завершаться выражением с оператором возврата (return).

осторожно
def foo(x):
    if x >= 0:
        return math.sqrt(x)
    # Что если x < 0?

Будьте последовательны в операторах возврата. Либо все операторы return в функции должны возвращать выражение, либо ни один из них не должен. Если какой­‑либо оператор возврата возвращает выражение, любые операторы возврата, в которых не возвращается значение, должны явно указывать это как, а в конце функции должен присутствовать явный оператор возврата (если он доступен):return None

подсказка
def foo(x):
    if x >= 0:
        return math.sqrt(x)
    else:
        return None

def bar(x):
    if x < 0:
        return None
    return math.sqrt(x)
осторожно
def bar(x):
    if x < 0:
        return
    return math.sqrt(x)

Индентация​

Используйте 4 пробела на каждом уровне вложенности. Горизонтальная табуляция не рекомендуется.

Строки продолжения должны выравнивать обернутые элементы либо по вертикали, используя неявное соединение строк Python внутри круглых и фигурных скобок, либо с помощью висячего отступа. При использовании висячего отступа следует учитывать следующее; в первой строке не должно быть аргументов, и следует использовать дополнительный отступ, чтобы четко отличить себя от строки продолжения:

подсказка
# Добавлены 4 пробела, чтобы отличить аргументы от тела функции.
def long_function_name(
        var_one, var_two, var_three,
        var_four):
    print(var_one)

# Аргументы выровнены по открывающей скобке.
foo = long_function_name(var_one, var_two,
                         var_three, var_four)

# Hanging indents should add a level.
foo = long_function_name(
    var_one, var_two,
    var_three, var_four)
осторожно
# Arguments on first line forbidden when not using vertical alignment.
foo = long_function_name(var_one, var_two,
    var_three, var_four)

# Further indentation required as indentation is not distinguishable.
def long_function_name(
    var_one, var_two, var_three,
    var_four):
    print(var_one)

Правило 4 пробелов является необязательным для строк продолжения.

По желанию:

warning
# Hanging indents *may* be indented to other than 4 spaces.
foo = long_function_name(
  var_one, var_two,
  var_three, var_four)

Когда условная часть if­‑оператора достаточно длинная, чтобы ее можно было записать в несколько строк, стоит отметить, что комбинация двухсимвольного ключевого слова (т.е. if), плюс один пробел, плюс открывающая скобка создает естественный 4 пробел для последующих строк многострочного условного предложения. Это может привести к визуальному конфликту с набором кода с отступом, вложенным в if­‑statement, который также естественным образом будет иметь отступ в 4 пробела. Этот PEP не занимает явной позиции в отношении того, как (и нужно ли) дополнительно визуально отличать такие условные строки от вложенного набора внутри if­‑оператора. Приемлемые варианты в этой ситуации включают, но не ограничиваются:

# No extra indentation.
if (this_is_one_thing and
    that_is_another_thing):
    do_something()

# Add a comment, which will provide some distinction in editors
# supporting syntax highlighting.
if (this_is_one_thing and
    that_is_another_thing):
    # Since both conditions are true, we can frobnicate.
    do_something()

# Add some extra indentation on the conditional continuation line.
if (this_is_one_thing
        and that_is_another_thing):
    do_something()

(Также смотрите обсуждение того, следует ли прерывать работу до или после бинарных операторов ниже.)

Закрывающая фигурная скобка/скобка/круглая скобка в многострочных конструкциях может располагаться под первым непробельным символом последней строки списка, например:

my_list = [
    1, 2, 3,
    4, 5, 6,
    ]
result = some_function_that_takes_arguments(
    'a', 'b', 'c',
    'd', 'e', 'f',
    )

или он может быть выстроен под первым символом строки, с которой начинается многострочная конструкция, например:

my_list = [
    1, 2, 3,
    4, 5, 6,
]
result = some_function_that_takes_arguments(
    'a', 'b', 'c',
    'd', 'e', 'f',
)

Должна ли строка разрываться до или после бинарного оператора?​

В течение десятилетий рекомендуемым стилем было прерывание после бинарных операторов. Но это может ухудшить читабельность двумя способами: операторы, как правило, разбросаны по разным столбцам на экране, и каждый оператор перемещается от своего операнда на предыдущую строку. Здесь глаз должен проделать дополнительную работу, чтобы определить, какие элементы добавляются, а какие вычитаются:

осторожно
# operators sit far away from their operands
income = (gross_wages +
          taxable_interest +
          (dividends - qualified_dividends) -
          ira_deduction -
          student_loan_interest)

Чтобы решить эту проблему удобочитаемости, математики и их издатели следуют противоположному соглашению. Дональд Кнут объясняет традиционное правило в своей серии «Компьютеры и набор текста»: «Хотя формулы внутри абзаца всегда прерываются после бинарных операций и отношений, отображаемые формулы всегда прерываются перед бинарными операциями».

Следование традиции из математики обычно приводит к более читабельному коду:

подсказка
# easy to match operators with operands
income = (gross_wages
          + taxable_interest
          + (dividends - qualified_dividends)
          - ira_deduction
          - student_loan_interest)

В коде Python допустимо прерывание до или после бинарного оператора, если соглашение локально согласовано. Для нового кода предлагается стиль Кнута.

Импорт​

Импорт обычно должен быть в отдельных строках:

подсказка
import os
import sys
from subprocess import Popen, PIPE
осторожно
import sys, os

Импорт всегда помещается в начало файла, сразу после любых комментариев модуля и строк документации, а также перед глобальными параметрами и константами модуля.

Импорт должен быть сгруппирован в следующем порядке:

  • импорт стандартных библиотек.
  • связанный импорт третьих сторон.
  • импорт локального приложения/библиотеки.

Вы должны поместить пустую строку между каждой группой импорта.

Абсолютный импорт рекомендуется, так как он обычно более удобочитаем и, как правило, ведет себя лучше (или, по крайней мере, дает более качественные сообщения об ошибках), если система импорта настроена неправильно (например, когда каталог внутри пакета заканчивается на sys.path):

import mypkg.sibling
from mypkg import sibling
from mypkg.sibling import example

Однако явный относительный импорт является приемлемой альтернативой абсолютному импорту, особенно при работе со сложными макетами пакетов, где использование абсолютного импорта было бы излишне многословным:

from . import sibling
from .sibling import example

Код стандартной библиотеки должен избегать сложных макетов пакетов и всегда использовать абсолютный импорт.

При импорте класса из модуля, содержащего классы, обычно можно написать так:

from myclass import MyClass
from foo.bar.yourclass import YourClass

Если это написание вызывает конфликты локальных имен, укажите их явно:

import myclass
import foo.bar.yourclass

и используйте «myclass.MyClass» и «foo.bar.yourclass.YourClass».

Импорта с подстановочными знаками () следует избегать, так как он делает неясным, какие имена присутствуют в пространстве имен, что сбивает с толку как читателей, так и многие автоматизированные инструменты. Существует один оправданный вариант использования импорта подстановочных знаков, который заключается в повторной публикации внутреннего интерфейса как части общедоступного API (например, перезапись реализации интерфейса на чистом Python определениями из необязательного модуля ускорителя и точными определениями, которые будут перезаписано заранее неизвестно).

from <module> import *

При повторной публикации имен таким образом по­‑прежнему применяются приведенные ниже рекомендации в отношении общедоступных и внутренних интерфейсов.

Имена Dunder уровня модуля

«Dunders» уровня модуля (т. е. имена с двумя начальными и двумя конечными символами подчеркивания), такие как __all__, __author__, __version__, и т. д., должны быть помещены после строки документации модуля, но перед любыми операторами импорта, кроме imports. Python предписывает, что импорт будущего должен появиться в модуле перед любым другим кодом, кроме строк документации: from __future__.

"""This is the example module.

This module does stuff.
"""

from __future__ import barry_as_FLUFL

__all__ = ['a', 'b', 'c']
__version__ = '0.1'
__author__ = 'Cardinal Biggles'

import os
import sys

Строковые цитаты​

В Python строки в одинарных и двойных кавычках — это одно и то же. Этот PEP не дает рекомендаций по этому поводу. Выберите правило и придерживайтесь его. Однако, если строка содержит символы одинарной или двойной кавычки, используйте другую, чтобы избежать обратной косой черты в строке. Это улучшает читаемость.

Для строк в тройных кавычках всегда используйте символы двойных кавычек, чтобы соответствовать соглашению о строках документации в PEP 257.

Пробелы в выражениях и инструкциях

Пэт Пивз

Избегайте лишних пробелов в следующих ситуациях:

Непосредственно внутри скобок, скобок или фигурных скобок:

подсказка
spam(ham[1], {eggs: 2})
осторожно
spam( ham[ 1 ], { eggs: 2 } )

Между завершающей запятой и следующей закрывающей скобкой:

подсказка
foo = (0,)
осторожно
bar = (0, )

Непосредственно перед запятой, точкой с запятой или двоеточием:

подсказка
if x == 4: print(x, y); x, y = y, x
осторожно
if x == 4 : print(x , y) ; x , y = y , x

Однако в срезе двоеточие действует как бинарный оператор и должно иметь одинаковое количество с обеих сторон (рассматривая его как оператор с самым низким приоритетом). В расширенном фрагменте оба двоеточия должны иметь одинаковый интервал. Исключение: когда параметр среза опущен, пробел опускается:

подсказка
ham[1:9], ham[1:9:3], ham[:9:3], ham[1::3], ham[1:9:]
ham[lower:upper], ham[lower:upper:], ham[lower::step]
ham[lower+offset : upper+offset]
ham[: upper_fn(x) : step_fn(x)], ham[:: step_fn(x)]
ham[lower + offset : upper + offset]
осторожно
ham[lower + offset:upper + offset]
ham[1: 9], ham[1 :9], ham[1:9 :3]
ham[lower : : upper]
ham[ : upper]

Непосредственно перед открывающей скобкой, с которой начинается список аргументов вызова функции:

подсказка
spam(1)
осторожно
spam (1)

Непосредственно перед открывающей скобкой, которая запускает индексацию или нарезку:

подсказка
dct['key'] = lst[index]
осторожно
dct ['key'] = lst [index]

Более одного пробела вокруг оператора присваивания (или другого), чтобы выровнять его с другим:

подсказка
x = 1
y = 2
long_variable = 3
осторожно
x             = 1
y             = 2
long_variable = 3

Другие рекомендации​

Избегайте пробелов в конце в любом месте. Поскольку обычно он невидим, это может сбивать с толку: например, обратная косая черта, за которой следует пробел, и перевод строки не считаются маркером продолжения строки. Некоторые редакторы не сохраняют его, и многие проекты (например, сам CPython) имеют хуки перед фиксацией, которые его отклоняют.

Всегда окружайте эти бинарные операторы одним пробелом с обеих сторон: присваивание (=), расширенное присваивание (+=, -= и т. д.), сравнения (==, <, >, !=, <>, <=, >=, in, not, not in, is not), логические операции (not, and, or, not).

Если используются операторы с разными приоритетами, рассмотрите возможность добавления пробелов вокруг операторов с наименьшим(и) приоритетом(ами). Используйте свое собственное суждение; однако никогда не используйте более одного пробела и всегда используйте одинаковое количество пробелов с обеих сторон бинарного оператора:

подсказка
i = i + 1
submitted += 1
x = x*2 - 1
hypot2 = x*x + y*y
c = (a+b) * (a-b)
осторожно
i=i+1
submitted +=1
x = x * 2 - 1
hypot2 = x * x + y * y
c = (a + b) * (a - b)

Аннотации функций должны использовать обычные правила для двоеточий и всегда иметь пробелы вокруг стрелки, если она присутствует. (Дополнительную информацию об аннотациях функций см. в разделе Аннотации функций ниже.):

подсказка
def munge(input: AnyStr): ...
def munge() -> PosInt: ...
осторожно
def munge(input:AnyStr): ...
def munge()->PosInt: ...

Не используйте пробелы вокруг =знака, когда он используется для указания аргумента ключевого слова или когда он используется для указания значения по умолчанию для неаннотированного параметра функции:

подсказка
def complex(real, imag=0.0):
    return magic(r=real, i=imag)
осторожно
def complex(real, imag = 0.0):
    return magic(r = real, i = imag)

Однако при объединении аннотации аргумента со значением по умолчанию используйте пробелы вокруг =знака:

подсказка
def munge(sep: AnyStr = None): ...
def munge(input: AnyStr, sep: AnyStr = None, limit=1000): ...
осторожно
def munge(input: AnyStr=None): ...
def munge(input: AnyStr, limit = 1000): ...

Составные операторы (несколько операторов в одной строке) обычно не рекомендуются:

подсказка
if foo == 'blah':
    do_blah_thing()
do_one()
do_two()
do_three()
А не:
осторожно
if foo == 'blah': do_blah_thing()
do_one(); do_two(); do_three()

Хотя иногда допустимо помещать операторы if/for/while с небольшим телом в ту же строку, никогда не делайте этого для операторов, состоящих из нескольких предложений. Также избегайте складывания таких длинных строк!

А не:

осторожно
if foo == 'blah': do_blah_thing()
for x in lst: total += x
while t < 10: t = delay()

Точно нет:

осторожно
if foo == 'blah': do_blah_thing()
else: do_non_blah_thing()

try: something()
finally: cleanup()

do_one(); do_two(); do_three(long, argument,
                             list, like, this)

if foo == 'blah': one(); two(); three()

Когда использовать конечные запятые​

Конечные запятые обычно необязательны, за исключением того, что они обязательны при создании кортежа из одного элемента. Для ясности рекомендуется заключать последнее в скобки (технически избыточные):

подсказка
FILES = ('setup.cfg',)
осторожно
FILES = 'setup.cfg',

Когда конечные запятые избыточны, они часто бывают полезны при использовании системы контроля версий, когда ожидается, что список значений, аргументов или импортированных элементов будет расширяться с течением времени. Шаблон заключается в том, чтобы помещать каждое значение (и т. д.) в отдельную строку, всегда добавляя завершающую запятую, и добавляя закрывающую скобку/скобку/скобку на следующей строке. Однако не имеет смысла ставить запятую на той же строке, что и закрывающий разделитель (за исключением приведенного выше случая одноэлементных кортежей):

подсказка
FILES = [
    'setup.cfg',
    'tox.ini',
    ]
initialize(FILES,
           error=True,
           )
осторожно
FILES = ['setup.cfg', 'tox.ini',]
initialize(FILES, error=True,)

Комментарии​

Комментарии, противоречащие коду, хуже, чем отсутствие комментариев. Всегда уделяйте приоритетное внимание обновлению комментариев при изменении кода!

Комментарии должны быть полными предложениями. Первое слово должно быть написано с большой буквы, если только это не идентификатор, начинающийся со строчной буквы (никогда не меняйте регистр идентификаторов!).

Блочные комментарии обычно состоят из одного или нескольких абзацев, построенных из полных предложений, каждое из которых заканчивается точкой.

Вы должны использовать два пробела после точки в конце предложения в комментариях, состоящих из нескольких предложений, за исключением последнего предложения.

Убедитесь, что ваши комментарии ясны и легко понятны другим носителям языка, на котором вы пишете.

Блокировать комментарии​

Блочные комментарии обычно применяются к некоторому (или всему) коду, следующему за ними, и имеют отступ на том же уровне, что и этот код. Каждая строка блочного комментария начинается с символа #и одиночного пробела (если это не текст с отступом внутри комментария).

Абзацы внутри блочного комментария разделяются строкой, содержащей одиночный #.

Встроенные комментарии​

Используйте встроенные комментарии с осторожностью.

Встроенный комментарий — это комментарий в той же строке, что и оператор. Встроенные комментарии должны быть отделены от оператора не менее чем двумя пробелами. Они должны начинаться с # и одного пробела.

Встроенные комментарии не нужны и на самом деле отвлекают, если они констатируют очевидное. Не делайте этого:

x = x + 1                 # Increment x

Но иногда это полезно:

x = x + 1                 # Compensate for border

Строки документации​

Соглашения о написании хороших строк документации (так называемые «строки документации») увековечены в PEP 257.

Напишите строки документации для всех общедоступных модулей, функций, классов и методов. Строки документации не нужны для закрытых методов, но у вас должен быть комментарий, описывающий, что делает метод. Этот комментарий должен стоять после defстроки.

PEP 257 описывает хорошие соглашения о строках документации. Обратите внимание, что наиболее важно то, """что конец многострочной строки документации должен находиться на отдельной строке:

"""Return a foobang

Optional plotz says to frobnicate the bizbaz first.
"""

Для строк документации с одним вкладышем, пожалуйста, держите закрытие """ на той же строке:

"""Return an ex­‑parrot."""

Соглашения об именах​

Существует множество различных стилей именования. Это помогает распознавать, какой стиль именования используется, независимо от того, для чего он используется.

Обычно различают следующие стили именования:

b(одна строчная буква)

B(одна заглавная буква)

lowercase

lower_case_with_underscores

UPPERCASE

UPPER_CASE_WITH_UNDERSCORES

CapitalizedWords(или CapWords, или CamelCase — названы так из­‑за неровного вида букв). Это также иногда называют StudlyCaps.

Примечание. При использовании акронимов в CapWords все буквы аббревиатуры должны быть заглавными. Таким образом, HTTPServerError лучше, чем HttpServerError.

mixedCase(отличается от CapitalizedWords начальной строчной буквой!)

Capitalized_Words_With_Underscores(уродливый!)

Существует также стиль использования короткого уникального префикса для группировки связанных имен. Это редко используется в Python, но упоминается для полноты картины. Например, os.stat()функция возвращает кортеж, элементы которого традиционно имеют такие имена, как, st_modeи т. д. (Это сделано, чтобы подчеркнуть соответствие с полями структуры системного вызова POSIX, что помогает программистам, знакомым с этим.)st_sizest_mtime

Библиотека X11 использует начальный X для всех своих общедоступных функций. В Python этот стиль обычно считается ненужным, поскольку имена атрибутов и методов имеют префикс объекта, а имена функций — имя модуля.

Кроме того, распознаются следующие специальные формы, использующие начальные или конечные символы подчеркивания (обычно их можно комбинировать с любым соглашением о регистре):

_single_leading_underscore: слабый показатель «внутреннего использования». Например, не импортирует объекты, имена которых начинаются с подчеркивания.

from M import *
single_trailing_underscore_:

используется по соглашению, чтобы избежать конфликтов с ключевым словом Python, например

tkinter.Toplevel(master, class_='ClassName')
__double_leading_underscore:

при присвоении имени атрибуту класса вызывает изменение имени (внутри класса FooBar __booстановится _FooBar__boo; см. ниже).

__double_leading_and_trailing_underscore__: «магические» объекты или атрибуты, которые живут в контролируемых пользователем пространствах имен. Например __init__, __import__ или __file__. Никогда не придумывайте таких имен; используйте их только в соответствии с документацией.

Проектирование для наследования​

Проще сделать свойство или метод общедоступным позже, чем сделать общедоступный атрибут закрытым.

Общедоступные атрибуты не должны иметь ведущих символов подчеркивания.

Избегайте использования свойств для ресурсоемких операций; нотация атрибута заставляет вызывающую сторону полагать, что доступ (относительно) дешев.

Если ваш класс предназначен для подкласса, и у вас есть атрибуты, которые вы не хотите использовать в подклассах, рассмотрите возможность их именования с двойным подчеркиванием в начале и без подчеркивания в конце. Это вызывает алгоритм изменения имени Python, где имя класса преобразуется в имя атрибута. Это помогает избежать конфликтов имен атрибутов, если подклассы непреднамеренно содержат атрибуты с одинаковыми именами.

Примечание 1. Обратите внимание, что в искаженном имени используется только простое имя класса, поэтому, если подкласс выбирает одно и то же имя класса и имя атрибута, вы все равно можете получить конфликты имен.

Примечание 2. Изменение имени может сделать некоторые виды использования, такие как отладка и __getattr__(), менее удобными. Однако алгоритм изменения имени хорошо задокументирован и его легко выполнить вручную.

Публичные и внутренние интерфейсы​

Любые гарантии обратной совместимости применяются только к общедоступным интерфейсам. Соответственно, важно, чтобы пользователи могли четко различать публичный и внутренний интерфейсы.

Задокументированные интерфейсы считаются общедоступными, если только документация явно не объявляет их временными или внутренними интерфейсами, на которые не распространяются обычные гарантии обратной совместимости. Все недокументированные интерфейсы следует считать внутренними.

Чтобы лучше поддерживать самоанализ, модули должны явно объявлять имена в своих общедоступных API, используя __all__атрибут. Установка __all__пустого списка указывает на то, что модуль не имеет общедоступного API.

Даже при __all__правильной настройке внутренние интерфейсы (пакеты, модули, классы, функции, атрибуты или другие имена) все равно должны иметь префикс с одним начальным символом подчеркивания.

Интерфейс также считается внутренним, если любое содержащее пространство имен (пакет, модуль или класс) считается внутренним.

Импортированные имена всегда следует рассматривать как деталь реализации. Другие модули не должны полагаться на косвенный доступ к таким импортированным именам, если только они не являются явно задокументированной частью API­‑интерфейса содержащего модуля, например, модуль os.pathпакета __init__, предоставляющий функциональность из подмодулей.

Рекомендации по программированию​

Сравнения с синглтонами, такими как None, всегда должны выполняться с помощью is или is not, а не с операторами равенства.

подсказка
if foo is not None:
осторожно
if not foo is None:

Всегда используйте оператор def вместо оператора присваивания, который связывает лямбда­‑выражение непосредственно с идентификатором:

подсказка
def f(x): return 2*x
осторожно
f = lambda x: 2*x

Первая форма означает, что имя результирующего функционального объекта конкретно «f», а не общее «лямбда». Это более полезно для трассировки и строковых представлений в целом. Использование оператора присваивания устраняет единственное преимущество, которое лямбда­‑выражение может предложить по сравнению с явным оператором def (то есть его можно встроить в более крупное выражение).

Наследуйте исключения от Exception, а не BaseException.

Создавайте иерархии исключений на основе различий, которые, вероятно, потребуются коду, перехватывающему исключения, а не на местах, где возникают исключения. Стремитесь ответить на вопрос «Что пошло не так?» программно, а не только заявлять, что «Произошла проблема» (см. PEP 3151 для примера этого урока, извлекаемого для встроенной иерархии исключений)

Здесь применяются соглашения об именах классов, хотя вы должны добавить суффикс «Error» к своим классам исключений, если исключение является ошибкой. Исключения без ошибок, которые используются для нелокального управления потоком или других форм сигнализации, не нуждаются в специальном суффиксе.

Правильно используйте цепочку исключений. следует использовать для указания явной замены без потери исходной трассировки.raise X from Y

При намеренной замене внутреннего исключения (с помощью) убедитесь, что в новое исключение перенесены соответствующие сведения (например, сохранение имени атрибута при преобразовании KeyError в AttributeError или встраивание текста исходного исключения в новое сообщение об исключении).

raise X from None

При перехвате исключений по возможности упоминайте конкретные исключения вместо использования голого except:предложения:

try:
    import platform_specific_module
except ImportError:
    platform_specific_module = None

Голое except:предложение будет перехватывать исключения SystemExit и KeyboardInterrupt, что усложнит прерывание программы с помощью Control­‑C и может замаскировать другие проблемы. Если вы хотите перехватывать все исключения, сигнализирующие об ошибках программы, используйте (голый, кроме, эквивалентен).except Exception:except BaseException:

Хорошее эмпирическое правило состоит в том, чтобы ограничить использование голых предложений «кроме» двумя случаями:

Если обработчик исключений будет распечатывать или регистрировать трассировку; по крайней мере, пользователь будет знать, что произошла ошибка.

Если код должен выполнить некоторую очистку, но затем позволяет исключению распространяться вверх с помощью raise. try...finally может быть лучшим способом справиться с этим случаем.

При отлове ошибок операционной системы отдавайте предпочтение явной иерархии исключений, представленной в Python 3.3, а не самоанализу errno значений.

Кроме того, для всех предложений try/except ограничьте tryих абсолютным минимумом необходимого объема кода. Опять же, это позволяет избежать маскировки ошибок:

подсказка
try:
    value = collection[key]
except KeyError:
    return key_not_found(key)
else:
    return handle_value(value)
осторожно
try:
    # Too broad!
    return handle_value(collection[key])
except KeyError:
    # Will also catch KeyError raised by handle_value()
    return key_not_found(key)

Когда ресурс является локальным для определенного раздела кода, используйте withоператор, чтобы обеспечить его быструю и надежную очистку после использования. Оператор try/finally также допустим.

Менеджеры контекста должны вызываться через отдельные функции или методы всякий раз, когда они делают что­‑то кроме получения и высвобождения ресурсов:

подсказка
with conn.begin_transaction():
    do_stuff_in_transaction(conn)
осторожно
with conn:
    do_stuff_in_transaction(conn)

Последний пример не предоставляет никакой информации, указывающей на то, что методы __enter__и __exit__делают что­‑то кроме закрытия соединения после транзакции. В этом случае важно быть явным.

Используйте ''.startswith()и ''.endswith() вместо нарезки строки для проверки префиксов или суффиксов.

startwith() и endwith() чище и менее подвержены ошибкам:

подсказка
if foo.startswith('bar'):
осторожно
if foo[:3] == 'bar':

При сравнении типов объектов всегда следует использовать isinstance() вместо прямого сравнения типов:

подсказка
if isinstance(obj, int):
осторожно
if type(obj) is type(1):

Для последовательностей (строки, списки, кортежи) используйте тот факт, что пустые последовательности являются ложными:

подсказка
if not seq:
if seq:
осторожно
if len(seq):
if not len(seq):

Не пишите строковые литералы, которые полагаются на значительные пробелы в конце. Такие конечные пробелы визуально неразличимы, и некоторые редакторы (или, в последнее время, reindent.py) обрезают их.

Не сравнивайте логические значения с True или False, используя ==:

подсказка
if greeting:
осторожно
if greeting == True:

Худший:

осторожно
if greeting is True:

Не рекомендуется использовать операторы управления потоком return// внутри набора finally break, где оператор управления потоком выходит за пределы набора finally. Это связано с тем, что такие операторы неявно отменяют любое активное исключение, которое распространяется через набор finally:continuetry...finally

осторожно
def foo():
    try:
        1 / 0
    finally:
        return 42

Аннотации функций​

С принятием PEP 484 правила стиля для аннотаций функций изменились.

Аннотации функций должны использовать синтаксис PEP 484 (некоторые рекомендации по форматированию аннотаций приведены в предыдущем разделе).

Эксперименты со стилями аннотаций, которые ранее рекомендовались в этом PEP, больше не поощряются.

Однако за пределами stdlib теперь поощряются эксперименты в рамках правил PEP 484. Например, разметка большой сторонней библиотеки или приложения с помощью аннотаций типа стиля PEP 484, проверка того, насколько легко было добавить эти аннотации, и наблюдение за тем, повышает ли их присутствие понятность кода.

Стандартная библиотека Python должна быть консервативна в принятии таких аннотаций, но их использование разрешено для нового кода и для больших рефакторингов.

Для кода, который хочет по­‑другому использовать аннотации функций, рекомендуется поместить комментарий в форме:

# type: ignore

в верхней части файла; это говорит программам проверки типов игнорировать все аннотации. (Более подробные способы отключения жалоб от средств проверки типов можно найти в PEP 484.)

Как и линтеры, средства проверки типов являются необязательными, отдельными инструментами. Интерпретаторы Python по умолчанию не должны выдавать никаких сообщений из­‑за проверки типов и не должны изменять свое поведение на основе аннотаций.

Пользователи, которые не хотят использовать средства проверки типов, могут их игнорировать. Однако ожидается, что пользователи пакетов сторонних библиотек могут захотеть запустить проверку типов для этих пакетов. Для этой цели PEP 484 рекомендует использовать файлы­‑заглушки: файлы .pyi, которые читаются средством проверки типов вместо соответствующих файлов .py. Файлы­‑заглушки могут распространяться вместе с библиотекой или отдельно (с разрешения автора библиотеки) через репозиторий typeshed.

Аннотации переменных​

PEP 526 представил переменные аннотации. Рекомендации по стилю для них аналогичны рекомендациям по аннотациям функций, описанным выше:

Аннотации для переменных уровня модуля, переменных класса и экземпляра, а также локальных переменных должны иметь один пробел после двоеточия.

Перед двоеточием не должно быть пробела.

Если присваивание имеет правую часть, то знак равенства должен иметь ровно один пробел с обеих сторон:

подсказка
code: int

class Point:
    coords: Tuple[int, int]
    label: str = '<unknown>'
осторожно
code:int  # No space after colon
code : int  # Space before colon

class Test:
    result: int=0  # No spaces around equality sign
