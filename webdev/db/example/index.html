Пример выполнения задания

Проектирование БД​

Выполним диаграмму «сущность — связь» (ERD).

Создание БД​

  1. Запустим клиент СУБД MySQL:

    mysql --user=root --password

    Введём пароль пользователя БД.

    Должно быть выдано приглашение ко вводу команд:

    mysql>
    К сведению

    По умолчанию сервер MySQL, реализованный как демон (фоновый процесс) mysqld, занимает порт 3306, и клиент подаёт запросы на этот же порт. Это обусловлено следующими строками в конфигурационном файле:

    [mysqld]
    port=3306

    [client]
    port=3306

    Если мы сменили IP­‑адрес или номер порта, то их следует явно указывать при подаче команд, например

    mysql --user=root --password --bind-address=127.0.0.1 --port=3306

    Чтобы узнать номер порта, подайте команду

    • Windows
    • Linux
    • macOS
    mysqld --help --verbose | findstr /b /c:"port "
    mysqld --help --verbose | grep '^port '
    mysqld --help --verbose | grep '^port '

  2. Создадим БД shop, подав команду

    CREATE SCHEMA shop
    DEFAULT CHARACTER SET utf8mb4;
    -- Сравнение `utf8mb4_0900_ai_ci` будет установлено автоматически.
    -- Удалить БД можно, подав запрос `DROP SCHEMA shop;`.

    Должно быть выведено:

    Query OK, 1 row affected...

  3. Остановим клиент СУБД MySQL, подав команду

    EXIT;

    Должно быть выведено:

    Bye

  4. Выведем перечень БД, находящихся под управлением MySQL, и убедимся, что БД shop создана.

    mysqlshow --user=root --password

    Введём пароль пользователя БД.

    Должно быть выведено:

    +-----------+
    | Databases |
    +-----------+
    | ...       |
    | shop      |
    | ...       |
    +-----------+

Настройка соединения между приложением БД и СУБД​

Конфиденциальные сведения, необходимые для взаимодействия Laravel с программным окружением, следует хранить в скрытом файле .env, который размещён в корневом каталоге проекта и не подлежит фиксации в системе контроля версий.

  1. Откроем файл .env на редактирование, найдём следующие переменные и, если требуется, изменим их значения.

    Фрагмент файла .env
    DB_CONNECTION=mysql
    DB_HOST=127.0.0.1
    DB_PORT=3306
    DB_DATABASE=shop
    DB_USERNAME=root
    DB_PASSWORD=

    Пояснения к переменным:

    • DB_CONNECTION — идентификатор СУБД;
    • DB_HOST — IP­‑адрес, используемый демоном (фоновым процессом) СУБД;
    • DB_PORT — номер порта, занятого демоном СУБД;
    • DB_DATABASE — имя БД;
    • DB_USERNAME — имя пользователя БД;
    • DB_PASSWORD — пароль пользователя БД.
    Имя и пароль пользователя БД

    Во многих «сборках» для Windows, в т. ч. в XAMPP, пользователю root в MySQL пароль не назначен. В Laravel, в файле .env, эта комбинация (имя пользователя root с «пустым» паролем) используется по умолчанию.

    В некоторых версиях Open Server Panel пользователю root назначен пароль root.

    Пользователям MAMP (для операционной системы macOS) придётся добавить в файл .env ещё одну строчку:

    DB_SOCKET=/Applications/MAMP/tmp/mysql/mysql.sock

  2. Сохраним изменения в файле .env.

Создание сценария миграции​

  1. Перейдём в каталог проекта:

    cd project

  2. Описание данных интуитивно легче начать с главных таблиц — таких, которые не содержат столбцов во внешних ключах. На выполненной нами диаграмме «сущность — связь» (ERD) заданному правилу соответствуют две сущности (будущих таблицы): products («Товары») и tags («Теги»). Начнём в алфавитном порядке и создадим связанную с сущностью products модель данных Product и несколько шаблонов компонентов (модулей) для приложения БД.

    php artisan make:model --all --api Product
    Имя модели имеет значение!

    Имя модели должно быть именем существительным, должно быть приведено на английском языке, в единственном числе (Product, а не products!), в верхнем верблюжьем регистре (UpperCamelCase). Желательно, чтобы имя состояло из одного слова.

    Дело в том, что Laravel опирается на имя модели при именовании компонентов (модулей), причём для классов приложения БД он использует форму единственного числа, а для таблиц БД подбирает форму множественного числа — регулярную или нерегулярную: Product — products, Student — students, но Aircraft (ед. ч.) — aircraft (мн. ч.), Child — children, Data (ед. ч.) — data (мн. ч.), Man — men, Species (ед. ч.) — species (мн. ч.).

    Должно быть выведено:

    INFO Model... created successfully.

    INFO Factory... created successfully.

    INFO Migration... created successfully.

    INFO Seeder... created successfully.

    INFO Request... created successfully.

    INFO Request... created successfully.

    INFO Controller... created successfully.

    INFO Policy... created successfully.

    В результате выполнения команды должны быть созданы следующие модули:

    • сценарий миграции БД — в файле database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php (элементы YYYY, MM, DD, hh, mm и ss обозначают соответственно год, месяц, день, час, минуту и секунду создания сценария);
    • модель объекта предметной области на уровне приложения БД — в файле app/Models/Product.php;
    • сеялка, заполняющая БД, — в файле database/seeders/ProductSeeder.php;
    • фабрика, подготавливающая данные для сеялки, — в файле database/factories/ProductFactory.php;
    • ресурсный контроллер с методами, подходящими для реализации API, — в файле app/Http/Controllers/ProductController.php;
    • обработчики для HTTP­‑запросов на создание и корректировку записи в БД — в файлах app/Http/Requests/StoreProductRequest.php и app/Http/Requests/UpdateProductRequest.php;
    • политика доступа — в файле app/Policies/ProductPolicy.php.

  3. Откроем на редактирование файл database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php.

Сценарий миграции БД записан на языке программирования PHP. Сценарий представлен анонимным классом, который наследует от класса Migration и переопределяет методы up и down, отвечающие соответственно за создание и удаление таблицы products. Рассмотрим эти методы подробнее.

Метод up​

Метод up по умолчанию реализован следующим образом:

Фрагмент файла database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php
public function up(): void
{
    Schema::create('products', function (Blueprint $table) {
        $table->id();
        $table->timestamps();
    });
}

В приведённом фрагменте исходного кода происходит статический (через ::) вызов из класса Schema метода create с двумя аргументами: именем создаваемой таблицы ('products') и анонимной функцией обратного вызова, которая автоматически принимает в качестве параметра объект $table класса Blueprint. Это — так называемый строитель схем (слово «схема» здесь — синоним словосочетания «база данных», к диаграммам, т. е. к графическому представлению схемы, строитель непосредственного отношения не имеет); он позволяет описать структуру таблицы.

При вызове метода up происходит следующее (на примере СУБД MySQL):

  • вызов метода id создаёт в таблице products столбец id беззнакового целого типа данных BIGINT UNSIGNED с запретом пустых значений (NOT NULL), автоинкрементом и первичным ключом;
  • вызов метода timestamps создаёт в таблице products два столбца типа данных TIMESTAMP для меток времени создания (created_at) и последнего изменения (updated_at) строки таблицы.
Что ещё умеет строитель схем?

Строитель схем, встроенный в Laravel, располагает множеством методов для описания данных: https://laravel.com/docs/10.x/migrations#available-column-types.

Все методы реализуют «текучий интерфейс» (англ. fluent interface), т. е. возвращают ссылку на объект, из которого вызваны, поэтому вызовы можно объединять в цепочки для повышения удобочитаемости.

Опишем структуру таблицы.

  1. Для хранения наименования товара добавим столбец name с небольшой максимальной длиной (в символах) и запретом на повтор значений:

    Фрагмент файла database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php
    $table->string('name', 55)
          ->collation('utf8mb4_0900_ai_ci')
          ->unique()
          ;

    СУБД MySQL при миграции БД создаст столбец name типа данных VARCHAR(55). Кроме того, будет добавлен индекс products_name_unique, запрещающий повторы значений в указанном столбце.

    Почему именно 55?

    1. Тип данных VARCHAR параметризован максимальной допустимой длиной строки, измеряемой в символах, тогда как на уровне физической схемы БД длина строки измеряется в байтах с учётом набора символов — так называемой «кодировки». Так, набор utf8mb4 в MySQL 8 выделяет на каждый символ от 1 до 4 байтов: латинская буква занимает 1 байт, русская — 2 байта и т. д.

      Кроме того, MySQL 8 хранит информацию о длине строки (в символах) в значении столбца типа данных VARCHAR: если длина лежит в интервале [0;28−1][0; 2^{8} - 1][0;28−1], то информация занимает один старший байт; длина в интервале [28;216−1][2^{8}; 2^{16} - 1][28;216−1] означает, что будут заняты два старших байта. Строки большей длины VARCHAR не поддерживает.

      Таким образом, при использовании набора символов utf8mb4 в VARCHAR можно хранить строку, длина которой в символах не превышает

      ⌊216−14⌋=16383.\left\lfloor \frac{2^{16} - 1}{4} \right\rfloor = 16383.⌊4216−1​⌋=16383.

      При этом информация о длине строки (в символах) займёт оба старших байта. На 1 Б можно записать информацию о такой длине строки, которая в символах не превышает

      ⌊28−14⌋=63.\left\lfloor \frac{2^{8} - 1}{4} \right\rfloor = 63.⌊428−1​⌋=63.

    2. Поисковые системы ограничивают длину заголовка веб­‑страницы. Заголовок сайта МПТ в поисковой выдаче Google ограничен приблизительно 55 знаками.

      Конечно, в заголовке веб­‑страницы обычно выводят не только название самой веб­‑страницы, но и название сайта. Поэтому следует ориентироваться на число 55 (или меньшее) как на максимальную допустимую длину наименования товара.

  2. Для хранения цены товара добавим столбец price, позволяющую хранить денежные величины:

    Фрагмент файла database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php
    $table->decimal('price', 7, 2)
          ->index()
          ;

    СУБД MySQL при миграции БД создаст столбец price типа данных DECIMAL(7, 2). Он позволит хранить 7­‑разрядное число, два младших разряда которого отведены под дробную часть (например, копейки или центы). Таким образом, возможные значения столбца будут лежать в интервале [−99999,99;99999,99][-99999,99; 99999,99][−99999,99;99999,99]. Благодаря методу index значения столбца будут проходить индексацию при вставке, что ускорит поиск.

    Справка

    Точность NUMERIC определяет стандарт SQL, а точность DECIMAL зависит от СУБД. В MySQL эти типы данных реализованы одинаково.

    Не используйте float или double для денежных величин

    Широко известные типы данных, реализующие стандарт IEEE 754, не вполне пригодны для денежных операций, так как не обеспечивают должную точность представления чисел. Так, при использовании float или double сумма 0,10 ₽ и 0,70 ₽ будет записана как 0,7⁠(9) ₽.

  3. Добавим столбец description для описания товара:

    Фрагмент файла database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php
    $table->text('description')
          ->collation('utf8mb4_0900_ai_ci')
          ;

    СУБД MySQL при миграции БД создаст столбец description типа данных TEXT.

  4. Добавим столбец, позволяющий хранить число единиц товара на складе:

    Фрагмент файла database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php
    $table->integer('quantity')
          ->unsigned()
          ->default(0)
          ;

    СУБД MySQL при миграции БД создаст столбец quantity целого неотрицательного типа данных INT UNSIGNED со значением 0 по умолчанию. Таким образом, если значение столбца не задано при вставке, то считается, что на складе не осталось данного товара.

  5. Так как товар может быть снят с продажи, потребуется столбец бу́лева типа данных.

    Фрагмент файла database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php
    $table->boolean('available')
          ->default(false)
          ;

    СУБД MySQL при миграции БД создаст столбец available целочисленного типа данных TINYINT со значением 0 по умолчанию. Другими словами, если значение столбца не задано при вставке, то считается, что товар недоступен для заказа. Обратного эффекта можно добиться, заменив default(false) на default(true) в сценарии.

  6. Убедимся, что метод up приобрёл следующий вид:

    Фрагмент файла database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php
    public function up(): void
    {
        Schema::create('products', function (Blueprint $table) {
            $table->id();
            $table->string('name', 55)
                ->collation('utf8mb4_0900_ai_ci')
                ->unique();
            $table->decimal('price', 7, 2)
                ->index();
            $table->text('description')
                ->collation('utf8mb4_0900_ai_ci');
            $table->integer('quantity')
                ->default(0);
            $table->boolean('available')
                ->default(false);
            $table->timestamps();
        });
    }

Метод down​

Метод down имеет следующий вид:

Фрагмент файла database/migrations/YYYY_MM_DD_hhmmss_create_products_table.php
public function down(): void
{
    Schema::dropIfExists('products');
}

Если таблица products существует, то она будет удалена при откате миграции.

Отладка​

Команда php artisan migrate находит ранее не выполненные сценарии миграции и вызывает реализованные в них методы up. Впрочем, перед миграцией следует убедиться, что сценарии не содержат ошибок. Осуществим отладочную печать предложений SQL, подготовленных Laravel, подав ту же команду с флагом --pretend.

php artisan migrate --pretend

Флаг --pretend сигнализирует, что интерпретация сценариев миграции должна быть осуществлена без воздействия на БД. В консоль должны быть выведены примерно такие предложения SQL (исходный код отформатирован для удобства чтения):

CREATE TABLE products (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(55) COLLATE utf8mb4_0900_ai_ci NOT NULL,
  price DECIMAL(7, 2) NOT NULL,
  description TEXT COLLATE utf8mb4_0900_ai_ci NOT NULL,
  quantity INT UNSIGNED NOT NULL DEFAULT 0,
  available TINYINT(1) NOT NULL DEFAULT 0,
  created_at TIMESTAMP NULL,
  updated_at TIMESTAMP NULL
) DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

ALTER TABLE products ADD UNIQUE products_name_unique(name);
ALTER TABLE products ADD INDEX products_price_index(price);

Выполнение миграции и её откат​

  1. Выведем перечень таблиц, размещённых в БД shop, подав команду

    mysqlshow --user=root --password -- shop

    Введём пароль пользователя БД.

    Должно быть выведено сообщение, подтверждающее, что таблиц в БД нет:

    Database: shop
    +--------+
    | Tables |
    +--------+
    +--------+

  2. Для выполнения всех сценариев миграции, которые размещены в каталоге database/migrations/ и не были выполнены ранее (в данном случае их пять), следует подать ту же команду php artisan migrate, но без флага --pretend. Сценарии будут выполнены в порядке возрастания даты и времени, указанных в именах файлов.

    php artisan migrate

    Должно быть выведено:

    INFO Running migrations.

    ..._create_users_table ............................................ ... DONE
    ..._create_password_resets_table .................................. ... DONE
    ..._create_failed_jobs_table ...................................... ... DONE
    ..._create_personal_access_tokens_table ........................... ... DONE
    ..._create_products_table ......................................... ... DONE
    Если возникла проблема

    Если при выполнении команды не был найден сценарий миграции, то следует выполнить команду composer dump-autoload из каталога проекта.

  3. Для проверки повторно подадим команду

    php artisan migrate

    Должно быть выведено:

    INFO Nothing to migrate.

    Это означает, что невыполненных сценариев миграции не осталось.

  4. Выведем перечень таблиц, размещённых в БД shop, подав команду

    mysqlshow --user=root --password -- shop

    Введём пароль пользователя БД.

    Должно быть выведено:

    Database: shop
    +------------------------+
    |         Tables         |
    +------------------------+
    | failed_jobs            |
    | migrations             |
    | password_resets        |
    | personal_access_tokens |
    | products               |
    | users                  |
    +------------------------+

  5. Обратим внимание, что Laravel создал в БД таблицу migrations, сценарий для которой отсутствует в каталоге database/migrations/. В этой таблице Laravel регистрирует выполненные миграции.

    Подадим команду на извлечение всех записей из этой таблицы:

    mysql --user=root --password --execute="SELECT * FROM migrations;" -- shop

    Введём пароль пользователя БД.

    Должно быть выведено:

    +----+-----------------------------------------+-------+
    | id | migration                               | batch |
    +----+-----------------------------------------+-------+
    |  1 | ..._create_users_table                  |     1 |
    |  2 | ..._create_password_resets_table        |     1 |
    |  3 | ..._create_failed_jobs_table            |     1 |
    |  4 | ..._create_personal_access_tokens_table |     1 |
    |  5 | ..._create_products_table               |     1 |
    +----+-----------------------------------------+-------+

    Значение в столбце batch указывает на порядковый номер сеанса, в рамках которого выполнен сценарий миграции. Мы подали команду php artisan migrate впервые, и рамках этого первого сеанса были выполнены все перечисленные сценарии.

  6. Убедимся в правильности структуры таблицы products. Подадим команду:

    mysql --user=root --password --execute="DESCRIBE products;" -- shop

Отношение 1:М​

Сущности images («Изображения») и products («Товары») в реляционной модели данных связаны отношением «один ко многим»: по требованию заказчика на каждом изображении (например, фотоснимке) представлен только один товар, но каждый товар представлен на многих изображениях.

  1. Создадим связанную с сущностью images модель данных Image и несколько шаблонов компонентов (модулей) для приложения БД.

    php artisan make:model --all --api Image

    Должно быть выведено:

    INFO Model... created successfully.
    ...
    INFO Policy... created successfully.

  2. Откроем на редактирование файл database/migrations/YYYY_MM_DD_hhmmss_create_images_table.php.

  3. Придадим методу up следующий вид:

    public function up(): void
    {
        Schema::create('images', function (Blueprint $table) {
            $table->id();
            $table->bigInteger('product_id') // Поле product_id целого типа данных
                ->unsigned()               // (беззнакового).
            ;
            $table->foreign('product_id')    // Внешний ключ на столбце product_id
                ->references('id')         // ссылается на столбец id
                ->on('products')           // в таблице products.
                ->onDelete('CASCADE')      // Фото удаляются с товаром.
                ->onUpdate('RESTRICT')     // Нельзя изменять id в products.
            ;
            $table->string('path', 191)
                ->collation('utf8mb4_0900_ai_ci')
                ->unique();
            $table->timestamps();
        });
    }

  4. Подадим команду на вывод предложений SQL, подготовленных Laravel:

    php artisan migrate --pretend

    Должно быть выведено (исходный код отформатирован для удобства чтения):

    CREATE TABLE images (
      id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY,
      product_id BIGINT UNSIGNED NOT NULL,
      path VARCHAR(191) COLLATE utf8mb4_0900_ai_ci NOT NULL,
      created_at TIMESTAMP NULL,
      updated_at TIMESTAMP NULL
    ) DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

    ALTER TABLE images
    ADD CONSTRAINT images_product_id_foreign
    FOREIGN KEY (product_id)
    REFERENCES products (id)
    ON DELETE CASCADE
    ON UPDATE RESTRICT;

    ALTER TABLE images
    ADD UNIQUE images_path_unique (path);

  5. Отдадим команду на выполнение сценариев миграции:

    php artisan migrate

Отношение М:М​

Сущности tags («Теги») и products («Товары») в реляционной модели данных связаны отношением «многие ко многим»: каждым тегом могут быть помечены многие товары, и каждый товар может быть помечен многими тегами.

  1. Создадим связанную с сущностью tags модель данных Tag и несколько шаблонов компонентов (модулей) для приложения БД.

    php artisan make:model --all --api Tag

    Должно быть выведено:

    INFO Model... created successfully.
    ...
    INFO Policy... created successfully.

  2. Откроем на редактирование файл database/migrations/YYYY_MM_DD_hhmmss_create_tags_table.php.

  3. Придадим методу up следующий вид:

    public function up(): void
    {
        Schema::create('tags', function (Blueprint $table) {
            $table->id();
            $table->string('label', 55)
                ->collation('utf8mb4_0900_ai_ci')
                ->unique();
            $table->timestamps();
        });
    }

  4. Подадим команду на вывод предложений SQL, подготовленных Laravel:

    php artisan migrate --pretend

    Должно быть выведено (исходный код отформатирован для удобства чтения):

  5. Создадим сценарий миграции для таблицы, которая связывает таблицы «Тег» и «Товар».

    Имена имеют значение

    Обратите внимание, что название файла должно быть записано в формате «create_x_y_table», где x и y — имена связываемых таблиц (в единственном числе и перечисленные в алфавитном порядке!).

  6. Придадим методу up следующий вид:

    public function up(): void
    {
        Schema::create('product_tag', function (Blueprint $table) {
            $table->id();

                ->unsigned()               // (беззнакового).
            ;
            $table->bigInteger('tag_id')     // Поле tag_id целого типа данных
                ->unsigned()               // (беззнакового).
            ;
            $table->foreign('product_id')    // Внешний ключ на столбце product_id
                ->references('id')         // ссылается на столбец id
                ->on('products')           // в таблице products.
                ->onDelete('CASCADE')      // Связь удаляется с товаром.
                ->onUpdate('RESTRICT')     // Нельзя изменять id в products.
            ;
            $table->foreign('tag_id')        // Внешний ключ на столбце tag_id
                ->references('id')         // ссылается на столбец id
                ->on('tags')               // в таблице tags.
                ->onDelete('CASCADE')      // Связь удаляется с тегом.
                ->onUpdate('RESTRICT')     // Нельзя изменять id в tags.
            ;
            $table->timestamps();
        });
    }
