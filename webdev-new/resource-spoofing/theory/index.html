Теоретические сведения

Ресурсы и подресурсы​

Всемирная паутина (англ. World­‑wide web, WWW или Web) — это информационное пространство, элементам которого, называемым ресурсами, присвоены единообразные идентификаторы (URI) [1, разд. 1]. Например, URI https://example.com/index.html указывает на ресурс, представляющий собой документ HTML.

Подресурс (субресурс) — это ресурс, который извлекается при обработке другого ресурса. Подресурсами документа HTML могут являться файлы, содержащие CSS, сценарии на языке JavaScript, шрифты и др.

Нередко веб­‑разработчики размещают подресурсы на других доменах, в т. ч. в сетях доставки контента — content delivery networks, сокращённо CDN. В некоторых случаях это позволяет ускорить их доставку клиенту.

Атака subresource spoofing​

Субъект атаки (например, автор подресурса, взломщик или сотрудник CDN) может изменить содержимое подресурса, оставив URI подресурса прежним, или перенаправить запрос клиента на другой подресурс. Такие атаки называют subresource spoofing («подмена подресурса»).

Объектом атаки является пользовательский агент (браузер), который получает и выполняет потенциально опасное содержимое подресурса.

Между прочим, злонамеренность авторов библиотек — не редкость: так, в нескольких пакетах Node.js обнаружили код для cryptojacking’а — несанкционированного использования вычислительных мощностей устройств, на которых выполняется сценарий, для майнинга криптовалют; координация расчётов велась через сервисы Coinhive, CoinIMP и т. п.

Нет CDN — нет subresource spoofing?

В недавнем прошлом пользовательские агенты, в т. ч. браузеры, обрабатывали подресурсы таким образом, что подключение одной и той же версии подресурса (например, популярной библиотеки Bootstrap) из одной и той же CDN ко множеству веб­‑сайтов позволяло пользовательскому агенту (браузеру) при обращении к одному из этих веб­‑сайтов скачать и кешировать содержимое подресурса на срок, установленный CDN, а при последующих обращениях в течение этого срока заменить повторное скачивание извлечением из кеша, с локального носителя (т. е. с жёсткого диска клиента), что ускоряло отображение веб­‑сайтов.

Изолированный кеш, который разработчики браузеров внедрили в связи с реализацией в 2019 г. атаки HTTP cache cross­‑site leaks (XSLeaks), в значительной мере обесценил идею CDN. Теперь содержимое подресурса, подключённого к нескольким веб­‑сайтам из одной и той же CDN, хранится у клиента в виде изолированных копий. Если тот же подресурс будет подключён к ещё одному веб­‑сайту, то содержимое будет заново скачано браузером, а не взято из кеша.

Однако исчезновение CDN не решит проблему resource spoofing, так как подмена подресурса не обязательно связана с использованием CDN.

Защита посредством SRI​

Для борьбы с resource spoofing Консорциум Всемирной паутины (W3C) издал спецификацию Subresource integrity (далее — SRI). Она определяет алгоритм, позволяющий пользовательскому агенту (например, браузеру) проверить целостность некоторых видов подресурсов.

На данный момент существует рекомендация 2016 г. и редакторский черновик за 10.07.2025.

Алгоритм​

  1. Веб­‑разработчик применяет хеш­‑функцию SHA‑256, SHA‑384 или SHA‑512 к содержимому подресурсов (файлов CSS, JavaScript и пр.), чтобы вычислить их контрольные суммы.

  2. Веб­‑разработчик подключает подресурсы к документу HTML посредством элементов link и script. (Поддержка a, audio, embed, iframe, img, object, source, track, video ожидается в последующих ревизиях SRI.)

  3. Веб­‑разработчик дополняет элементы link и script атрибутами integrity и crossorigin.

    <link rel="stylesheet"
        href="/путь/к/стилю.css"
        integrity="sha256-47DEQpj8HBSa+/TImW+5JCeuQeRkm5NMpJWZG3hSuFU="
        crossorigin="anonymous"
        >
    <script defer
            src="/путь/к/сценарию.js"
            integrity="sha256-47DEQpj8HBSa+/TImW+5JCeuQeRkm5NMpJWZG3hSuFU="
            crossorigin="anonymous"
            ></script>

    Значение атрибута integrity состоит из двух частей, разделённых дефисоминусом. Левая часть может принимать значения sha256, sha384 или sha512 и таким образом идентифицирует хеш­‑функцию, применённую для вычисления контрольной суммы. Правая часть содержит эту сумму, закодированную по схеме base64 согласно RFC 4648.

    Атрибут crossorigin со значением anonymous разрешает пользовательскому агенту (браузеру) без аутентификации запрашивать подресурсы, размещённые на других веб­‑сайтах (cross­‑origin resource sharing, CORS).

  4. Пользовательский агент (браузер) запрашивает ресурс и приступает к скачиванию подресурсов.

    CORS позволяет загружать содержимое подресурсов только по ограниченному набору схем: data (не рекомендуется), http (не рекомендуется) и https, поэтому URL со схемой file не будет обработан (см. консоль по F12):

    Access to script at 'file:///путь/к/сценарию.js' from origin 'null' has been blocked by CORS policy: Cross origin requests are only supported for protocol schemes: http, data, chrome, chrome-extension, https.
    Access to CSS stylesheet at 'file:///путь/к/стилю.css' from origin 'null' has been blocked by CORS policy: Cross origin requests are only supported for protocol schemes: http, data, chrome, chrome-extension, https.

    Можно попробовать убрать атрибут crossorigin, но без него SRI не сработает:

    Subresource Integrity: The resource 'file:///путь/к/сценарию.js' has an integrity attribute, but the resource requires the request to be CORS enabled to check the integrity, and it is not. The resource has been blocked because the integrity cannot be enforced.
    Subresource Integrity: The resource 'file:///путь/к/стилю.css' has an integrity attribute, but the resource requires the request to be CORS enabled to check the integrity, and it is not. The resource has been blocked because the integrity cannot be enforced.

  5. Пользовательский агент (браузер) анализирует атрибут integrity. Узнав, какую хеш­‑функцию применял автор, агент рассчитывает контрольную сумму содержимого подресурса и сверяет результат с тем, что записано в значении атрибута. Если хеши не совпали, то подресурс не будет применён, а в консоль (F12) будет выведено сообщение с тем хешем, который вычислил пользовательский агент (браузер):

    Failed to find a valid digest in the 'integrity' attribute for resource 'http://127.0.0.1:8000/script.js' with computed SHA-256 integrity 'A77/FIoHOGjeI5i/rPLyY3QGTSXkJ3r+pbdozEsKtTg='. The resource has been blocked.
    Failed to find a valid digest in the 'integrity' attribute for resource 'http://127.0.0.1:8000/style.css' with computed SHA-256 integrity 'WwvhFMMGzGuT3vWhT1U/DIrGBH0hL/MIurzYJonkfpQ='. The resource has been blocked.

Условия применения​

Для полноценной работы SRI необходимо соблюсти ряд условий:

  • как говорилось выше, хост подресурса должен разрешить совместное использование своих ресурсов разными источниками (англ. cross­‑origin resource sharing, CORS), передав в ответном сообщении HTTP заголовок Access-Control-Allow-Origin. Его значение * означает, что к ресурсам сервера можно обратиться из любого домена;

    <VirtualHost *:80>
        ServerName доменное.имя
        DocumentRoot "путь/к/каталогу/с/сайтом"
        Header always set Access-Control-Allow-Origin "*"
    </VirtualHost>

  • отсутствие уязвимости для атаки MiTM, в результате которой и содержимое подресурса, и значение атрибута inregrity могут быть подменены во время передачи по каналу связи. Контроль целостности информации в этом случае может обеспечить протокол TLS (схема https);

  • отсутствие у злоумышленника возможности воздействовать на объектную модель документа HTML в пользовательском агенте (браузере). Вообще, случаи, когда злоумышленник может инъецировать в содержимое веб­‑страницы элемент script со ссылкой на потенциально вредоносный сценарий, выходят за рамки ответственности SRI — потребуется устранить уязвимости для XSS и вести белый список источников посредством Content-Security-Policy: script-src;

  • оценка безопасности подресурсов: SRI не поможет, если разработчик веб­‑сайта опрометчиво или умышленно создаст ссылку на непроверенный или заведомо вредоносный сценарий и укажет правильную контрольную сумму.

Кроме того, ограничения SRI, установленные в отношение подресурса, не распространяются автоматически на его собственные подресурсы. Рассмотрим пример. Пусть файл a.js размещён в сети доставки контента (англ. CDN) и подключён к документу HTML в соответствии с SRI:

Файл http://victim.example/index.html
<!DOCTYPE html>
<html lang="ru-luna1918">
<head>
  <meta charset="UTF-8">
  <title>Атакуемый сайт</title>
  <script defer
          src="http://cdn.example/a.js"
          integrity="sha256-F5pZeHHm0pWtW+R7qlP1/kiuffGQkZAcQRvXNaiP2Fk="
          crossorigin="anonymous"></script>
</head>
<body>
  <h1>Атакуемый сайт</h1>
</body>
</html>

Файл a.js содержит следующий сценарий JavaScript:

Файл http://cdn.example/a.js
(function () {
  'use strict';

  window.addEventListener('load', function () {
    console.log('Выполнен сценарий JavaScript с cdn.example!');

    var head = document.getElementsByTagName('head');
    if (head.length) {
      var script = document.createElement('script');
      script.defer = 'defer';
      script.src = 'http://third-party.example/b.js';
      head[0].appendChild(script);
    }
  });
})();

Как мы видим, сценарий a.js создаёт и добавляет в документ HTML элемент script, который обязывает браузер скачать и выполнить сценарий b.js, размещённый на стороннем веб­‑сайте. Сценарий b.js может не являться вредоносным изначально, однако подключён к документу HTML без учёта SRI. Следовательно, злоумышленник в любой момент может подменить или модифицировать b.js.

Решением будет насаждение ограничений SRI в файле a.js.

Файл http://cdn.example/a.js
(function () {
  'use strict';

  window.addEventListener('load', function () {
    console.log('Выполнен сценарий JavaScript с cdn.example!');

    var head = document.getElementsByTagName('head');
    if (head.length) {
      var script = document.createElement('script');
      script.defer = 'defer';
      script.src = 'http://third-party.example/b.js';
      script.integrity = 'sha256-OdRL64E0NQAJcfA0EtBLnr1hUU6IbTW4xPrB+iTcsww=';
      script.crossOrigin = 'anonymous'; // Именно crossOrigin, а не crossorigin!
      head[0].appendChild(script);
    }
  });
})();
